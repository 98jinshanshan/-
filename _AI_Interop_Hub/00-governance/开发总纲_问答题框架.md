# Phase 1 自问自答深度检查清单

> **定位**：这是 Agent 在 Phase 1（意图锚定）中自问自答的深度模板。不是问用户的，是问 Agent 自己的。
> **目标**：Agent 按这 8 个维度、44 个问题逐层自审，全部通过后产出一句话理解，质量足以替代人工多轮交互。
> **配套总纲**：`开发总纲_人类可读版.md` / `开发总纲_可执行版.json`

---

## 为什么需要这个清单

```
用户的一句话："开发节点1的题材挖掘功能"

Agent如果不自审，可能理解为：
  "写一个爬虫脚本" ← 太窄，漏了热度排序、链接验证、相关性过滤

Agent如果按本清单自审，会追问自己：
  "最终产出是什么？→ 一个带热度排序的题材列表"
  "包含哪些功能？→ 多平台搜索、热度计算、链接验证、相关性过滤"
  "排除哪些？→ 不包含AI生成内容、不包含付费内容"
  ...
  → 最终理解："开发一个能从6个平台搜索热门小说题材并按热度排序的节点，含链接验证和相关性过滤"
```

**核心原则**：Agent 不是"猜"用户的意图，而是通过结构化自审，把模糊想法里所有隐含的维度都挖出来。每一个"我默认用户是这么想的"都是潜在偏差。

---

## 自审流程

```
用户一句话
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Agent 自问自答（本清单）                                     │
│                                                              │
│  维度1: 意图锚定（5问）→ 维度2: 边界划定（5问）               │
│  维度3: 架构建模（5问）→ 维度4: 接口契约（5问）               │
│  维度5: 模块拆解（5问）→ 维度6: 实现路径（5问）               │
│  维度7: 验证闭环（7问）→ 维度8: 交付收尾（7问）               │
│                                                              │
│  共 44 个自审问题，Agent 必须逐一回答                          │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  合成：Agent 基于 44 个回答，产出：                            │
│  1. 一句话精炼理解（内部使用，不输出给用户）                     │
│  2. spec.json（后续 Phase 2-8 的输入）                        │
│  3. 置信度自评（0-100%）                                     │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
  自动进入 Phase 2（全程不询问用户，不等待用户）
```

## 维度1：意图锚定（自审5问）

**目标**：把模糊想法变成一句精确的需求描述

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 用户最终想要的是什么？（一个具体的结果，不是过程描述） | 用户输入 + 项目上下文 |
| 2 | 这个结果服务于谁？（谁在用、谁受益） | 全局配置 + 项目上下文 |
| 3 | 为什么现在需要这个？（触发背景是什么） | 对话历史 + 项目状态 |
| 4 | 如果只能实现一个核心功能，那是什么？（砍到只剩骨架） | 用户输入推断 |
| 5 | 用一句话描述用户的需求。（不是"我想"，是"用户想要"） | 综合以上 4 问 |

**自审通过标准**：
- 第5问的一句话是前面4问的浓缩，不是凭空编的
- 这句话第三方能听懂，且能据此开始工作

---

## 维度2：边界划定（自审5问）

**目标**：明确什么做、什么不做，防止范围蔓延

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 明确包含哪些功能？（列出具体清单，不能是"大概"） | 用户输入 + 维度1的结论 |
| 2 | 明确排除哪些功能？（哪些看起来很相关但这次不做） | 用户输入推断 + 项目范围 |
| 3 | 有哪些硬性约束？（时间、资源、技术栈、兼容性） | 全局配置查阅 |
| 4 | 哪些功能不确定该不该做？（列入 uncertain 清单） | 诚实自评 |
| 5 | 功能和约束是否自洽？（included 都在约束内可实现？） | 交叉验证 |

**自审通过标准**：
- included 和 excluded 没有模糊地带
- uncertain 中的每一项都有"暂不处理"的约定
- 没有"先做做看"的灰色项目

---

## 维度3：架构建模（自审5问）

**目标**：确定数据从哪来、到哪去、经过哪些模块

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 数据从哪里来，到哪里去？（端到端数据流） | 用户输入 + 知识库 |
| 2 | 系统分为几个大模块？（每个模块一句话定义） | 维度1-2结论 |
| 3 | 每个模块的核心技术选型是什么？（语言、框架、依赖） | 全局配置查阅 |
| 4 | 模块之间是什么关系？（并行/串行/依赖/独立） | 数据流分析 |
| 5 | 有没有已有的系统或代码需要兼容？（外部依赖清单） | 项目状态检查 |

**自审通过标准**：
- 每个模块的数据输入和输出都明确
- 没有"悬空"的模块（不接收也不产出数据）
- 技术选型有明确理由（不是"我觉得这个好用"）

---

## 维度4：接口契约（自审5问）

**目标**：定义模块之间的精确接口——输入什么、输出什么、什么格式

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 每个模块的输入是什么格式？（JSON Schema，不是"大概"） | 维度3结论 |
| 2 | 每个模块的输出是什么格式？（JSON Schema） | 维度3结论 |
| 3 | 输入和输出之间有没有格式转换层？（A→JSON，B→CSV，谁转？） | 接口对齐检查 |
| 4 | 异常情况怎么处理？（输入为空、格式错误、超时） | 全局配置 + 经验 |
| 5 | 中间产物存放在哪里？（统一路径） | 全局配置查阅 |

**自审通过标准**：
- 任意两个相邻模块的 Schema 可以串联（上游输出 = 下游输入）
- 每个异常场景都有明确的降级策略
- 中间产物路径统一

---

## 维度5：模块拆解（自审5问）

**目标**：把每个模块拆到"一个函数能实现"的粒度

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 每个模块内部需要几个子步骤？（列出子步骤清单） | 维度3-4结论 |
| 2 | 每个子步骤的输入和输出是什么？ | 接口推导 |
| 3 | 子步骤之间有依赖关系吗？（A必须在B之前？可并行？） | 依赖分析 |
| 4 | 每个子步骤有没有可能失败？（失败后的降级策略） | 异常分析 |
| 5 | 子步骤是否足够简单？（一个函数能搞定？如果不能，继续拆） | 递归自检 |

**自审通过标准**：
- 每个叶子节点可以用一个函数实现
- 叶子节点之间没有未定义的依赖
- 递归深度不超过 4 层

---

## 维度6：实现路径（自审5问）

**目标**：确定开发顺序——先做什么、后做什么

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 所有叶子节点中，哪些不依赖任何其他节点？（第一批开发） | 维度5结论 |
| 2 | 第一批完成后，哪些可以开始？（第二批） | 依赖链推导 |
| 3 | 依此类推，画出完整的开发 DAG（有向无环图） | 拓扑排序 |
| 4 | 有没有可以并行开发的节点？（同一批次内） | 并行度分析 |
| 5 | 有没有循环依赖？（A→B→C→A，必须打破） | 环检测 |

**自审通过标准**：
- 每个节点的前置依赖都已在该节点之前开发
- 没有循环依赖
- 并行节点之间没有共享状态冲突

---

## 维度7：验证闭环（自审7问）

**目标**：定义每个节点的测试标准——"开发完" ≠ "跑通了"，"验收通过" = "开发完"

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 每个节点的验收标准是什么？（具体可验证的条件） | 维度1-2结论 |
| 2 | 正常路径测试：输入标准数据，期望输出是什么？ | 接口契约推导 |
| 3 | 异常路径测试：输入脏数据，期望行为是什么？ | 降级策略推导 |
| 4 | 边界路径测试：输入极限数据，期望行为是什么？ | 边界分析 |
| 5 | 节点之间的集成测试怎么做？（A输出→B输入，端到端） | 接口链推导 |
| 6 | 测试失败的判定标准是什么？（二元的，没有灰色地带） | 验收标准定义 |
| 7 | 测试通过后，谁来确认？（自动化/审核Agent） | 全局配置查阅 |

**自审通过标准**：
- 每个节点至少 3 个测试用例（正常/异常/边界）
- 集成测试覆盖所有节点连接点
- 验收标准是二元的（通过/不通过）

---

## 维度8：交付收尾（自审7问）

**目标**：确认所有产出物到位、归档、任务关闭

**Agent 自问**：

| # | 问题（问自己） | 回答来源 |
|---|---------------|---------|
| 1 | 所有代码是否写入协作目录？ | 全局配置查阅 |
| 2 | 所有文档是否更新？（INDEX.md、manifest.json） | 文档规范查阅 |
| 3 | 是否跑过集成测试？（维度7的测试用例全通过） | 测试结果 |
| 4 | 有没有遗留问题？（uncertain 清单、已知局限） | 维度2的 uncertain |
| 5 | 中间产物是否清理或归档？ | 路径规范查阅 |
| 6 | 下游节点能否直接使用本节点的产出？ | 接口验证 |
| 7 | 这个任务可以关闭了吗？（自检清单全部打勾） | 综合判定 |

**自审通过标准**：
- 所有维度7的测试用例通过
- 文档索引已更新
- 交付物齐全，下游可消费

---

## 合成规则

Agent 完成 44 个自审问题后，执行以下合成：

```
1. 提取维度1第5问的答案 → 作为"一句话理解"的骨架
2. 用维度2的 included 清单 → 补充具体功能细节
3. 用维度3-4的架构和接口 → 检查技术上是否可行
4. 用维度7的验收标准 → 检查是否可验证
5. 合成最终的一句话理解，写入 spec.json 的 clarified_requirement.one_liner
6. 自评置信度：44 问中，有多少问是"推断"的（inferred），有多少是"确认"的（confirmed）
   - 如果 inferred > 30% → 在 spec.json 中标记 low_confidence = true，但不阻塞执行
   - 如果 inferred < 10% → 在 spec.json 中标记 low_confidence = false
7. 写入 spec.json，自动进入 Phase 2，不询问用户，不等待用户
```

## 与开发总纲的关系

| 本清单 | 开发总纲 | 关系 |
|--------|---------|------|
| 维度1-2 | Phase 1（意图锚定） | 本清单是 Phase 1 的深度执行模板 |
| 维度3-4 | Phase 2-4（路径锚定→接口契约） | 本清单的预判，为 Phase 2-4 提供输入 |
| 维度5 | Phase 5（递归拆解） | 本清单的预拆解，为 Phase 5 提供方向 |
| 维度6-7 | Phase 6-7（实现→验证） | 本清单的预规划，为 Phase 6-7 提供蓝图 |
| 维度8 | Phase 8（交付收尾） | 本清单的预验收，为 Phase 8 提供检查清单 |

**本清单是"Phase 1 的深度自审工具"**，不是独立的执行流程。Agent 在 Phase 1 中调用本清单进行自问自答，产出高质量的一句话理解和 spec.json，然后自动进入 Phase 2。全程不询问用户，不等待用户。
