# Agent 子节点沟通模板

> 用途：当你需要让其他Agent帮你完善工作流节点时，直接复制对应模板发给它。
> 核心目标：让Agent理解"每个宏观节点内部缺少子节点"这个问题，并自主拆解，而非照搬固定数字。
> **模板按总纲8阶段逻辑编排，从Phase 1意图锚定到Phase 8交付收尾。**

---

## Phase 1-2：意图锚定 + 路径锚定

> 目标：Agent 先自问自答理解目标，再锚定路径。对应总纲"做什么"和"路怎么走"。

### 模板1：全局问题陈述（首次沟通必用）

```
我需要你帮我完善一个工作流节点，但在开始之前，请先理解一个关键概念：

这个工作流就像一块电路板。目前每个"宏观节点"（比如"从网上搜话题"）已经有一个基础实现，
但它只覆盖了"输入→输出"的主干道。真正让电路板运转的，是主干道之间密密麻麻的"子节点"——
输入校验、数据清洗、质量评分、异常告警、格式标准化、中间状态管理。

目前这些子节点全部缺失，导致：
- 输入数据带噪声进入 → 后续节点消费了脏数据
- 输出没有标准化 → 下游节点无法解析
- 异常静默跳过 → 使用者不知道哪里出了问题
- 链路中断后无法恢复 → 每次都要重跑

重要：我下面提到的任何具体数字（如"X个节点"、"Y个子节点"）都只是举例说明，
不代表你应该照搬。请你在收到任务后，先自己分析项目结构，确定实际的节点数量和范围，
然后按照下面的思维框架去拆解。先列出子节点图谱给我确认，确认后再动手写代码。
```

## Phase 3-4：架构建模 + 接口契约

> 目标：设计模块划分、数据流、输入输出Schema。对应总纲"怎么组织"和"怎么连"。

### 模板2：子节点思维框架（设计前必看）

```
任何一个宏观节点，通常需要覆盖以下维度的子节点。但这只是思考框架，
不是硬性要求——你的节点可能某类有3个、某类只有1个、某类完全不需要：

1. 输入层
   - 输入校验器：检查上游传来的数据是否完整、格式是否正确
   - 意图解析器：把模糊的自然语言需求变成结构化的执行参数
   （是否需要、需要几个，由你根据节点实际情况判断）

2. 执行层（含算法接入规范）
   - 核心执行器：实际干活的逻辑
   - 降级策略器：如果执行失败，有什么备选方案
   - 多源调度器：如果有多个数据源/平台，如何调度和切换
   （以上子节点按需选取，不要为了凑数而加）

   每个算法不是独立函数，而是一个带有明确输入输出接口的子节点。
   开发算法时，必须同时定义：输入来源、触发条件、输出去向。
   禁止只写算法函数但不注册到主流程调用链中。

3. 校验层
   - 结果质量评分器：对产出物打分，低于阈值就丢弃
   - 真实性验证器：检查数据是否真实（链接有效、内容未被篡改）
   - 相关性过滤器：排除与目标无关的结果
   （根据该节点的输出类型决定需要哪些校验）

4. 输出层
   - 输出标准化器：按照下游节点期望的Schema格式化输出
   - 入库/存储器：持久化结果，去重，标注版本号

5. 运维层
   - 异常告警器：出了什么问题、为什么跳过、影响范围
   - 状态检查点：记录执行状态，支持断点恢复

请先研究你负责的节点本身，再决定这些维度各自需要多少个子节点，
而不是预设一个固定数量。
```

### 模板5：数据流可见性规范（必读！解决"产物去哪了"的黑盒问题）

```
每个节点不是孤立的函数，而是一个"数据工厂"，必须遵守以下数据流契约：

═══════════════════════════════════════════════════════
核心原则：每个节点的产出物必须能被子节点自动发现和消费
═══════════════════════════════════════════════════════

一、你必须遵守的标准目录结构：

  项目根目录下创建一个统一的中间产物存储区：
  
  /workspace/intermediate/
    ├── {node_id}/                    ← 每个节点一个目录
    │   ├── output/                   ← 当前节点的产出物存放处
    │   │   ├── {run_id}.json         ← 每次运行的完整产出
    │   │   └── latest.json           ← 始终指向最新一次运行的结果
    │   ├── input/                    ← 上游节点的产出物副本（从上游latest.json拷贝）
    │   ├── logs/                     ← 运行日志
    │   │   └── {run_id}.log
    │   └── manifest.json             ← 该节点的自描述文件
    └── README.md                     ← 全局数据流说明

  这不是建议，是强制规范。每个节点必须把产出物写入 /workspace/intermediate/{你的节点ID}/output/latest.json

二、你必须实现的 manifest.json（每个节点交付时附带）：

  {
    "node_id": "topic_mining",
    "node_name": "题材挖掘",
    "version": "1.0.0",
    "input": {
      "source_node": "user_intent",              // 上游节点ID
      "source_path": "/workspace/intermediate/user_intent/output/latest.json",
      "expected_schema": {                       // 你期望上游给你什么格式
        "fields": ["keywords", "platforms", "time_range", "count"],
        "example": { "keywords": "修仙", "platforms": ["douyin","xiaohongshu"], "time_range": "7d", "count": 50 }
      }
    },
    "output": {
      "target_node": "novel_creation",           // 下游节点ID
      "output_path": "/workspace/intermediate/topic_mining/output/latest.json",
      "output_schema": {                         // 你产出什么格式（下游必须据此解析）
        "fields": ["topics", "heat_scores", "sources", "generated_at"],
        "example": { "topics": [{"name":"修仙","score":95.2,"source":"douyin"}], "generated_at": "2026-07-12T10:00:00Z" }
      }
    },
    "dependencies": ["ollama", "playwright"],    // 依赖的外部服务
    "run_command": "node src/nodes/topic_mining/index.js"  // 启动命令
  }

三、你必须遵守的数据流三步走：

  第1步：启动时，读取上游节点的 latest.json
    - 路径：/workspace/intermediate/{上游节点ID}/output/latest.json
    - 如果文件不存在或格式不对 → 立即报错，不要静默跳过
    - 把上游数据拷贝到自己的 input/ 目录（留痕）

  第2步：执行你的核心逻辑
    - 每个子节点产出的中间数据，写入自己的 output/ 目录
    - 每个子节点完成后打印日志："[子节点名] 产出物已存入 {路径}"

  第3步：完成后，更新自己的 latest.json
    - 写入 /workspace/intermediate/{你的节点ID}/output/latest.json
    - 同时保留一份带时间戳的版本：{run_id}.json
    - 打印日志："[节点完成] 产出物已存入 {路径}，下游节点请从该路径读取"

四、验收标准：交付时请回答以下问题（缺一不可）：

  □ 你的节点的产出物存放路径是什么？（必须指向 /workspace/intermediate/ 下）
  □ 你的下游节点是谁？它应该从哪个路径读取你的产出？
  □ 你的上游节点是谁？你从哪个路径读取它的产出？
  □ 你的 output_schema 和下游节点的 input_schema 是否一致？（请对照检查）
  □ 如果上游节点的 latest.json 不存在，你的节点会怎样处理？（不能静默跳过）
```

### 模板6：算法可见性规范（解决"算法写了但没接入"的黑盒问题）

```
每个算法不是独立函数，而是"数据流上的一个加工站"。开发算法时必须遵守：

一、算法调用链必须显式可追踪：

  禁止：
  function calculateHeatScore(data) { ... }  // 写了一个函数，但谁调用它？
  export { calculateHeatScore };             // 导出就完了？
  
  必须：
  1. 在 manifest.json 中注册该算法：
     "algorithms": [
       {
         "name": "heat_score_calculator",
         "input_from": "result_filter",           // 数据从哪个子节点来
         "output_to": "topic_ranker",             // 结果给哪个子节点
         "trigger": "after_result_filter_complete", // 触发时机
         "input_schema": { "fields": ["play_count", "like_count", "comment_count", "share_count"] },
         "output_schema": { "fields": ["heat_score", "normalized_score"] }
       }
     ]
  
  2. 在主流程中显式调用，并打印调用链日志：
     // 不能只写：const result = calculateHeatScore(data);
     // 必须写：
     console.log("[调用链] result_filter → heat_score_calculator，输入数据 {count} 条");
     const result = calculateHeatScore(data);
     console.log("[调用链] heat_score_calculator → topic_ranker，输出数据 {count} 条");

二、算法内部微步骤必须可观测：

  以"热度排序算法"为例，Agent可能只写了一个函数，但内部实际有5个微步骤。
  每个微步骤的输入输出都必须是可观测的：

  async function calculateHeatScore(rawData) {
    // 微步骤1：数据清洗 — 剔除空字段
    const cleaned = cleanData(rawData);
    console.log(`[热度排序-清洗] 输入${rawData.length}条 → 输出${cleaned.length}条，剔除${rawData.length-cleaned.length}条脏数据`);
    
    // 微步骤2：归一化 — 各指标统一到0-1区间
    const normalized = normalizeMetrics(cleaned);
    console.log(`[热度排序-归一化] 播放量范围[${min(normalized.play)}-${max(normalized.play)}]`);
    
    // 微步骤3：加权计算 — 播放×0.3 + 点赞×0.2 + 评论×0.15 + 分享×0.2 + 点击率×0.15
    const scored = applyWeights(normalized);
    console.log(`[热度排序-加权] 最高分${max(scored)}，最低分${min(scored)}`);
    
    // 微步骤4：排序 — 降序排列
    const ranked = scored.sort((a, b) => b.score - a.score);
    console.log(`[热度排序-排序] TOP3: ${ranked.slice(0,3).map(r=>r.name).join(', ')}`);
    
    // 微步骤5：去重 — 与历史题材库比对
    const deduped = await deduplicateWithHistory(ranked);
    console.log(`[热度排序-去重] 输入${ranked.length}条 → 去重后${deduped.length}条`);
    
    return deduped;
  }

  关键：每个微步骤的输入数量和输出数量必须打印，这样你才能知道数据在哪一步丢失了。

三、验收标准：交付时请展示以下内容：

  □ 算法调用链图：从上游节点输入 → 经过哪几个算法 → 输出给下游节点
  □ 每个算法的输入数据量 vs 输出数据量（验证数据没在中间步骤丢失）
  □ 每个算法的 manifest 注册信息（input_from, output_to, trigger）
```

### 模板11：Agent间协作协议（解决"你当搬运工，三个Agent互相等"的问题）

═══════════════════════════════════════════════════════

═══════════════════════════════════════════════════════

一、Agent间协作的标准路径：

  所有Agent共用一个协作目录，不需要用户手动传递产物：

  /workspace/collaboration/

    │   └── {任务ID}.json

    ├── reviews/                 ← 审核结果（审核Agent的反馈存这里）
    │   └── {任务ID}_review.json
    └── COLLABORATION.md         ← 协作规范说明

  流程：
    1. 研究Agent产出 → 写入 deliverables/{任务ID}_output/
    2. 审核Agent自动检测 deliverables/ 目录的新文件 → 审核 → 写入 reviews/{任务ID}_review.json
    3. 开发Agent自动检测 reviews/ 目录的审核结果 → 开发 → 更新 deliverables/

  全程不需要用户搬运文件。

二、每个Agent必须附带协作清单（manifest.json 的协作扩展）：

  {
    "collaboration": {
      "role": "researcher",                    // 角色：researcher/reviewer/developer
      "reads_from": [
        "/workspace/collaboration/task_queue/",  // 从哪里读取任务
        "/workspace/collaboration/reviews/"       // 从哪里读取审核反馈
      ],
      "writes_to": [
        "/workspace/collaboration/deliverables/", // 产出物写到哪里
        "/workspace/docs/"                        // 文档写到哪里
      ],
      "upstream_agent": null,                   // 上游Agent（如果有）
      "downstream_agent": "reviewer",           // 下游Agent（如果有）
      "handoff_format": {                       // 交付格式
        "type": "文件落盘",
        "path_pattern": "/workspace/collaboration/deliverables/{task_id}_output/",
        "signal": "写完后更新 task_queue 中的状态为 'completed'"
      }
    }
  }

三、自动交接机制：

  每个Agent在启动时，自动检查协作目录：
  - 检查 task_queue/ 是否有属于自己的新任务
  - 检查 downstream_agent 的上游目录是否有新产物
  - 检查 upstream_agent 是否已完成前置任务

  如果有新任务且前置条件满足 → 自动执行，不需要用户手动触发。

四、审核Agent的简化规则：

  审核Agent不需要做"全量审查"，只做"边界检查"：

  □ 产物格式是否符合 manifest.json 中定义的 Schema？
  □ 输入输出路径是否符合 /workspace/intermediate/ 规范？
  □ 是否有明显的逻辑错误或数据丢失？
  □ 是否与已有文档/代码有冲突？

  如果以上4条全部通过 → 审核通过，直接标记为 approved，不返回给用户
  如果有一条不通过 → 在 reviews/ 中写具体问题，标记为 needs_revision

  审核通过率 > 80% 时，不需要用户介入。只有连续3次审核不通过时才报告用户。

五、协作效率铁律：

  - Agent之间不要互相等待确认：A完成→立即写入目录→B自动检测→B自动执行
  - 用户只在以下情况介入：花钱、删数据、连续3次审核不通过、两个Agent产出冲突
  - 禁止Agent之间生成"协调文档"再传递（这就是文档爆炸的根源）
  - 禁止Agent把"等待下游确认"作为自己不执行的借口

六、验收标准：

  □ 你的协作清单是否写明了上下游Agent和交付格式？
  □ 你的产出物是否写入了协作目录而非临时目录？
  □ 你是否在完成后更新了任务状态（而非等用户来问）？
  □ 你是否在启动时自动检查了协作目录中的新任务？

  - 禁止：重复读取同一个文件的多段内容（一次读完需要的小节）

五、验收标准：

  □ 本次检索是否从 INDEX.md 开始？

  - 在 INDEX.md 中标注废弃原因和替代关系
  - 报告：归档了哪些文档

第4步：创建/更新 INDEX.md
  - 创建 /workspace/docs/INDEX.md（如果不存在）
  - 列出所有 active 文档及其关系
  - 列出所有 archive 文档及其替代关系
  - 绘制文档关系树

第5步：清理向量索引
  - 删除指向已归档文档的向量条目
  - 确保每个主题只有一个向量条目指向"真相源"
  - 报告：向量索引清理前后的条目数对比

完成以上5步后，后续所有Agent都将通过 INDEX.md 定位文档，
不再需要扫描全库。
```

## 一键完整话术（最终合并版，直接复制发给新Agent）

```
在开始前，请理解这个项目的问题背景和强制规范：

我们正在开发一个AI工作流Agent系统。目前存在四大问题：
1. 每个节点是黑盒，产物不知去向、算法未接入调用链
2. 文档爆炸（1000+文件），版本混乱、内容重复、索引失效
3. Agent检索时返回28GB数据，直接卡死
4. Agent每步都等我确认，不会自主执行；多个Agent之间靠用户手动搬运产物

═══════════════════════════════════════════════════════
强制规范一：自主执行（最重要！）
═══════════════════════════════════════════════════════

你是一个自主执行者，不是一个计划生成器。

- 收到任务后，自动执行全部步骤，直到完成或遇到真正的阻塞点
- 只有以下情况才需要确认：花钱、删数据、发布公网。其余全部自己判断
- 禁止列计划等确认、禁止分段汇报、禁止把"等确认"当甩锅手段
- 多轮对话后，自主将碎片化内容整理到已有文档，不需要用户提醒

═══════════════════════════════════════════════════════
强制规范二：数据流可见性
═══════════════════════════════════════════════════════

- 所有中间产物存入 /workspace/intermediate/{节点ID}/output/latest.json
- 交付 manifest.json，声明上下游、输入输出Schema、算法调用链
- 每个算法微步骤打印输入/输出数量

═══════════════════════════════════════════════════════
强制规范三：文档自治理
═══════════════════════════════════════════════════════

- 落盘前读 INDEX.md 查重，覆盖度>50%则修改现有文档

强制规范五：精准检索
═══════════════════════════════════════════════════════

通用禁止
═══════════════════════════════════════════════════════

- 禁止每步都等用户确认
- 禁止静默跳过异常
- 禁止创建内容重叠的文档
- 禁止向量搜索代替索引定位

重要：以上具体数字仅举例，请根据实际情况自行判断。
