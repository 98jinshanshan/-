> **版本**：V3.9.1 | 更新于 2026-07-17
> **一句话定义**：一句话输入 → AI Agent 三角色协作 → 全自动产出交付物
> **阅读时间**：约 10 分钟

## 🚀 快速上手（5 分钟）

### 你需要知道的 3 件事

1. **这是一个 AI Agent 协作开发框架**——用户说一句话，三个 AI Agent（研究员/协调者/开发者）自动协作，从需求拆解到最终交付全流程自动化。
2. **核心机制是"双程递归 + 约束收敛"**——去程（Phase 1-5）把模糊意图逐层拆解成可执行的最小单元，回程（Phase 6-8）自底向上逐层验证合成。
3. **你扮演哪个角色，就只读哪个角色的规范**——不要试图一次理解全部内容。

### 第一步：确定你的角色

| 你的 IDE | 角色 | 你要做什么 | 必读文件 |
|---------|------|-----------|---------|
| **Codex** / 研究员 IDE | 🔬 研究员 | 去程设计：把一句话拆解成结构化蓝图（spec/path/nodes/interfaces/leaf_nodes） | `AGENT_INSTRUCTION.md` + `开发总纲_人类可读版.md` 的 Phase 1-5 |
| **OpenClaw** / 协调者 IDE | 🎯 协调者 + 监督者 | 任务分配 + 质量监督 + 异常处理 + 结果交付 | `AGENT_INSTRUCTION.md` + `三IDE角色自动化联通协议_V2.1.md` |
| **Cursor / Windsurf** / 开发者 IDE | 💻 开发者 | 回程实现：按蓝图写出代码/内容并验证交付 | `AGENT_INSTRUCTION.md` + `开发总纲_人类可读版.md` 的 Phase 6-8 |

### 第二步：启动 IDE Bridge（通信层）

```bash
# 1. 进入项目目录
cd /workspace

# 2. 启动 Bridge 服务端
node ide-bridge.js

# 3. 测试连接（另开终端）
node test-bridge-connection.js
```

> Bridge 是三 IDE 通信的中枢，默认运行在 `ws://localhost:9527`。三个 IDE 都连接到同一个 Bridge 就能协作。

### 第三步：接收并执行任务

1. 你的 IDE 会通过 Bridge 收到任务消息
2. 按 `AGENT_INSTRUCTION.md` 中的角色规范执行
3. 完成后通过 Bridge 提交成果
4. 等待协调者审核

## 🏗️ 整体架构

### 一句话架构图

```
用户一句话输入
    ↓
Phase 0: 约束收敛引擎（5层筛选，从~1920种可能收敛到唯一解）
    ↓
门禁引擎（强制8阶段顺序，防跳过）
    ↓
┌─────────────────────────────────────────────┐
│  去程设计（研究员 Codex）                    │
│  Phase 1→2→3→4→5                            │
│  意图→路径→节点→接口→递归拆解                │
└──────────────────────┬──────────────────────┘
                       ↓
         目标对齐监督者（协调者 OpenClaw）
         8个检查点 + goal_trace + 防漂移
                       ↓
┌──────────────────────┴──────────────────────┐
│  回程实现（开发者 Cursor/Windsurf）          │
│  Phase 6→7→8                                │
│  实现→验证→交付                              │
└─────────────────────────────────────────────┘
```

### 8 Phase 速查表

| Phase | 名称 | 产出 | 核心检查点 |
|-------|------|------|-----------|
| **0** | 约束收敛 | 领域/子类型/平台/结构/规则 | 5 层收敛，唯一解 |
| **1** | 意图锚定 | spec.json（根目标 + 44问） | inferred < 30%，一致性校验通过 |
| **2** | 路径锚定 | path.json（执行路径） | 节点数合理，连通无环，终点对齐 |
| **3** | 节点识别 | nodes.json（各层级节点） | 每层级节点数达标，无重叠 |
| **4** | 接口契约 | interfaces.json（节点间接口） | 字段完整性 ≥ 80%，类型转换合法 |
| **5** | 递归拆解 | leaf_nodes.json（叶节点） | 原子性满足，覆盖度 ≥ 100% |
| **6** | 实现集成 | 代码/内容产出 | 质量指标达标，无越权偏差 |
| **7** | 通路验证 | verification.json | 五层验证 100% 通过 |
| **8** | 交付收尾 | manifest.json + 最终交付物 | 核心文件齐全，可运行 |

### 三角色分工

| 角色 | 负责 Phase | 不负责 | 产出目录 |
|------|-----------|--------|---------|
| 🔬 研究员（Codex） | Phase 0-5（去程设计） | 写代码/写正文 | `collaboration/deliverables/{task}_research/` |
| 🎯 协调者（OpenClaw） | 全程监督 + 任务分配 + 异常处理 | 具体设计和实现 | 任务队列 + 检查点审核 |
| 💻 开发者（Cursor） | Phase 6-8（回程实现） | 修改设计蓝图 | `collaboration/deliverables/{task}_code/` |

## 🔧 如何执行业务

### 任务生命周期

```
用户发起任务
    ↓
调度器（orchestrator.js）创建任务
    ├── 生成 task.json
    ├── 放入 collaboration/task_queue/
    └── 通过 Bridge 推送给研究员
    ↓
研究员（Codex）执行去程
    ├── Phase 0: 约束收敛
    ├── Phase 1: 意图锚定 → spec.json
    ├── Phase 2: 路径锚定 → path.json
    ├── Phase 3: 节点识别 → nodes.json
    ├── Phase 4: 接口契约 → interfaces.json
    └── Phase 5: 递归拆解 → leaf_nodes.json
    ↓
协调者（OpenClaw）审核去程
    ├── 检查 cp_0 ~ cp_4
    ├── 通过 → 推送给开发者
    └── 不通过 → 打回研究员修改（最多 3 次）
    ↓
开发者（Cursor/Windsurf）执行回程
    ├── Phase 6: 实现集成
    ├── Phase 7: 通路验证 → verification.json
    └── Phase 8: 交付收尾 → manifest.json
    ↓
协调者（OpenClaw）审核回程
    ├── 检查 cp_5 ~ cp_7
    ├── 通过 → 交付给用户
    └── 不通过 → 打回开发者修改（最多 3 次）
    ↓
任务完成 ✅
```

### 你作为研究员怎么干活

1. 从 Bridge 收到任务
2. 读取 `AGENT_INSTRUCTION.md` 确认你的角色规范
3. 执行 Phase 0 约束收敛（确定领域/子类型/平台/结构/规则）
4. 执行 Phase 1-5，每个 Phase 产出对应的 JSON 文件
5. 每完成一个 Phase，做自检检查点
6. 全部完成后，通过 Bridge 提交给协调者审核
7. 如果被打回，按修改意见调整后重新提交

### 你作为开发者怎么干活

1. 从 Bridge 收到去程蓝图（spec/path/nodes/interfaces/leaf_nodes）
2. 读取 `AGENT_INSTRUCTION.md` 确认你的角色规范
3. 按蓝图执行 Phase 6-8
4. Phase 6: 实现所有叶节点，满足质量指标
5. Phase 7: 自底向上五层验证
6. Phase 8: 整理交付物，生成 manifest
7. 通过 Bridge 提交给协调者审核

### 你作为协调者怎么干活

1. 从用户/调度器收到任务
2. 分配给研究员
3. 去程完成后审核 cp_0~cp_4
4. 通过后分配给开发者
5. 回程完成后审核 cp_5~cp_7
6. 通过后交付给用户
7. 全程监控 goal_trace，防止目标漂移

## 🔍 约束收敛引擎（Phase 0）

> **最重要的机制**——在开始任何工作之前，必须先跑 Phase 0。

### 5 层收敛

| 层 | 名称 | 做什么 | 字典文件 |
|----|------|--------|---------|
| L0 | 领域分类 | 确定是创作/开发/分析/研究哪一类 | `domain-dict.json` |
| L1 | 子类型分类 | 领域内具体是什么（小说/短剧/Web应用...） | `subtype-dict.json` |
| L2 | 平台/技术栈 | 运行平台/格式要求 | `platform-dict.json` |
| L3 | 结构参数 | 数量/规模/结构（集数/章数/功能点） | `structural-patterns.json` |
| L4 | 领域规则 | 硬约束 + 软约束 | `domain-rules.json` |
| L5 | 唯一性锚定 | 从多个候选中选唯一最优解 | 质量评分公式 |

### 为什么需要 5 层？

**漏维即发散**——缺少任意一层，就会有多种合法解并存，无法收敛到唯一路径。

- 没有 L0：创作和开发的路径都对
- 没有 L1：小说和短剧的路径都对
- 没有 L2：抖音和B站的格式都对
- 没有 L3：1集和100集都对
- 没有 L4：可能违反领域规则

### 收敛速度配置

| 级别 | 适用场景 | gap_threshold | min_confidence |
|------|---------|--------------|---------------|
| conservative | 量化风控、合规审查 | 2 | 0.9 |
| medium（默认） | 创作拆解、开发任务 | 1 | 0.7 |
| aggressive | 探索性任务、快速原型 | 0 | 0.5 |

## ⚠️ 重要原则

### 双程规则（核心中的核心）

- **去程只设计不实现**：研究员不写代码/不写正文，只产出结构化蓝图
- **回程只实现不改设计**：开发者严格按蓝图实现，不修改设计
- **越界需要授权**：任何一方发现需要修改对方的工作，必须通过协调者走变更流程

### 唯一文档原则

每个概念只有一个权威定义源：

| 概念 | 权威文件 |
|------|---------|
| 总纲机制 | `开发总纲_人类可读版.md` |
| 通信协议 | `三IDE角色自动化联通协议_V2.1.md` |
| Agent 行为 | `AGENT_INSTRUCTION.md` |

> 不要在多个文件里复制同一段定义。需要就引用，不要复制。

### 演进三原则

所有后续开发必须遵守：

1. **研究必须有工程锚点**——不能光有理论，要有具体落地的文件和函数
2. **标准对齐从需求出发**——不是"行业标准是什么就做什么"，而是"总纲需要什么就对齐什么"
3. **完成度评估用最小可行链路**——按 8 个维度评估，不是拍脑袋

## 🔧 常见问题

### Q: 我收到任务后第一步做什么？

A: 先读 `AGENT_INSTRUCTION.md` 确认你的角色和职责，然后执行 Phase 0 约束收敛。

### Q: Phase 0 收敛失败怎么办？

A: 看是哪一层失败：
- L0/L1 失败 → 可能字典里缺少关键词，标记为 ambiguous 并向协调者报告
- L3 提取率低 → 标记 high_inference_risk，继续执行但在 spec 中注明哪些是推断的
- L4 硬约束违反 → 任务无法执行，向协调者报告

### Q: 我发现需要修改其他角色的工作怎么办？

A: 不要直接改！通过 Bridge 向协调者发起变更请求，由协调者协调。双程规则不能打破。

### Q: 检查点没通过怎么办？

A: 每个检查点最多重试 3 次。3 次都失败：
- 如果是 cp_0 或 cp_7（关键路径）→ 升级给协调者
- 其他检查点 → 记录失败原因，继续但标记风险

### Q: 怎么连接 IDE Bridge？

A: 参考 `三IDE角色自动化联通协议_V2.1.md`。简单来说：
1. 启动 `ide-bridge.js`（服务端）
2. 每个 IDE 使用 `ide-bridge-client.js` 连接
3. 注册角色后就能收发消息
