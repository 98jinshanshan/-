## 二、算法演进时间线（按对话顺序）

### 第 1 轮：用户核心诉求（原始需求锚定）

**用户输入**：
> "我认为你们两个在逗我，首先叫你进行算法纠偏的意义在于让唯一答案真的是唯一，而不是你们在这里进行泛泛而谈的说它拆开又合并的认为是唯一答案的对吗？你最起码每个节点进行纠偏有个阈值范围约束条件吧？我们设计纠偏的目的不就是用算法快速从那么多无穷尽的可能你快速进行收束范围直到最后目标唯一吗？"

**提取的核心需求**：

| 编号 | 需求 | 状态 |
|------|------|------|
| R1 | 每个节点有阈值范围约束条件 | → 后续在 L0-L5 各层中定义了 scoring/threshold |
| R2 | 有参考标准——能判断一句话指向哪个领域/方向 | → L0-L1 字典 + L4 领域规则 |
| R3 | 筛选收束机制是核心，不是嘴上说穷举 | → 5 层逐层过滤，每层有收束度量 |
| R4 | 通过运算执行得到唯一结果，不是靠 Agent 理解 | → 关键词匹配 + 计分公式 + 阈值判断 |

**本轮产出**：明确了收敛引擎的 4 个核心需求，作为后续所有讨论的基准。

### 第 3 轮：豆包方案对比——向量锚定指数级收束算法

**豆包核心方案**：

| 组件 | 豆包方案 | 与 Trae 方案差异 |
|------|---------|-----------------|
| 入口 | BGE-M3 向量化 → 超平面切割穷举 | Trae 用关键词字典匹配 |
| 收束 | 三层向量筛选，每层淘汰 90%（3000→300→30→1） | Trae 用阈值判断，收束精度未量化 |
| 唯一性 | SHA-256 哈希指纹 + 字典序排列 | Trae 用阈值筛选到只剩一个 |
| 数学 | 有余弦相似度公式、加权打分公式、哈希碰撞概率证明 | Trae 只有计分公式 |

**Trae 识别出的豆包致命缺陷**：

| 编号 | 缺陷 | 状态 |
|------|------|------|
| DB1 | 超平面拟合依赖历史数据，冷启动死锁 | → 第 4 轮 DeepSeek 融合方案中解决 |
| DB2 | BGE-M3 对中文创作类任务语义区分度存疑（0.9 阈值拍脑袋） | → 第 4 轮中解决：字典硬分类 + 向量仅补漏 |
| DB3 | 哈希指纹字典序 = 随机选择，不是择优 | → 第 4 轮中解决：质量排序优先，哈希仅兜底 |

**Trae 提出的融合方向**：
- L0-L2 关键词粗筛（Trae，冷启动可用）
- L3-L4 规则约束（Trae，领域规则具体可验证）
- 唯一性锚定（豆包，SHA-256 哈希指纹）
- **不采用**豆包的超平面向量化穷举（冷启动死锁 + 中文区分度未验证）


### 第 5 轮：量化交易领域扩展验证与 8 项纠偏

**DeepSeek 量化交易探讨核心内容**：

| 应用方向 | 算法映射 | 可行性 |
|---------|---------|--------|
| 多策略自适应融合与切换 | 字典 = 市场制度规则，向量 = 相似日匹配，质量 = 夏普+回撤 | 高 |
| 因子组合动态筛选与赋权 | 字典 = 经济逻辑分类，向量 = 因子簇匹配，质量 = IC_decay+换手率 | 高 |
| 交易执行算法智能路由 | 字典 = 订单特征筛选，向量 = 盘口微观结构匹配，质量 = 滑点节省 | 中高 |

**DeepSeek 识别的 4 个局限**：
1. 输入从自然语言变成连续数值——需要语义化桥接
2. 时效性要求约束了计算复杂度——只能用于中低频
3. 超平面切割需处理非平稳性——市场制度突变
4. 向量嵌入模型需专项训练——通用模型对金融错配

**Trae 的 8 项纠偏**：

| 编号 | 纠偏内容 | 状态 |
|------|---------|------|
| QT1 | 过度抽象化掩盖"字典是领域锁定的"——框架通用，字典完全不同 | ✅ 已解决（框架/字典分离架构确认） |
| QT2 | 冷启动在量化中的真实价值是防过拟合，而非"无数据" | ✅ 已解决（概念纠正） |
| QT3 | L4 在量化中是监管/风控硬约束，权重应比创作领域更高 | ✅ 已解决（提出硬/软约束分离） |
| QT4 | 语义化桥接需要滞后阈值（hysteresis），防止边界震荡 | ⚠️ 未解决（方案已提出，未写入引擎） |
| QT5 | 收敛速度 vs 收敛精度需要可配置参数，随波动率自适应 | ⚠️ 未解决（方案已提出，未写入引擎） |
| QT6 | 决策影响环境的反馈循环——量化中策略改变市场，需因果推断 | ⚠️ 未解决（标记为 V4.0 扩展） |
| QT7 | 哈希锚定在量化中的角色是审计锚，非决策锚 | ⚠️ 未解决（方案已提出，审计日志未实现） |
| QT8 | 引擎自身验证框架缺失——如何知道收敛结果是对的？ | ⚠️ 未解决（对标基准和判定标准已提出，未固化） |

**量化交易验证的核心结论**：
> 量化交易不是收敛引擎的"另一个应用场景"，而是它的压力测试。量化对确定性、可解释性、风控合规的要求比创作领域严格得多。如果引擎能在量化交易中验证通过，那么它在所有领域都能通过。

## 四、V3.9.0 收敛引擎最终架构（融合后定稿）

### 4.1 5 层架构总览

```
                        输入一句话
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L0 领域分类（字典主导，确定性匹配）                            │
│ · 关键词命中/排除计分：score = hits - exclusion              │
│ · 阈值：score > 0 且 score ≥ second_best + 1                │
│ · 并列 → 双路径并行，后续自然淘汰（不调大模型）                │
│ · 全部无匹配 → 标记 ambiguous，双路径并行                     │
│ · 滞后阈值：上一轮领域标签保持，直到差距 ≥ 2 才切换            │
│ 收束：4大类 → 1-2个领域                                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L1 子类型约束（字典主导）                                      │
│ · 领域内关键词匹配，取最高分                                  │
│ · 阈值：score > 0 且 score ≥ second_best + 1                │
│ · 不满足 → 标记 inferred，取最高分                            │
│ 收束：N种子类型 → 1-2种子类型                                 │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L2 平台/技术栈（纯字典，100%确定性）                           │
│ · 平台关键词匹配 → 应用平台规则（格式/字数/技术栈等）           │
│ · 无匹配 → 通用默认规则，标记 inferred                        │
│ 收束：平台参数全部锁定                                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L3 结构参数提取（正则 + 规则）                                 │
│ · 正则提取：数量（前N章）、范围（2000-2500字）、枚举（视角）、格式│
│ · 提取率 ≥ 50% → 继续；< 50% → 标记 high_inference_risk      │
│ 收束：所有结构参数锁定为数值/范围                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L4 领域规则校验（硬约束淘汰 + 软约束扣分）                      │
│ · 硬约束（违反直接淘汰）：合规红线、仓位上限、风控限额          │
│ · 软约束（违反扣分不淘汰）：建议的对话占比、建议的章节结构      │
│ · 查询领域规则数据库，全部 applicable 规则必须应用              │
│ · 每条标注 verifiable: true/false                            │
│ 收束：领域专属约束全部锁定，违反硬约束的路径被淘汰              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│ L5 唯一性锚定（质量排序 + 哈希兜底）                           │
│ · 收敛阶段评分：Score = 0.3×完整性 + 0.3×冲突率               │
│                      + 0.2×精确度 + 0.2×规则合规度            │
│ · 选最高分，记录 confidence 字段                              │
│ · 平手时：SHA-256 哈希 + 十六进制字典序取第一                  │
│ · 每次决策输出 SHA-256 哈希到审计日志（无论是否平手）           │
│ 收束：N条合法路径 → 1条唯一路径                               │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                      输出唯一 spec.json
                    （附带 confidence 字段）
```

### 4.2 关键设计原则

| 原则 | 说明 |
|------|------|
| 确定性 | 同一输入永远产生同一输出。关键词匹配是确定性的，计分公式是确定性的，哈希是确定性的 |
| 不调大模型 | 字典匹配失败时，走双路径并行而非调大模型猜测。大模型引入不确定性 |
| 不弹窗 | 边界模糊时标记 confidence 而非中断流程。自动化优先 |
| 框架/字典分离 | 5 层框架通用，每个领域的字典独立构建。换领域 = 换字典，不改引擎 |
| 硬约束优先 | L4 中硬约束违反直接淘汰，软约束扣分。硬约束是"唯一答案"的底线 |
| 收敛速度可配 | 三级参数 conservative/medium/aggressive，控制阈值松紧度 |

## 七、字典设计深度专题（第 6 轮新增）

> 来源：用户提出四个字典设计核心问题。本专题回答字典的写法、领域数量、生命周期、冷热协同。

### 7.1 字典是什么——结构化 Schema

字典不是关键词列表，而是一个有明确 Schema 的结构化知识制品：

```
字典条目 Schema = {
  schema_version: "1.0.0",
  domain_id: "creative",
  domain_label: "创作类",
  domain_definition: "从零到一产出新内容的任务",
  L0: { keywords: [...], exclusion: [...], scoring: "hits - exclusion" },
  L1: { subtypes: { fiction: { keywords: [...], output: "fiction" }, ... } },
  L2: { platforms: { qidian: { keywords: [...], rules: {...} }, ... } },
  L4: { rules: { hard: [...], soft: [...] } },
  meta: { created, last_updated, entry_count, tested_inputs, accuracy }
}
```

**关键设计决策**：每个领域是一个独立字典文件，而不是一个巨型字典文件。这样：
- 添加新领域不需要修改已有领域的字典
- 每个领域可以独立测试准确率
- 字典文件可以独立版本控制

### 7.2 多少个领域够用——L0 穷尽性原理

**L0 的 4 个领域是穷尽的，基于"人类-AI 交互的四种基础动词"：**

| 动词 | L0 领域 | 本质 | 穷尽性证明 |
|------|---------|------|-----------|
| 创造 | creative | 从零到一产出新内容 | 所有"产出新东西"的请求 = creative |
| 构建 | development | 实现功能/系统 | 所有"实现具体功能"的请求 = development |
| 洞察 | analysis | 从数据中提取规律 | 所有"分析已有数据"的请求 = analysis |
| 求知 | research | 收集和整合信息 | 所有"调研收集信息"的请求 = research |

**为什么这 4 个是穷尽的？**

任何一句话任务，必然落在以下四个互斥范畴之一：
- 产出新内容（小说、文章、剧本、营销文案）→ creative
- 实现功能（代码、系统、API、插件）→ development
- 分析数据（报表、图表、归因、趋势）→ analysis
- 调研信息（综述、选型、可行性、最佳实践）→ research

不存在第五个范畴。如果出现"看起来像第五个"的请求，它一定是某个的子类型。例如：
- "帮我管理项目" → 拆解后是"分析进度(analysis) + 产出报告(creative) + 实现功能(development)"
- "帮我设计架构" → 本质是 research（调研对比方案）→ 产出 development

**但 L1-L4 不穷尽，是可扩展的：**

| 层 | 固定还是可扩展 | 扩展方式 |
|----|--------------|---------|
| L0 | 固定 — 4 个领域 | 不可增减，穷尽原理保证 |
| L1 | 可扩展 — 子类型 | 遇到新子类型，新增字典条目 |
| L2 | 可扩展 — 平台/栈 | 新平台出现，新增规则 |
| L3 | 固定 — 正则模式 | 模式库是有限的（数值/范围/枚举/格式） |
| L4 | 可扩展 — 领域规则 | 新领域场景，新增规则集 |

### 7.3 字典的三种生命周期形态

**字典不是"临时替代品"，而是"永久锚定层"。它从不退出，只改变权重。**

```
┌──────────────────────────────────────────────────────────────────┐
│                    字典生命周期三阶段                              │
├───────────────┬──────────────────┬───────────────────────────────┤
│ Phase 1       │ Phase 2          │ Phase 3                       │
│ 纯字典(冷启动) │ 字典+统计(温启动) │ 字典+向量+反馈(热启动)         │
│               │                  │                               │
│ 字典: 100%    │ 字典: 70%        │ 字典: 40%                     │
│               │ 统计: 30%        │ 向量: 40%                     │
│               │                  │ 反馈: 20%                     │
│               │                  │                               │
│ 触发条件:     │ 触发条件:         │ 触发条件:                      │
│ 无历史数据    │ 累积 ≥ 100 次    │ 累积 ≥ 1000 次                │
│ 无向量模型    │ 分类决策         │ 分类决策                       │
│               │                  │ 且向量模型已训练               │
├───────────────┼──────────────────┼───────────────────────────────┤
│ 字典作用:     │ 字典作用:         │ 字典作用:                      │
│ 唯一决策源    │ 主导决策          │ 锚定+安全网+审计               │
│               │ 统计辅助校准      │ 向量并行决策                  │
│               │                  │ 反馈闭环优化                  │
└───────────────┴──────────────────┴───────────────────────────────┘
```

**字典在热启动阶段的四个不可替代作用：**

| 作用 | 机制 | 为什么向量做不到 |
|------|------|-----------------|
| **硬地板** | 字典保证最低准确率，即使向量完全失效 | 向量在未见过的输入类型上可能给出高置信度错误答案 |
| **安全网** | 向量置信度 < 字典置信度时，回退字典 | 向量模型的置信度是"模型觉得自己有多对"，不是"实际有多对" |
| **训练种子** | 字典条目作为正例，训练向量模型 | 没有字典就没有初始训练数据 = 冷启动死锁 |
| **审计轨迹** | 每次决策可追溯到"字典说X，向量说Y，综合选Z" | 纯向量是黑盒，无法解释"为什么选了某个分类" |

### 7.4 字典与向量的协同机制——并行双通道

**关键设计**：字典和向量不是串行（先字典后向量），而是**并行双通道**。两个独立分类器同时运行，交叉验证：

```
输入
  │
  ├─→ 字典通道 ──→ 关键词匹配 ──→ 领域分类 + 置信度 + 匹配证据
  │
  └─→ 向量通道 ──→ 嵌入相似度 ──→ 领域分类 + 置信度 + 相似度分数
        │
        ▼
  ┌─────────────────────────────────────────────────────┐
  │ 合并决策规则（L5 内执行）：                           │
  │                                                     │
  │ 两者一致 → 高置信度，直接通过                        │
  │ 两者不一致 → 字典优先（硬地板原则），但标记 discrepancy │
  │ 字典未覆盖、向量有结果 → 标记 inferred，置信度 = 向量置信度×0.8 │
  │ 两者都无结果 → 双路径并行，等待 L5 评分淘汰           │
  │                                                     │
  │ 输出：classification + confidence + evidence +       │
  │       dictionary_result + vector_result +            │
  │       discrepancy_flag                              │
  └─────────────────────────────────────────────────────┘
```

### 7.5 冷启动截断与断点重续标准

**截断 = 标记后继续，非停止：**

| 截断点 | 触发条件 | 处理方式 | 是否停止引擎 |
|--------|---------|---------|------------|
| L0-T1 | 所有领域 score = 0 | 标记 ambiguous，取最高分两个领域双路径并行 | 否 |
| L0-T2 | 最高分并列 | 双路径并行，不调大模型 | 否 |
| L1-T1 | score < second_best + 1 | 标记 inferred，取最高分 | 否 |
| L3-T1 | 提取率 < 50% | 标记 high_inference_risk | 否 |
| L4-T1 | 硬约束违反 | 该路径淘汰，其他路径继续 | 否 |

**断点重续机制（文件系统驱动）：**

```
引擎启动
  │
  ├─ 检查字典文件 hash → 与上次运行对比
  │   ├─ 字典未变 → 检查各层 produce_file 是否存在且有效
  │   │   ├─ L5 已完成 → 直接返回结果
  │   │   ├─ L3 完成但 L4 未完成 → 从 L4 继续
  │   │   └─ 全部不存在 → 从 L0 开始
  │   └─ 字典已变 → 强制从头执行（字典变了，旧结果可能失效）
  │
  └─ 每次断点前写入 checkpoint 文件：
     { layer: "L3", status: "completed", timestamp, output_hash }
```

**核心原则**：断点记忆在文件系统里，不在引擎内存里。重启不丢。字典文件是唯一真相源。

### 7.6 字典编写规范（新增标准）

**一个合格的字典条目必须满足以下条件：**

| 规范 | 说明 | 验证方式 |
|------|------|---------|
| 关键词互斥性 | 同一领域内的关键词不应在另一个领域中出现 | 跨领域关键词去重检查 |
| 排除词完备性 | 每个领域必须定义排除词，防止误分类 | 人工审查 + 边界测试 |
| 关键词具体性 | 优先使用长尾关键词（如"悬疑推理"优于"悬疑"），越具体权重越高 | 关键词长度加权 |
| 覆盖度可验证 | 每个字典条目记录 tested_inputs 和 accuracy | 回归测试 |
| 版本可追溯 | 每次修改更新 last_updated 和变更日志 | Git diff |

**关键词长度加权规则**：

```
关键词权重 = 1 + (关键词字符数 - 2) × 0.2
例：
  "写"（1字）→ 权重 1.0
  "悬疑"（2字）→ 权重 1.0
  "悬疑推理"（4字）→ 权重 1.4
  "本格推理小说"（6字）→ 权重 1.8
```

**目的**：长尾关键词更具体，命中时给予更高权重，减少误匹配。

### 7.7 穷举维度完备性分析（第 7 轮新增）

> 来源：用户提出"模糊概念应该经历多少个维度的穷举？为什么做这个设定？缺失还能得到唯一正解吗？"

#### 7.7.1 核心问题

当前 5 层收敛架构（L0-L5）是直觉设计的，但从未明确定义：**为什么恰好是 5 层？它们是否正交？缺失任意一层是否还能得到唯一正解？**

#### 7.7.2 正交性证明（10 对全量验证）

两个维度正交 = 改变其中一个不影响另一个。所有 5 个维度之间必须两两正交：

| 维度对 | 正交性证明 | 结论 |
|--------|-----------|------|
| L0 ⟂ L1 | 同一领域（creative）下可有不同子类型（fiction vs article） | 正交 |
| L0 ⟂ L2 | 同一领域（creative）下可有不同平台（起点 vs 晋江） | 正交 |
| L0 ⟂ L3 | 同一领域下可有不同结构参数（1章 vs 10章） | 正交 |
| L0 ⟂ L4 | 同一领域下可有不同规则集（悬疑规则 vs 玄幻规则） | 正交 |
| L1 ⟂ L2 | 同一子类型（fiction）下可有不同平台 | 正交 |
| L1 ⟂ L3 | 同一子类型下可有不同结构参数 | 正交 |
| L1 ⟂ L4 | 同一子类型下可有不同规则集 | 正交 |
| L2 ⟂ L3 | 同一平台（起点）下可有不同字数 | 正交 |
| L2 ⟂ L4 | 平台规则（起点格式）≠ 领域规则（悬疑钩子） | 正交 |
| L3 ⟂ L4 | 结构参数（字数）≠ 领域规则（钩子必须存在） | 正交 |

**全部 10 对均正交。5 个维度相互独立，不可互相推导。**

#### 7.7.3 缺失维度的后果——"漏维即发散"

假设只有 4 层（缺失某一层），会发生什么：

| 缺失层 | 残留不确定性 | 结果 |
|--------|-------------|------|
| 无 L0 | 不知道是创作还是开发 | 两个完全不同的路径树并存，无法收敛 |
| 无 L1 | 知道是创作，不知道是小说还是剧本 | 小说路径和剧本路径并存，规则冲突 |
| 无 L2 | 知道是悬疑小说，不知道目标平台 | 起点规则（2000-2500字/章）和番茄规则（1500-2000字/章）并存 |
| 无 L3 | 知道是起点悬疑小说，不知道写多少章 | 1章和100章都是合法解，质量评分无法区分 |
| 无 L4 | 知道是起点悬疑小说10章，不知道领域规则 | 可能产出没有钩子的章节，违反悬疑类型核心约束 |

**结论：缺少任意一层 → 至少一个维度未收束 → 无限个合法解并存 → 无法得到唯一正解。**

#### 7.7.4 为什么恰好是 5 层——穷举维度完备性证明

这 5 个维度不是随意的，而是穷尽的。任何模糊概念的不确定性，必然落在以下 5 个正交问题之一：

| 维度 | 问题 | 穷尽性论证 |
|------|------|-----------|
| L0 | **做什么领域？** | 所有任务必然属于一个领域。4 个领域穷尽所有人类-AI 交互动词 |
| L1 | **做哪个子类型？** | 每个领域内必然有一个子类型。子类型是领域内的互斥分类 |
| L2 | **在什么环境/平台？** | 所有产出必然有目标环境。不存在"无平台"的产出 |
| L3 | **有多少/多大？** | 所有任务必然有量化参数。至少存在一个可提取的结构参数 |
| L4 | **受什么规则约束？** | 所有领域必然有专属规则。领域规则是"唯一答案"的硬约束 |

**反证法**：假设存在第 6 个正交维度。它必须回答一个与 L0-L4 完全不同的问题。尝试列举：

| 候选维度 | 为什么不是第 6 个正交维度 |
|---------|--------------------------|
| "谁来做？" | 这是执行分配问题（Phase 5 之后），不是结构收敛问题 |
| "什么时候做？" | 这是时序调度问题，不是结构收敛问题 |
| "为什么做？" | 这是 L0 的子问题——领域判定已经包含目的 |
| "怎么做？" | 这是 L1 的子问题——子类型决定方法 |
| "多少钱？" | 这是 L3 的子问题——成本是结构参数的一种 |

**找不到第 6 个正交维度。5 层是完备的。**

#### 7.7.5 数学表达——可能性空间收束

```
模糊概念 U 的不确定性空间 = L0 × L1 × L2 × L3 × L4

其中：
  |L0| = 4     (creative/development/analysis/research)
  |L1| = 4-5   (每个领域下的子类型数)
  |L2| = 3-5   (每个子类型下的平台数)
  |L3| = ∞     (连续参数，但正则提取后可离散化为 ~10 种)
  |L4| = 2-5   (每个子类型的规则数)

总可能性空间 = |L0| × |L1| × |L2| × |L3_discrete| × |L4|
              ≈ 4 × 4 × 4 × 10 × 3 = 1920 种组合（保守估计）
```

逐层收束过程：

```
L0 未收束：可能性 = 1920（全部保留）
L0 收束：  可能性 = 1 × 4 × 4 × 10 × 3 = 480
L0-L1 收束：可能性 = 1 × 1 × 4 × 10 × 3 = 120
L0-L2 收束：可能性 = 1 × 1 × 1 × 10 × 3 = 30
L0-L3 收束：可能性 = 1 × 1 × 1 × 1 × 3 = 3
L0-L4 收束：可能性 = 1 × 1 × 1 × 1 × 1 = 1 ← 唯一正解
L5 锚定：  哈希指纹物理锁定，确保即使 L0-L4 因平手留下 2+ 候选，也强制唯一
```

**L5 是兜底层，不是维度层。** L5 不增加新的不确定性维度，只在 L0-L4 收束后仍有平手时强制唯一化。

#### 7.7.6 与"44 问 8 维度"的关系

Phase 1 的 44 问覆盖 8 个维度（意图、边界、架构、接口、拆解、实现、验证、交付），但那是 **Phase 1 的 spec 细化维度**，不是**Phase 0 的收敛维度**。两者的关系：

```
Phase 0（收敛引擎）：5 个正交维度，粗粒度收束
  "这是什么？" → 领域/子类型/平台/结构/规则
  产出：唯一 spec 骨架

Phase 1（44 问细化）：8 个维度，细粒度填充
  在收敛引擎确定的骨架内，填充具体细节
  产出：完整 spec.json
```

**Phase 0 的 5 维 + Phase 1 的 8 维 = 从模糊到精确的 13 维收束。** 5 维确保方向对，8 维确保细节对。

#### 7.7.7 结论

| 问题 | 答案 |
|------|------|
| 模糊概念需要多少维度穷举？ | 5 个正交维度（L0-L4） |
| 为什么是 5？ | 正交性证明（10 对全正交）+ 完备性证明（反证法找不到第 6 个） |
| 缺失任意层能否得到唯一正解？ | 不能。"漏维即发散"——缺失层 = 该维度未收束 = 无限合法解 |
| 5 维和 44 问 8 维的关系？ | Phase 0 粗粒度收束 + Phase 1 细粒度填充，13 维完整链路 |
