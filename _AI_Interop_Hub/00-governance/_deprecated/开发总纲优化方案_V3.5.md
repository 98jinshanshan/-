## 一、P0 级修复（不修复则总纲核心机制失效）

### S1：监督者形同虚设

**现状：** Agent 自己设计、自己实现、自己验证、自己批准。三次运行 24 个检查点通过率 100%，但实际产出存在大量问题。

**根因：** 监督者和执行者是同一个 Agent。自己审核自己永远通过。

**修复方案：**

**方案 A：硬约束 —— 监督者检查点必须由独立 Agent 执行**

在 `开发总纲_可执行版.json` 的 `goal_alignment_supervisor` 中增加强制规则：

```json
"supervisor_independence": {
  "rule": "监督者检查点必须由与执行 Agent 分离的独立 Agent 执行",
  "implementation": "在执行 Agent 产出每个 Phase 产物后，系统必须调用一个独立的监督 Agent 执行对应的检查点",
  "禁止行为": "执行 Agent 不得在产物中预填 alignment_check 字段",
  "违规后果": "如果执行 Agent 在产物中填写了 alignment_check，监督者必须将其标记为 INVALID 并重新检查"
}
```

**具体改动：**

1. 删除所有产物文件中 Agent 自填的 `supervisor_checkpoint_cp_N` 块
2. 每个 Phase 产物只保留 `goal_trace` 字段，不包含 `alignment_check` 结果
3. 新增 `supervisor_checkpoints/` 目录，每个检查点由独立 Agent 写入独立文件
4. 检查点文件格式：`{checkpoint_id, timestamp, executing_agent, check_result, evidence, pass/fail}`

**监督者独立性如何保证：**
- 在单 Agent 环境下（当前实际情况）：监督者检查点改为**结构化自动化校验**而非"Agent 阅读后判断"。
  - cp_1：检查 path.json 是否存在 → 文件存在性检查（自动化）
  - cp_2：检查 nodes.json 中每个节点的 `maps_to_root_goal` 字段是否非空 → Schema 校验（自动化）
  - cp_3：检查 interfaces.json 中每个接口的 `schema_match` 是否有实际校验逻辑 → 格式校验（自动化）
  - cp_5：检查 HTML 中 Section 数量是否 ≥ nodes.json 规定的节点数 → 内容校验（自动化）
  - cp_7：检查 goal_trace 链是否完整（所有文件是否存在且引用正确）→ 链路校验（自动化）

### S2：递归从未真正执行

**现状：** 总纲定义了 L0→L1→L2→L3→L4 五级递归，但三次运行中递归深度始终为 1。Agent 在 spec.json 中写了一个预拆解，然后直接跳到 Phase 6。

**根因：** Phase 5 的"递归拆解"被 Agent 解释为"在 spec.json 中写一段拆解描述"，而非"对每个子节点创建新的执行上下文并重新走 Phase 1-5"。

**修复方案：**

**核心改动：递归不再是"描述"，而是"强制产出"**

在 `开发总纲_可执行版.json` 的 `phase_5` 中修改：

```json
"phase_5": {
  "id": "recursive_decomposition",
  "name": "递归拆解",
  "type": "recursion_entry",
  "强制规则": {
    "rule_1": "对每个子节点，必须创建独立目录并产出该子节点的 spec.json",
    "rule_2": "子节点 spec.json 的 goal_trace 必须包含父节点的引用",
    "rule_3": "递归终止条件：当前节点被判定为叶节点（不可再分，一个函数/一个内容单元能实现）",
    "rule_4": "如果子节点数 > 3，必须至少递归一层。不得将所有子节点声明为叶节点",
    "rule_5": "递归产物目录结构：{task_dir}/nodes/{node_id}/spec.json"
  },
  "minimum_requirement": "至少产出 1 个子节点的递归 spec.json，否则 cp_4 不通过",
  "cp_4_check": "检查是否存在 nodes/ 子目录且包含至少 1 个 spec.json"
}
```

**实际落地：**
- 对于创作类任务（如小说大纲），递归深度降到 2 层即可（项目级 → 卷级），不需要拆到 4 层
- 对于开发类任务，递归深度维持 4 层
- 在 `level_definitions` 中增加 `applicable_task_types` 字段，区分创作类和开发类的递归深度

## 二、P1 级修复（影响产出质量）

### P4-1：接口契约在迭代时未更新

**现状：** R3 的 spec/path/nodes 更新了，但 interfaces.json 仍是 V1 内容（章节数约束仍是 20-30 而非 400）。

**根因：** Phase 4 在迭代时被 Agent 跳过，因为 Agent 认为"接口不需要变"。

**修复方案：**

**在 Phase 循环中增加"接口一致性检查"**

在 `开发总纲_可执行版.json` 中增加：

```json
"phase_4": {
  "迭代规则": {
    "rule": "每次 spec.json 被更新时，Phase 4 必须重新执行",
    "check": "interfaces.json 中的数值约束（minItems, maxItems, enum）必须与 spec.json 中的 scope 字段一致",
    "示例": "如果 spec.json.scope.target_chapters = 400，则 interfaces.json 中章节数约束必须为 minItems: 350, maxItems: 450",
    "违规检测": "cp_3 增加自动化校验：提取 spec.json 和 interfaces.json 中的关键数值，逐一比对"
  }
}
```

**具体改动：**
1. cp_3 从"Agent 阅读判断"改为"自动化数值比对"
2. 比对项：章节数、伏笔数、人物数、红鲱鱼数、卷数
3. 比对脚本逻辑：`if spec.chapters != interfaces.chapter_constraint → cp_3 FAIL`

---

### P7-2：验证数据与实际内容不符

**现状：** R3 的 verification.json 声称"章节规划模块: tests: 400, failures: 0"，但实际 HTML 只详细列出了 70 章。

**根因：** Agent 把"声明了 400 章结构"当作"验证了 400 章内容"。

**修复方案：**

**cp_6 增加自动化内容校验**

```json
"cp_6": {
  "自动化校验": {
    "html_section_count": "提取 HTML 中所有 h2 标签数量，与 nodes.json 的节点数比对",
    "html_chapter_count": "提取 HTML 中所有章节行（table tr）数量，与 spec.json 的 target_chapters 比对",
    "html_word_count": "提取 HTML 中所有中文文本字数，与 spec.json 的 target_word_count 比对（允许 ±30% 误差）",
    "通过条件": "所有三项比对误差 < 30%"
  }
}
```

**同时修改 Phase 7 的 failure 处理：**
```json
"phase_7": {
  "on_failure": "标记具体哪些指标不达标，返回 Phase 6 重做相关内容，而非下调验收标准",
  "禁止行为": "不得在验收不达标时写'完整 X 章概要已超出单个 HTML 的展示范围'等降标声明"
}
```

## 三、P2 级修复（增强机制完整性）

### S3：去程蓝图不被回程强制执行

**现状：** R3 的 path.json 定义了 7 个节点，但 HTML 实际生成了 10 个 Section。去程设计和回程实现之间松动。

**修复方案：**

**Phase 6 增加"蓝图对照"步骤**

```json
"phase_6": {
  "新增步骤": {
    "step_0": "在开始实现前，读取 path.json 和 nodes.json，提取'必须实现的 Section 列表'",
    "step_0_check": "确认该列表与 spec.json 的 acceptance_criteria 一致",
    "step_final": "实现完成后，列出实际生成的 Section，与蓝图对照",
    "偏差处理": "如果实际 Section 数 > 蓝图 Section 数，必须在 cp_5 中标注'超出蓝图'并给出理由；如果实际 Section 数 < 蓝图 Section 数，cp_5 直接 FAIL"
  }
}
```

---

### P1-1：44 问自审可被压缩为摘要

**现状：** R1 只产出了 8 维度摘要。Agent 在"急于出结果"时会跳过 44 个独立问答。

**修复方案：**

**spec.json 的 self_audit_log 改为结构化强制格式**

```json
"self_audit_log": {
  "强制格式": {
    "dimension_1": { "q1": "必填", "q2": "必填", "q3": "必填", "q4": "必填", "q5": "必填" },
    "dimension_2": { "q1": "必填", "q2": "必填", "q3": "必填", "q4": "必填", "q5": "必填" },
    "...": "8 维度 × 5-7 问 = 44 个独立字段，全部必填",
    "校验": "cp_0 自动化检查：self_audit_log 中必须有 44 个独立字段，缺 1 个 → FAIL"
  }
}
```

**同时修改总纲的 Phase 1 描述：**
```json
"phase_1_autonomous": {
  "agent_workflow": {
    "step_2": "调用自问自答深度检查清单，按 8 维度 44 问逐层自审。必须输出 44 个独立回答，不得合并为 8 维度摘要。"
  }
}
```

## 四、P3 级修复（长期治理）

### S4：产物无版本化

**现状：** R2→R3 时 spec/path/nodes 被原地覆写，V1 版本丢失。

**修复方案：**

```json
"versioning": {
  "rule": "每次 spec.json 被更新时，旧版本移动到 archive/ 目录",
  "archive_path": "{task_dir}/archive/spec_v{version}_{timestamp}.json",
  "current_path": "{task_dir}/spec.json（始终指向最新版本）",
  "manifest_tracking": "manifest.json 的 version_history 字段记录所有版本的关键参数变更"
}
```

---

### P3-1：必要性审计永远全部通过

**现状：** `necessity_audit.all_necessary` 永远为 `true`。

**修复方案：**

将其从"Agent 自评"改为"自动化规则检查"：

```json
"necessity_audit_automated": {
  "rule_1": "每个节点的 maps_to_root_goal 字段必须与 spec.json 的 acceptance_criteria 中的某个维度对应",
  "rule_2": "如果某个节点的 maps_to_root_goal 无法在任何 acceptance_criteria 中找到对应，标记为 extraneous",
  "rule_3": "如果 acceptance_criteria 中某个维度在 nodes.json 中找不到对应节点，标记为 missing",
  "cp_2_check": "must not have extraneous AND must not have missing"
}
```

## 五、新增机制：产物体检报告

在 Phase 8 之后增加一个**自动体检步骤**，不依赖 Agent 自我评估：

```json
"phase_8_health_check": {
  "自动检查项": {
    "文件完整性": "检查 task_dir 下是否有 spec.json, path.json, nodes.json, interfaces.json, verification.json, manifest.json",
    "数值一致性": "检查 spec/path/nodes/interfaces 中的关键数值是否一致",
    "goal_trace 链路": "检查每个文件的 goal_trace.trace_path 是否形成连续链",
    "实际产出 vs 承诺": "检查 HTML 实际内容（Section 数、章节数、字数）是否 ≥ spec.json 承诺的 70%",
    "输出": "{task_dir}/health_check.json，包含所有检查项的 pass/fail 和具体偏差数据"
  }
}
```

## 七、不做什么

以下问题在当前方案中**不修复**，原因如下：

| 问题 | 不修复原因 |
|------|------|
| 监督者需要另一个物理 Agent | 当前单 Agent 环境无法实现。通过自动化校验替代，效果接近 |
| 递归到 L4 层 | 对于创作类任务无意义（卷→章即到叶节点）。通过任务类型分级解决 |
| 置信度由外部校验 | 需要独立评估模型，当前环境不具备。通过自动化数值校验替代 |
| 完整 400 章概要 | 400 章概要写出来约 10 万字，超出单次对话的合理输出范围。改为"每卷详细前 20 章 + 其余章节声明模板" |
