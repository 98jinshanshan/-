## 核心契约：一句话的定义

```
你说的"一句话" = 一个模糊的想法，例如："开发节点1的题材挖掘功能"

Agent做的事：
  0. Phase 0：Agent 先通过领域分类器确定任务领域（创作/开发/分析/研究）和子类型，选择对应的领域问题包。
  1. Phase 1：Agent 使用领域问题包 + 44 问完整性检查，自问自答，产出根目标 spec.json。不需要你参与。
  2. 目标对齐监督者启动，锁定根目标为唯一校验基准。
  3. 去程（Phase 2-5）：从项目级 → 宏观节点 → 模块 → 子节点 → 叶节点，逐层拆解。每个 Phase 使用对应领域的变体定义。全自动。
  4. 回程（Phase 6-8）：从叶节点 → 子节点 → 模块 → 宏观节点 → 全链路，逐层合成验证。每个 Phase 使用对应领域的变体定义。全自动。
  5. 每一层、每一个节点走完8个阶段后，监督者立即校验目标对齐。
  6. 全链路通路 + 全节点对齐 = 开发完成。

你参与的次数：0。
全流程 Agent 自动完成，不需要你确认、不需要你回答、不需要你参与。
```

## Phase 0：约束收敛引擎（V3.9.0 替换 Phase 0）

> 在 Phase 1 之前执行。**不是 Agent 问问题靠主观判断，而是 5 层确定性筛选算法。** 字典是永久锚定层，经历三阶段生命周期，从不退出。同一输入永远产生同一输出。

### 5 层收敛架构

```
输入一句话
    ↓
L0 领域分类：关键词匹配 + 排除词过滤 + 长度加权计分
    score = Σ(weight × keyword_hit) - Σ(exclusion_hit)
    阈值：score ≥ second_best + gap_threshold
    滞后：领域切换需 gap ≥ 2 才触发
    收束：4 大类 → 1-2 个领域
    ↓
L1 子类型约束：领域内关键词匹配 + 长度加权
    阈值：score ≥ second_best + 1
    收束：N 种子类型 → 1 种子类型
    ↓
L2 平台/技术栈：平台关键词匹配 → 应用平台规则
    无匹配 → 通用默认规则，标记 inferred
    收束：平台参数全部锁定
    ↓
L3 结构参数提取：正则提取数量/范围/枚举/格式
    提取率 ≥ 50% → 继续；< 50% → 标记 high_risk
    收束：所有结构参数锁定为数值/范围
    ↓
L4 领域规则校验：硬约束（违反淘汰）+ 软约束（违反扣分）
    查询领域规则数据库，全部 applicable 规则必须应用
    收束：违反硬约束的路径被淘汰
    ↓
L5 唯一性锚定：质量评分排序 + SHA-256 哈希兜底
    Score = 0.3×完整性 + 0.3×冲突率 + 0.2×精确度 + 0.2×规则合规度
    平手 → SHA-256 哈希 + 字典序取第一
    每次决策输出审计日志
    收束：N 条合法路径 → 1 条唯一路径
    ↓
输出唯一 spec.json（附带 confidence 字段）
```

### 字典三阶段生命周期

| 阶段 | 触发条件 | 字典权重 | 机制 |
|------|---------|---------|------|
| 冷启动 | 无历史数据，无向量模型 | 100% | 纯字典匹配，唯一决策源 |
| 温启动 | 累积 ≥ 100 次分类决策 | 70% | 字典主导，统计辅助校准关键词权重 |
| 热启动 | 累积 ≥ 1000 次决策且向量模型已训练 | 40% | 字典+向量并行双通道，字典为硬地板和安全网 |

**字典在热启动阶段的四个不可替代作用**：硬地板（保证最低准确率）、安全网（向量失效时回退）、训练种子（字典条目作为正例训练向量）、审计轨迹（每次决策可完整追溯）。

### 冷启动截断与断点重续

截断 = 标记后继续，非停止。五个截断点：L0 所有 score=0 → ambiguous 双路径；L0 并列 → 双路径；L1 score < second_best+1 → inferred；L3 提取率 < 50% → high_risk；L4 硬约束违反 → 淘汰。

断点重续：引擎启动 → 检查字典文件 hash → 与上次对比。字典未变 → 从已完成层继续。字典已变 → 强制从头执行。断点记忆在文件系统，重启不丢。

### 收敛速度可配置

| 级别 | gap_threshold | min_confidence | 适用场景 |
|------|--------------|---------------|---------|
| conservative | 2 | 0.9 | 量化风控、合规审查 |
| medium | 1 | 0.7 | 创作拆解、开发任务（默认） |
| aggressive | 0 | 0.5 | 探索性任务、快速原型 |

### 为什么这样设计

小说测试中，6/44 个推断项中有 4 个是悬疑小说专属的。旧方案靠 Agent 问问题，不仅慢，而且不同 Agent 可能问出不同结果。收敛引擎用关键词字典（如"悬疑""推理""凶手""诡计"）直接命中子类型，用领域规则（每章必须有钩子）直接锁定硬约束。**算出来的，不是猜出来的。**

### 穷举维度完备性：为什么恰好是 5 层？

**核心原理**：一个模糊概念的不确定性分布在 5 个正交维度上。5 个维度必须全部收束，才得到唯一正解。缺少任意一个 = "漏维即发散"。

| 维度 | 问题 | 正交性 | 缺失后果 |
|------|------|--------|---------|
| L0 领域 | 做什么？ | 与 L1-L4 全正交 | 创作和开发路径并存，无法收敛 |
| L1 子类型 | 哪个类型？ | 与 L0/L2/L3/L4 正交 | 小说和剧本路径并存，规则冲突 |
| L2 平台 | 什么环境？ | 与 L0/L1/L3/L4 正交 | 起点和番茄规则并存 |
| L3 结构 | 多少/多大？ | 与 L0/L1/L2/L4 正交 | 1章和100章都是合法解 |
| L4 规则 | 什么约束？ | 与 L0/L1/L2/L3 正交 | 可能产出违反类型规则的产物 |

**10 对维度两两正交，全部通过验证。反证法找不到第 6 个正交维度。5 层是完备的。**

**数学表达**：可能性空间 = 4 × 4 × 4 × 10 × 3 ≈ 1920 种组合。逐层收束：1920 → 480 → 120 → 30 → 3 → 1。L5 是兜底层（哈希锚定），不增加新维度，仅在 L0-L4 平手时强制唯一。

**与 Phase 1 的关系**：Phase 0 的 5 维（粗粒度收束）+ Phase 1 的 44 问 8 维（细粒度填充）= 13 维完整收敛链路。5 维确保方向对，8 维确保细节对。

## 对话级自审条款（V3.8.0 新增）

> Agent 在每次回答前，必须自问以下三问。任意一项为"是"→ 回答已偏离，重写。

**三问**：

1. **我的回答是否直接回应了用户的一句话？** — 如果回答在分析背景、展开相关话题，但没有先给出用户要的答案 → 偏离。

2. **我是否引入了用户没问的新概念？** — 如果回答中出现了用户原问题里没有的新术语、新框架，且不是回答问题的必要部分 → 偏离。

3. **我是否在不必要的时候把决策推回给了用户？** — 如果回答结尾是"你想让我...？""你觉得...？"而不是直接给出结论并自动推进 → 偏离。

**典型案例**：用户问"其他 Phase 有没有同样问题？"，Agent 回答是"逐 Phase 分析了漏洞，然后引入了领域翻译层新概念，最后问用户要不要继续"——三问全部失败。没有先给出明确结论（q1 失败），引入了用户没问的新概念（q2 失败），把决策推回给用户（q3 失败）。

## 目标对齐监督者（防漂移机制）

### 为什么需要

```
梨树嫁接的三种事故：

树根嫁接（Phase 1 理解偏差）：
  你说"题材挖掘" → Agent理解为"关键词提取" → 整棵树长成关键词提取器
  监督者阻止：Phase 1 产出 spec.json 后，监督者校验 44 问自审日志与根目标的一致性

树干嫁接（Phase 2-3 路径/节点偏差）：
  根目标"题材挖掘" → 路径图里少了一个平台 → 数据源不完整
  监督者阻止：每层 Phase 2-3 产出后，监督者反向追溯"这条路径能否回到根目标"

树枝嫁接（Phase 4-5 接口/拆解偏差）：
  接口定义"输出热度排名" → 拆解时变成"输出话题列表"
  监督者阻止：每层 Phase 4-5 产出后，监督者校验接口契约是否服务于根目标

树梢嫁接（Phase 6-8 实现偏差）：
  蓝图要求"链接验证" → 代码实现成"链接去重"
  监督者阻止：每层 Phase 6-8 产出后，监督者校验代码行为是否匹配蓝图定义
```

### 监督者角色

监督者是一个独立的 Agent 角色（OpenClaw 兼任），它不参与任何执行，只做一件事：

**拿着根目标（Phase 1 的 spec.json.one_liner），在每一个检查点问同一个问题："这个节点的产出，能回到根目标吗？"**

- 能 → 放行
- 不能 → 拒绝，标记偏差点，强制该节点从 Phase 1 重新执行

### 检查点（V3.9.1 全部量化）

```
根目标锁定（Phase 1 项目级完成后）
  │
  ├── 检查点 0：Phase 1 产出自身是否自洽？
  │   量化标准：44 问自审日志中 inferred 比例 < 30%
  │              consistency_check 5 条规则中 hard 规则全部通过
  │              hard_constraints 4 条全部满足
```

### 统一失败处理（V3.9.1 新增）

所有 8 个检查点适用同一套失败处理策略：

- **重试**：同一 Phase 同一检查点最多重试 3 次
- **重试耗尽**：广播 `error.blocked` → 记录阻塞原因。关键路径（cp_0、cp_7）→ 上报用户请求决策。非关键路径（cp_1-cp_6）→ 标记 blocked 但允许其他分支继续
- **升级**：连续 3 次 fail 且为关键路径 → 上报用户

### 目标追溯链（Goal Trace Chain）

## 双程递归模型：果树的生长

```
┌─────────────────────────────────────────────────────────────────┐
│                        去程（拆解程）                              │
│              种子 → 扎根 → 发芽 → 长大                            │
│          Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5         │
│         "做什么"  "路怎么走"  "有哪些节点"  "怎么连"  "拆到不可再分" │
│                                                                  │
│   项目级 ──→ 宏观节点 ──→ 模块 ──→ 子节点 ──→ 叶节点             │
│   (Level 0)   (Level 1)   (Level 2)  (Level 3)   (Level 4)      │
│   粗                                                                细 │
│                                                                  │
│   ════════════════════════════════════════════════════════════   │
│                                                                  │
│                        回程（合成程）                              │
│              成熟 → 开花 → 结果                                   │
│          Phase 6 → Phase 7 → Phase 8                              │
│         "实现集成"  "通路验证"  "交付收尾"                          │
│                                                                  │
│   叶节点 ──→ 子节点 ──→ 模块 ──→ 宏观节点 ──→ 全链路             │
│   (Level 4)   (Level 3)  (Level 2)  (Level 1)    (Level 0)      │
│   细                                                                粗 │
│                                                                  │
│   ════════════════════════════════════════════════════════════   │
│                                                                  │
│              目标对齐监督者（全程值守）                             │
│   检查点0 → 检查点1 → 检查点2 → 检查点3 → 检查点4                 │
│   检查点5 → 检查点6 → 检查点7                                     │
│   每层每节点每阶段产出后，校验：能不能回到根目标？                   │
│   不能 → 拒绝 + 强制重做                                          │
└─────────────────────────────────────────────────────────────────┘
```

**关键规则**：
- **去程不写代码，只做设计**：Phase 1-5 产出的是 spec、路径图、接口契约、拆解方案，不写一行代码。
- **回程不设计，只做实现**：Phase 6-8 按照去程的蓝图，从叶节点开始写代码、集成、验证、交付。
- **监督者全程值守**：每一个节点产出一个 Phase 的产出物，监督者立即校验。通过才放行，不通过则强制重做。
- 所有层级（包括项目级）的 Phase 1 都是 Agent 自问自答，不需要你参与。
- 去程到达叶节点（不可再分）后，立即转为回程。
- 回程到达全链路通路 + 全节点对齐后，开发完成。

### 双程边界违规判定（V3.9.1 新增）

> 回程中 Agent 发现去程设计错误时的处理规则。

回程 Agent 发现设计错误 → **不得自行修改设计**。必须：
1. 标记 `suspect_design_issue`
2. 通过 Bridge 上报 supervisor
3. supervisor 决定是否回退到去程重新设计

**不是违规**：回程中补充蓝图未覆盖的实现细节（如错误处理、日志格式）
**是违规**：回程中修改 spec.json 的 acceptance_criteria、修改 path.json 的路径结构、修改 interfaces.json 的接口签名

**升级路径**：Agent → Bridge → supervisor → 评估 → 回退去程 or 标记为 known_issue 继续

## V3.9.1 GAP 修复补全：Phase 1-8 量化标准

> 基于 5D 收敛引擎对总纲 18 个节点的自诊断，Phase 1-8 在 D-L3（结构参数）和 D-L4（规则约束）上存在 25 个维度缺失。以下为修复后的量化标准摘要。完整定义见 `开发总纲_可执行版.json`。

### Phase 1：一致性校验 + 硬约束

**44 问内部一致性校验（5 条规则）**：

| 规则 | 检查内容 | 严重度 |
|------|---------|--------|
| CC-1 频率一致性 | q12(更新频率) 与 q28(交付节奏) 数值必须一致 | hard |
| CC-2 范围一致性 | q3(范围边界) 与 q44(交付物清单) 枚举不矛盾 | hard |
| CC-3 格式一致性 | q22(输出格式) 与 q41(交付格式) 一致 | soft |
| CC-4 数量一致性 | q13(规模预估) ≥ q35(最小交付量) | hard |
| CC-5 角色一致性 | q8(目标用户) 与 q19(接口消费者) 兼容 | soft |

**硬约束（4 条，违反 → cp_0 FAIL）**：
- 核心维度不可全推断：dimension_1 和 dimension_2 的 inferred 比例不得同时 > 50%
- 44 问完整性：self_audit_log 必须包含 44 个独立字段
- 一致性校验零 hard 失败：所有 severity=hard 的规则必须全部通过
- one_liner 不可为空：≥ 10 个字符

### Phase 2：路径结构参数 + 硬约束

- **节点数范围**：min=3, max=15（由 Phase 0 L3 结构参数动态调整，如 L3 提取 10 章则 max=15）
- **粒度标准**：一个节点 = 一个独立可交付的产出物，有明确的输入和输出
- **硬约束**：连通性（无断点）、无循环（A→B→C→A）、终点对齐根目标、节点数在范围内

### Phase 3：节点结构参数 + 重叠检测

- **每层级节点数**：宏观节点 2-5、模块 2-8、子节点 2-6（由领域变体进一步细化）
- **重叠检测**：两个节点映射到同一 acceptance_criteria 维度且无明确分工差异 → 标记 redundancy → 合并或重做

### Phase 4：字段完整性 + 语义匹配

- **字段完整性阈值**：consumer.consumes / producer.outputs ≥ 80%。低于 80% → 标记 data_loss_risk
- **同义字段映射**：已知同义字段（如 title/name/heading、id/identifier/uuid）自动匹配
- **类型转换**：允许 string↔number，禁止 object→string（信息丢失）和 array→object（结构不兼容）

### Phase 5：原子性标准 + 覆盖度校验

- **原子性 3 条客观标准**（与 Phase 0 L3 原子性标准统一）：
  - ATOM-1 单一交付物：有且仅有一个明确的业务交付物
  - ATOM-2 无跨节点前置依赖：可独立启动执行
  - ATOM-3 可量化验收：验收标准能用数字表达
  - 三条全部满足 → 叶节点，去程终点
- **覆盖度**：子节点覆盖的维度并集 / 父节点 acceptance_criteria 维度 ≥ 100%

### Phase 6：质量指标 + 偏差分类

**量化指标**：开发类（测试覆盖率 ≥ 80%、lint 零 error、构建成功）、创作类（字数偏差 ≤ 30%、章数 ≥ 承诺 80%）

**偏差分类**：
- 正向纠偏（Agent 发现蓝图遗漏，主动补充）→ 标记 [ADDED]，不 FAIL
- 偏离（未标记且与蓝图不一致）→ cp_5 FAIL

### Phase 7：验证阈值 + 降标判定

**五层验证阈值**：叶节点单元测试（覆盖率 ≥ 80%，≥ 3 用例）→ 子节点集成（≥ 3 用例，通过率 100%）→ 模块集成（≥ 3 用例，通过率 100%）→ 宏观节点（通过率 100%）→ 项目 E2E（通过率 100%）

**降标判定**：偏差 < 10% 且不影响核心 acceptance_criteria → 合理调整。偏差 ≥ 10% 或影响核心功能 → 降标，cp_6 FAIL。禁止使用"完整 X 章概要已超出单个 HTML 的展示范围"等降标声明。

### Phase 8：可运行性检查

- 开发类：构建成功 + 应用启动无致命错误 + 关键路径可跑通
- 创作类：内容能正常解析（HTML 能打开、Markdown 能渲染）
- 任一 hard 检查失败 → cp_7 FAIL

### Sprint Contract 引擎（V3.9.1 新增）

每次任务启动时自动生成验收契约。acceptance_criteria 从 Phase 0 收敛引擎自动派生：
- L4 hard 约束 → Contract 的 must_have 验收项
- L4 soft 约束 → Contract 的 should_have 验收项
- L3 结构参数 → Contract 的 numeric_targets

Contract 检查：acceptance_criteria 是否包含 Phase 0 L4 的所有 hard 约束？缺少 → 拒绝 contract。

### 递归架构联动（V3.9.1 新增）

- **深度联动**：max_depth = ceil(log2(L3_extracted_chapter_count)) + 1。L3 未提取 → 使用领域变体默认值
- **终止条件统一**：递归终止条件 = Phase 5 原子性标准 ATOM-1/2/3。不在收敛引擎之外定义第二套原子性标准
- **跨领域处理**：子节点 maps_to_root_goal 指向不同领域 → 重新执行 Phase 0 收敛引擎

## Phase 门禁引擎（V3.6 新增，不可跳过）

### 为什么需要

V3.4 的三次运行中，Phase 5（递归拆解）、Phase 7（通路验证）、Phase 8（交付收尾）和全部 8 个检查点被系统性跳过。根因：总纲是规范文档，不是可执行引擎——Agent 天然跳过"不直接产出代码/内容"的 Phase。

### 核心原理

**门禁即执行。** Agent 不需要"理解"Phase 的价值——它只需要知道：不产出 gate_file 就无法进入下一 Phase。

```
每个 Phase 读取 gate_file（前置条件），写入 produce_file（强制产出）。
produce_file 是下一 Phase 的 gate_file。
```

### 门禁链

```
Phase 1 → spec.json          gate: 无（起点）
Phase 2 → path.json          gate: spec.json 存在
Phase 3 → nodes.json         gate: path.json 存在
Phase 4 → interfaces.json    gate: nodes.json 存在
Phase 5 → leaf_nodes.json    gate: interfaces.json 存在（创作类简化）
Phase 6 → 最终产出            gate: interfaces.json + leaf_nodes.json 存在
Phase 7 → verification.json  gate: 最终产出存在
Phase 8 → manifest.json      gate: verification.json 存在且 passed
        + health_check.json
```

### 反跳过机制

如果 Phase 6 的前置 gate_file（interfaces.json 或 leaf_nodes.json）不存在，Agent 不得执行 Phase 6，必须从缺失的最早 Phase 开始。连续 2 个 produce_file 缺失 → 认定为系统性故障。

### 单 Agent 自检模式

当前环境没有独立监督者 IDE 实例。每个 Phase 完成后，Agent 必须以"审核者"身份（不是"作者"身份）自检，写入 `cp_N.self_check.json`。自检不通过 → 标记 failed → 重做。同一 Phase 连续 3 次 failed → 标记 blocked，通知用户。

### 创作类任务简化

创作类任务（如小说正文）递归深度为 2 层。如果节点数 ≤ 6 且每个节点可直接映射到内容单元，可以不递归创建 nodes/ 子目录，但必须显式声明叶节点列表并写入 `leaf_nodes.json`。

### 强制启动流程

Agent 启动任何任务前，必须执行：
1. 读取 `开发总纲_可执行版.json` 的 `phase_gate_engine` 章节
2. 列出该任务类型的所有 gate_file 和 produce_file
3. 从 Phase 1 开始，逐个检查 gate_file 是否存在
4. gate_file 不存在 → 该 Phase 是当前 Phase，开始执行
5. 执行完成后，写入 produce_file → 进入下一 Phase

## 8阶段定义（适用于每一层）

### 去程：Phase 1-5（拆解程 — 只设计，不写代码）

#### Phase 1：意图锚定（全自动自问自答）

**所有层级**：Agent 自问自答，不需要你参与。

**项目级**：Agent 调用 `开发总纲_问答题框架.md`（Phase 1 自问自答深度检查清单），按 8 个维度、44 个问题逐层自审。自审完成后产出 spec.json（含 `goal_trace[root]` 标记为根目标），自动进入检查点 0。

**递归层级**：Agent 读取父节点的 spec，自问自答确认本节点的目标。问题模板中新增第5问："我的目标能否反向追溯到根目标？如果能，追溯路径是什么？" 写入本节点的 spec.json 的 `goal_trace` 字段。

**自审工具**：`开发总纲_问答题框架.md` —— 44 个自审问题，覆盖意图、边界、架构、接口、拆解、实现、验证、交付 8 个维度。

**监督者检查点 0**：校验 Phase 1 产出。项目级：44 问自审日志中 inferred 比例是否 < 30%？递归层级：goal_trace 能否反向追溯到根目标？不能 → 拒绝，强制重做 Phase 1。

#### Phase 2：路径锚定

**做什么**：从起点到终点画出一条完整的路径图。

- 输入：Phase 1 产出的 spec.json
- 输出：完整路径图（起点→经过哪些节点→终点）
- 验证：路径是否连通？有没有断点？
- **产出物**：path.json（路径图，不写代码）

**监督者检查点 1**：校验 path.json。路径终点是否等于根目标描述的最终产出？路径能否反向追溯到根目标？不能 → 拒绝，强制重做 Phase 2。

#### Phase 3：节点识别

**做什么**：识别路径上的所有节点，划分层级。

- 输入：路径图
- 输出：节点清单（当前层级的子节点列表，每个节点的输入输出）
- 验证：每个节点都是路径上必要的吗？有没有遗漏？
- **产出物**：nodes.json（节点清单，不写代码）

**监督者检查点 2**：校验 nodes.json。每个节点是否映射到根目标的某个必要组成部分？有无与根目标无关的节点（过度拆解）？不能 → 拒绝，强制重做 Phase 3。

#### Phase 4：接口契约

**做什么**：定义相邻节点之间的接口——输入什么、输出什么、什么格式。

- 输入：节点清单
- 输出：每对相邻节点的输入/输出Schema（JSON Schema格式）
- 验证：上游输出Schema = 下游输入Schema？
- **产出物**：interfaces.json（接口契约，不写代码）

**监督者检查点 3**：校验 interfaces.json。接口的输入输出方向是否服务于根目标的数据流方向？上游输出 Schema 是否等于下游输入 Schema？不能 → 拒绝，强制重做 Phase 4。

#### Phase 5：递归拆解

**做什么**：对每个子节点，递归进入下一层，重新走去程（Phase 1-5）。

- 递归深度：项目 → 宏观节点 → 模块 → 子节点 → 叶节点（不可再分）
- 递归终止条件：节点不可再分（一个函数能实现）→ 到达去程终点，转为回程
- **关键**：每层递归都走 Phase 1-5 + 检查点 0-4，不是只走 Phase 5
- **产出物**：子节点递归结果（spec + path + nodes + interfaces，不写代码）

**监督者检查点 4**：校验递归拆解结果。子节点集合是否完整覆盖父节点的功能需求？有无多余子节点？不能 → 拒绝，强制重做 Phase 5。

---

### 回程：Phase 6-8（合成程 — 只实现，不设计）

#### Phase 6：实现集成

**做什么**：从叶节点开始，按照去程的蓝图，自底向上实现和集成。

- 叶节点：写代码实现该节点
- 父节点：等待所有子节点回程完成后，集成子节点的产出物
- **关键**：回程不重新设计，严格按去程的 spec + interfaces 执行
- **产出物**：代码

**监督者检查点 5**：校验代码行为。代码实现的功能是否与蓝图（spec）定义一致？是否实现了蓝图定义的全部功能，没有遗漏？代码行为有没有超出蓝图（自行添加功能）？不能 → 拒绝，强制重做 Phase 6。

#### Phase 7：通路验证

**做什么**：每一层集成后，验证通路。

- 叶节点通路：单元测试通过（正常/异常/边界）
- 子节点通路：其下所有叶节点集成测试通过
- 模块通路：其下所有子节点集成测试通过
- 宏观节点通路：其下所有模块集成测试通过
- 全链路通路：端到端测试通过

**验证标准**：数据能从起点流到终点，每个节点产出格式正确，异常情况有降级策略。
**失败处理**：自动修复，最多3次。3次仍失败 → 标记failure，报告父节点。

**监督者检查点 6**：校验测试用例。测试用例的验收标准是否与根目标的验收标准对齐？测试覆盖是否包含了根目标的所有核心功能？不能 → 拒绝，强制重做 Phase 7。

#### Phase 8：交付收尾

**做什么**：更新索引、生成manifest、归档、标记本节点通路完成。

- 每层通路验证通过后，该层的交付物就绪
- 全链路通路后，项目级交付完成
- **产出物**：manifest.json、代码、测试报告、INDEX.md更新

**监督者检查点 7**：最终校验。最终交付物能否完整回答根目标？能否从最终交付物反向追溯到根目标的 one_liner？goal_trace 全链路是否完整无断点？不能 → 拒绝，强制重做 Phase 8。

## 通路验证规则

### 单节点通路

```
节点通路 = 该节点的Phase 7测试全部通过 + 监督者检查点 6 通过
         = 正常路径测试通过 + 异常路径测试通过 + 边界路径测试通过
```

### 层级通路（回程逐层验证）

```
叶节点通路 = 所有叶节点全部通路 + 全部检查点通过
子节点通路 = 该子节点下所有叶节点通路 + 集成测试通过 + 全部检查点通过
模块通路   = 该模块下所有子节点通路 + 集成测试通过 + 全部检查点通过
宏观节点通路 = 该宏观节点下所有模块通路 + 集成测试通过 + 全部检查点通过
全链路通路 = 所有宏观节点通路 + 端到端测试通过 + 全部检查点通过
```
