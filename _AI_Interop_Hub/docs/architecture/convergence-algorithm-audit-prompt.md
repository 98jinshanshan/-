## 一、算法原生定义（归零对齐）

### 1.1 算法本名

**任务穷举双向收敛拆解算法**（Task Exhaustive Enumeration Bidirectional Convergence Decomposition Algorithm）

完整逻辑链：海量合法路径穷举 → 多层规则过滤择优 → 双向校验收敛至唯一交付成果。

### 1.2 两大物理特性（仅此而已，无任何附加含义）

| 特性 | 含义 | 通俗类比（仅用于解释，非算法定义） |
|------|------|----------------------------------|
| 穷举层特性 | 复杂父节点任务在合法规则下存在海量、多维度、多组合的合规拆分方案，合法候选路径数量极大，无法人工穷尽，必须机器量化枚举 | "三千"——形容数量多 |
| 收敛层特性 | 无论合法拆分候选集多么庞大、拆分路径多么多样，所有合规路径最终必须收敛至同一顶层交付成果与验收标准，输出唯一有效拆解结构 | "唯一"——形容结果收敛 |

**此前错误**：将上述通俗类比固化为算法名称（"大道唯一版"）、版本标识、顶层设计理念。这属于典型的过度解读、语义嫁接，完全偏离了原生算法定义。现已彻底删除所有绑定冠名、主题定义、版本命名。

### 1.3 八大原始缺陷（精准无偏差复盘）

1. 枚举无边界，无限穷举导致算法不可执行
2. goal_trace 追溯无量化算法，依赖主观判断
3. 双优化目标权重冲突，无博弈规则
4. 原子性耦合 Agent 能力，破坏结构唯一性
5. 对"穷举多、收敛一"的比喻理解错位
6. 多合法路径共存时，唯一性收敛逻辑缺失
7. 仅有自上而下拆解，缺失自下而上合成收敛
8. 无模糊输入预处理，边界模糊场景直接失效

## 四、七项标准化优化方案（纯工程逻辑，零附加值释义）

### 优化一：量化枚举边界

**解决的问题**：无合法拆分边界定义，机器穷举范围无限膨胀，算力溢出、不可落地。

**方案**：二元量化过滤规则，将全局候选集上限锁定 3000 条。

| 类别 | 判定标准 |
|------|----------|
| 合法保留 | 语义闭环、分工互斥、粒度合规、拆解逻辑统一（时序/角色/阶段/资源/成果模块五类范式之一） |
| 直接剔除 | 语义无关、循环依赖、粒度两极（一步到原子或一步到顶层）、逻辑混杂（混合多套拆解逻辑） |

**工程目的**：服务于"海量穷举可控"的工程需求，无任何附加主题含义。

### 优化二：goal_trace 量化语义校验

**解决的问题**：语义包含、链路追溯依赖主观判定，多 Agent 校验结果不一致，破坏收敛唯一性。

**方案**：三层客观算法——
1. 需求要素抽取：从根目标固化核心验收要素（5W1H + 可量化验收指标），作为全链路语义基准
2. 链路向量化匹配：BGE-M3 模型计算"子节点→父节点→根节点"语义向量相似度，阈值 ≥ 0.85
3. 连续性校验：遍历完整 goal_trace 链路，确认无断层、无语义反转、无核心需求偏移

**工程目的**：保证海量穷举路径中，仅保留对齐顶层目标的合规路径，为最终收敛提供底层筛选保障。

### 优化三：分层加权决策

**解决的问题**："最少节点数"与"最大可验证性"天然冲突，无优先级规则导致择优混乱。

**方案**：两层决策——
- 第一层（硬约束前置）：可验证性一票否决，直接淘汰存在任意子节点无法数值化验收的拆分方案
- 第二层（加权打分）：综合得分 = 可验证性得分 × 0.7 − 节点数量惩罚分 × 0.3

**工程目的**：从海量候选路径内部择优，从三千级候选集中筛选最优唯一解，是收敛核心决策层。

### 优化四：原子性标准解耦

**解决的问题**：原子任务定义绑定模型执行能力，不同 Agent 拆解结果不一致，破坏收敛唯一性。

**方案**：拆分客观结构原子标准（全局唯一不变）与模型执行适配层（灵活可变）——

**客观原子性结构标准（全局不变）**：
- 有且仅有一个明确的业务交付物
- 无跨节点前置依赖，可独立启动执行
- 产出有清晰、可量化的验收指标

**模型适配层（仅调整执行步骤，不改动结构）**：
- 高能力 Agent：直接执行原子节点
- 低能力 Agent：在执行阶段拆分为内部执行子步骤，上报进度仍以原子节点为基准

**工程目的**：无论穷举路径多少、执行模型能力差异多大，最终拆解结构唯一、成果唯一。

### 优化五：多路径收敛逻辑标准化

**核心纠偏（本次最关键整改）**：

1. 彻底删除所有"大道唯一版""大道体系"等错误冠名与定义
2. 正确逻辑：
   - 现象：合法拆分路径海量多样（此前类比的"三千"，仅为数量描述）
   - 结果：所有有效路径择优、校验、合成后，成果收敛唯一（此前类比的"唯一"，仅为结果描述）
3. 算法不存在任何"大道体系设计"，仅存在"海量穷举→择优收敛"的纯工程逻辑

**多合法路径处理规则**：候选集择优主链路优先 + 次优链路兜底校验 + 成果收敛判定。结构择优唯一、成果绝对唯一。

### 优化六：自底向上合成收敛（双向闭环）

**解决的问题**：单向下拆存在隐蔽缺陷——单链路追溯合法，多节点合成后无法覆盖父目标。

**方案**：两步合成校验——
1. 逐层合成产出：从原子节点开始，将同属一个父节点的所有子节点产出按业务规则拼接
2. 合成校验判定：覆盖度 100% + 冲突率 0%
3. 回退容错：不通过则回退至上一级节点，从候选集选择次优方案重新校验

**工程目的**：解决"多路径拆解合法、落地失效"的核心工程问题，保障海量穷举路径最终落地唯一有效。

### 优化七：三级模糊需求预处理

**解决的问题**：原始输入模糊无验收标准，导致全链路校验失效、穷举发散。

**方案**：
1. 一级质询补全：Agent 主动反向质询核心信息（交付物类型、适用场景、量化验收指标、业务约束条件、上下游依赖）
2. 二级标准化固化：将用户补充后的需求转化为结构化根目标（根目标 ID、目标描述、验收标准数组、业务边界、依赖关系）
3. 三级模糊度拦截：计算需求模糊度得分，超过阈值则终止拆解并提示用户补充关键信息

## 五、完整算法流程（七步闭环）

```
1. 模糊需求预处理（输入标准化）
   ↓
2. 合法拆分边界过滤（可控海量穷举）
   ↓
3. goal_trace 量化语义链路校验（筛选合规路径）
   ↓
4. 分层加权多目标择优（海量路径选最优）
   ↓
5. 客观原子级任务拆解（固定唯一结构）
   ↓
6. 自底向上双向合校（最终成果收敛）
   ↓
7. 输出唯一合规任务拆解树
```

## 九、待审阅问题（第二轮）

基于完全归零后的算法定义，请审阅以下问题：

1. 七步闭环流程中，步骤 2 的"合法拆分边界过滤"和步骤 3 的"语义链路校验"是否存在重复过滤？两者筛选的维度是否正交？

2. BGE-M3 向量相似度阈值 0.85——对创作类任务（小说、文章），语义空间比开发类任务更"软"，0.85 是否会导致创作类任务中合法拆分被过度过滤？

3. 原子性客观标准的第三条"产出有清晰、可量化的验收指标"——对创作类任务，"可量化"的定义是什么？小说的"一章"可量化（字数），但"文学质量"不可量化。创作类任务如何定义原子性？

4. 分层加权打分中 0.7 vs 0.3 的权重——不同领域（创作 vs 开发 vs 分析）是否需要不同的权重配置？

5. 三级模糊需求预处理中，"模糊度得分"的计算公式和阈值（默认 30 分）的设定依据是什么？如果用户输入极度模糊（如"帮我写个好东西"），算法是终止还是引导用户缩小范围？

6. 合成校验中"覆盖度 100%"和"冲突率 0%"——是否过于理想化？真实场景中是否存在"覆盖度 95% 但剩余 5% 不影响核心目标"的合理情况？

7. 双向合校的回退容错机制——如果所有候选方案都无法通过合成校验，算法应如何处理？是否应该触发拆分逻辑的重新穷举？

8. 本算法与"开发总纲"的关系定位——本算法是总纲的底层拆分引擎，还是总纲本身就是本算法的一个应用实例？两者之间的接口是什么？
