# 自适应阈值体系架构方案

> 总纲普适性核心改造 — 从"硬编码140+阈值"到"元参数推导+校准闭环"

---

## 一、核心问题诊断：为什么硬编码阈值必然失效

本次短剧走查暴露的不是"某个阈值数值不对"的问题，而是**阈值体系的设计范式本身有缺陷**。当前总纲有 140+ 个硬编码阈值，但它们本质上是**把"软件开发 + 文字创作"两个领域的经验，压缩成了固定数值**。当任务进入新领域（短剧生成、视频制作、多模态创作）时，这些阈值就像拿直尺量体重——单位都不对。

### 当前 vs 目标

| 当前：硬编码阈值 | 目标：自适应阈值 |
|----------------|----------------|
| 140+ 个固定数值散落各处 | 少量元参数 + 推导公式 |
| 新增子类型需逐个调整阈值 | 新领域只需定义元参数，阈值自动推导 |
| 阈值之间的关系是隐含的、靠经验维持的 | 阈值之间有明确的数学关系，自洽 |
| 新领域 = 重新猜一套数 | 新领域 = 输入特征 → 自动生成阈值 |
| 无法自我校准，依赖人工调参 | 有校准闭环，越用越准 |
| 领域差异靠"权重调整"掩盖，范式错配无解 | 范式差异在元参数层表达，底层逻辑统一 |

### 三大根本矛盾

**矛盾一：阈值的"颗粒度错配"**

总纲在 L0 层面区分了 4 大领域（创作/开发/分析/研究），但阈值只做到了领域级差异，没有做到子类型级差异。创作领域下的小说、短剧、营销文案、技术文档——它们的质量指标、接口范式、递归深度完全不同，但共用一套"创作类阈值"。这就像医生给所有病人开同一种药，只是剂量不同。

**矛盾二：阈值的"维度不可比"**

"字段完整性 ≥ 80%"和"叙事要素传递率 ≥ 90%"——这两个 80% 和 90% 根本不是同一个维度。开发类的阈值是**结构型**的（行数、覆盖率、字段数），创作类的阈值是**质量型**的（连贯性、感染力、忠实度），短剧类的阈值是**感知型**的（画面一致性、表演自然度）。把它们都塞进"≥ X%"的格式里，只是自欺欺人。

**矛盾三：阈值的"静态性 vs 任务动态性"**

固定阈值假设"所有同类任务都一样"，但实际任务差异巨大——3集短剧和30集短剧的递归深度、质量要求、验收标准完全不同。同样是"创作类"，一首短诗和一部长篇小说的阈值怎么可能一样？

## 二、自适应阈值三层架构

核心思路：**用"元参数 + 推导公式 + 校准闭环"替代"硬编码数值"。阈值不应该是答案，阈值应该是"从任务特征推导出答案的过程"。

```
┌─────────────────────────────────────────────┐
│  L2：校准闭环层（Calibration Loop）         │
│  任务执行结果反馈 → 更新元参数分布 → 优化推导公式  │
└──────────────────────┬──────────────────────┘
                       │
┌──────────────────────▼──────────────────────┐
│  L1：推导引擎层（Derivation Engine）       │
│  从任务特征 + 元参数出发，动态计算每个检查点的阈值  │
└──────────────────────┬──────────────────────┘
                       │
┌──────────────────────▼──────────────────────┐
│  L0：元参数层（Meta-Parameters）          │
│  描述任务本质特征的少量参数，而非具体阈值      │
└─────────────────────────────────────────────┘
```

### L0：元参数层 — 用 6 个维度描述所有任务

每个任务（不管什么领域）都可以用 6 个元参数来描述其本质特征。这 6 个参数不是阈值，而是**推导阈值的原料**。

| 元参数 | 说明 | 评分范围 |
|--------|------|---------|
| 规模 Scale | 任务产出的量级（章节数/集数/功能点数/字数） | 1-5 |
| 模态 Modality | 涉及的信息模态数量（纯文字/图文/视频/多模态） | 1-5 |
| 创造性 Creativity | 从0到1的创造程度 vs 按图索骥的执行程度 | 1-5 |
| 可验证性 Verifiability | 产出质量是否容易客观判定（硬指标型 vs 主观感知型） | 1-5 |
| 迭代性 Iteration | 产出需要多少轮试错和优化才能达标 | 1-5 |
| 依赖性 Dependency | 任务各部分之间的耦合程度和依赖复杂度 | 1-5 |

每个子类型（如短剧、小说、Web应用）有一个默认的元参数画像，但具体任务可以根据输入特征动态调整。

### L1：推导引擎层 — 从元参数到具体阈值

推导引擎是一组公式，输入"元参数 + 检查点类型"，输出"该检查点的阈值和验收方式"。核心原则：**不是给每个领域一套阈值，而是给每个阈值一个推导公式**。

#### 示例1：递归深度推导

```
// 旧版：硬编码 + 简单公式
max_depth = ceil(log2(scale)) + 1   // 只考虑规模，不考虑模态

// 新版：元参数推导
base_depth = ceil(log2(scale))
modality_bonus = modality - 1       // 每多一种模态，深度+1
dependency_bonus = floor(dependency / 2)  // 高依赖任务多一层协调
max_depth = base_depth + modality_bonus + dependency_bonus
```

> 注：实际公式需要校准，这里只是展示思路。关键是：深度不再是固定值，而是从多个元参数推导出来的。

#### 示例2：接口契约范式选择

```
// 不是所有接口都用"字段完整性 ≥ 80%"
// 根据 verifiability 和 creativity 选择契约范式

if verifiability ≥ 4 and creativity ≤ 2:
    // 结构型契约（开发类、技术文档）
    contract_type = "field_completeness"
    threshold = 0.75 + 0.05 × verifiability   // 0.80 ~ 0.95

elif verifiability ≤ 2 and creativity ≥ 3:
    // 叙事型契约（创作类、短剧）
    contract_type = "narrative_cohesion"
    // 用人物一致性、时间线连续性、伏笔回收率替代字段完整性
    metrics = ["character_consistency", "timeline_continuity", "setup_payoff_ratio"]
    threshold = 0.80 + 0.05 × (5 - creativity)  // 创造性越高，阈值越灵活

else:
    // 混合型契约（分析类、研究类）
    contract_type = "mixed"
    structural_weight = verifiability / 5
    quality_weight = 1 - structural_weight
```

#### 示例3：质量指标数量推导

```
// 质量指标数量不是固定的，而是随模态和规模增长
base_metrics = 2                      // 任何任务至少2个核心指标
modality_metrics = (modality - 1) × 2  // 每种模态增加2个专属指标
scale_metrics = min(floor(scale / 5), 3)   // 规模越大，指标越多，但有上限

total_metrics = base_metrics + modality_metrics + scale_metrics

// 验证：
// 短篇小说（scale=2, modality=1）: 2 + 0 + 0 = 2 个指标 ✓
// Web应用（scale=5, modality=1）:    2 + 0 + 1 = 3 个指标 ✓
// 短剧（scale=3, modality=4）:      2 + 6 + 0 = 8 个指标 ✓
```

## 三、核心结论的普适性改造

当前总纲的核心结论（"8 Phase 双程递归 + 5D 收敛引擎 + 8 检查点监督"）本身是正确的，但它的**表达形式**不够普适——它用了太多开发类/文字创作类的具体例子和阈值，导致看起来像是"为这两类任务设计的"，而不是"通用框架"。

### 改造原则

> **总纲的结论应该是"过程正确"而非"数值正确"。
> 工业级的框架不承诺"给你正确的答案"，而是承诺"如果你按照这个过程走，你会得到你能得到的最好答案，并且你知道这个答案有多可靠"。

### 具体改造点

| 当前表述 | 问题 | 普适化改造 |
|---------|------|-----------|
| 字段完整性 ≥ 80% | 只适用于结构型任务，创作类/感知类不适用 | 改为**"接口契约达成率"**，契约类型由 verifiability 元参数决定：结构型用字段完整性、叙事型用要素传递率、感知型用一致性评分 |
| inferred 比例 < 30% | 探索性任务本身就需要大量推断，30% 太低 | 改为**"已知信息覆盖度"**，由 creativity 元参数动态调整：创造性越高，允许的推断比例越高（从 20% 到 70%） |
| 递归深度 = log2(N)+1 | 只考虑规模，不考虑模态和依赖关系 | 改为**深度推导公式**：base_depth + modality_bonus + dependency_bonus，由 3 个元参数共同决定 |
| 测试覆盖率 ≥ 80% | 只有开发类有"测试覆盖率"，其他领域没有对应概念 | 改为**"验证覆盖率"**，验证方式由 verifiability 决定：高可验证性用自动化测试、中可验证性用清单检查、低可验证性用 AI 评分 + 抽样 |
| 5 层验证模型 | "单元测试/集成测试/E2E"是开发术语，其他领域不适用 | 改为**"自底向上五级验证"**，每级的验证方式随领域变化：叶节点验证 → 子节点验证 → 模块验证 → 宏观节点验证 → 端到端验证（验证手段自适应） |
| 降标判定：偏差 ≥ 10% | 10% 是按功能点算的，创作类按什么算？ | 改为**"核心承诺偏差率"**，偏差基数由任务的 core_promises（核心承诺项）决定，而非固定百分比 |
| 4 大领域变体 | 领域是粗粒度分类，子类型差异巨大 | 改为**"元参数画像 + 领域模板"**：4 大领域提供默认元参数画像，子类型继承并微调，新子类型无需从零开始 |

### 普适化后的核心结论

任何从"模糊意图"到"具体产出"的任务，都遵循同一个通用过程：

1. **约束收敛**：从任务的规模、模态、创造性、可验证性、迭代性、依赖性六个元维度出发，收敛出唯一的执行路径
2. **去程拆解**（Phase 1-5）：意图 → 路径 → 节点 → 接口 → 递归，逐步将模糊意图拆解为可执行的最小单元
3. **回程合成**（Phase 6-8）：实现 → 集成 → 验证 → 交付，自底向上逐层验证并合成最终产出
4. **全程监督**：8 个检查点确保每一步的质量，但每个检查点的**具体验收方式和阈值由元参数动态推导**，而非固定数值

总纲的价值不在于告诉你"某个阈值是 80%"，而在于告诉你：**对于你这个具体任务，你应该关注哪些维度，这些维度如何相互关联，以及你怎样知道自己做对了。

## 四、L2 校准闭环：越用越准的阈值系统

自适应阈值不能只有"推导"，还必须有"校准"。否则只是把硬编码从数值层搬到了公式层。

### 校准闭环的工作原理

```
任务执行 → 结果评估 → 偏差归因 → 参数更新 → 下次更准
```

### 三个校准层级

| 校准层级 | 校准对象 | 触发条件 | 影响范围 |
|---------|---------|---------|---------|
| L1 任务级 | 单个任务的阈值微调 | 任务执行中发现明显不合理 | 仅当前任务 |
| L2 子类型级 | 子类型的元参数默认画像 | 同子类型 N 个任务（如 N≥10）的统计偏差 | 该子类型所有后续任务 |
| L3 框架级 | 推导公式本身 | 跨多个子类型的系统性偏差 | 全框架 |

### 冷启动策略

新子类型没有历史数据时怎么办？

1. **继承父领域默认值**：短剧继承创作领域的元参数画像
2. **基于特征插值**：根据 6 个元维度的评分，在已知子类型中找最相似的，借用其阈值体系
3. **保守阈值 + 快速校准**：前 3 个任务使用偏保守的阈值（质量要求偏高，数量要求偏低），每个任务后快速校准
4. **人工标记加速**：关键任务由人工标记"阈值是否合理"，加速校准曲线

## 五、实施路线图

从硬编码阈值到自适应阈值是一个渐进的过程，不需要一步到位。建议分四个阶段实施：

### Phase A：阈值清单化（1-2 周）

将当前 140+ 个阈值全部整理到一个统一的阈值注册表中（threshold-registry.json），每个阈值标注：适用领域、适用子类型、默认值、推导依据（如果有）、相关检查点。

**目标**：阈值不再散落各处，有统一的管理入口。

### Phase B：元参数化（2-4 周）

定义 6 个元参数维度，为现有 4 大领域 + 已有的子类型建立元参数画像。将最核心的 5-10 个阈值（递归深度、节点数范围、质量指标数量、接口契约类型、验证层级数）改为从元参数推导。

**目标**：新增子类型不需要逐个调阈值，只需定义元参数画像。

### Phase C：校准闭环（4-8 周）

实现 L1 任务级校准（任务执行后可记录阈值偏差）和 L2 子类型级校准（统计同子类型的偏差趋势）。建立校准日志和可视化仪表盘。

**目标**：阈值体系开始自我进化，越用越准。

### Phase D：全量推导（持续）

逐步将更多阈值从硬编码改为推导公式。最终目标：除了最底层的几个元参数默认值，所有阈值都是推导出来的。L3 框架级校准上线（推导公式的自我优化）。

**目标**：真正的"无阈值"框架——所有阈值都从任务特征动态生成。

## 六、预期收益与风险

### 收益

| 收益 | 说明 |
|------|------|
| 新领域适配速度 10x | 不需要为每个新领域重新猜 140 个阈值，只需定义 6 个元参数 + 验证几个核心推导公式 |
| 阈值自洽性 | 所有阈值从同一套元参数推导，不会出现"A阈值提高了但B阈值没跟上"的不一致问题 |
| 可解释性 | 每个阈值都有推导依据（"因为这个任务是4模态的，所以需要8个质量指标"），不是"经验值" |
| 持续进化 | 校准闭环让阈值体系越用越准，形成数据飞轮 |
| 普适性 | 总纲的核心结论不再绑定具体领域，而是真正的通用方法论 |

### 风险与应对

| 风险 | 应对策略 |
|------|---------|
| 过度工程化 | 渐进式实施，Phase A/B 就足够产生 80% 的价值，后面按需推进 |
| 推导公式不准 | 公式是起点不是终点，校准闭环会逐步修正。初期保留"人工 override"通道 |
| 元参数维度不够 | 6 个维度是初始设计，实践中可能需要增减。元参数本身也应该是可演进的 |
| 性能开销 | 阈值推导是一次性计算（Phase 0 完成），不影响执行阶段性能 |

## 七、总结回答

### 问题一：需要做多大的阈值范围修改？

**不是修改数值，而是修改范式。**

如果逐个修改 140+ 个阈值来适配短剧，那下一个新领域（比如 3D 建模、音乐生成）又要重来一遍。正确的做法是用"元参数 + 推导引擎 + 校准闭环"替代硬编码阈值。

短期来看：最紧急的是修改 **5-7 个核心阈值的定义方式**（接口契约范式、递归深度、质量指标数量、验证层级、推断比例上限、降标判定、节点数范围），它们决定了整个框架的"形状"。其他阈值可以保留硬编码，但统一管理。

### 问题二：核心结论如何修改才有普适性价值？

**把"是什么"改成"怎么推"。**

1. 总纲的核心不是"8 Phase + 8 检查点 + 140 个阈值"，而是**"双程递归 + 约束收敛 + 自适应验证"**这个过程本身
2. 每个检查点的**存在是必要的**（确保每一步质量），但检查点的**具体验收方式是可变的**（随任务特征自适应）
3. 领域变体的正确做法不是"每个领域一套阈值"，而是**"每个领域一个元参数画像，阈值从画像推导"**
4. 普适性的终极检验：给总纲一个全新的任务类型（比如"生成一首交响乐"），它能自动推导出合理的执行路径和质量标准，而不需要人类先写一套领域规则

---

> 核心洞察：阈值是领域知识的压缩表达 —— 与其压缩成数字，不如压缩成推导过程。
