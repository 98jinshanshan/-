### 0.2 实际走查发现

#### L0 领域分类（domain-dict.json 关键词匹配）

**计分过程**：

| 领域 | 命中关键词 | 排除词 | 计分 |
|------|-----------|--------|------|
| creative | 「章」（前十章）→ 1 hit | 无 | **1** |

| analysis | 无 | 无 | 0 |
| research | 无 | 无 | 0 |

**结果**：creative 胜出，score = 1，second_best = 0，满足 score ≥ second_best + 1。

**偏差发现**：
- **D-L0 关键词覆盖不足**：输入中"短剧""生成""集"三个核心词均未命中 creative 关键词列表。仅靠"章"一个字命中，得分极低（仅1分）。如果输入改成"把第七声钟响改成短剧"（不含"章"字），L0 将全部得0分，触发 ambiguous 双路径。

#### L1 子类型约束（subtype-dict.json 创作领域内匹配）

**计分过程**：

| 子类型 | 命中关键词 | 计分 |
|--------|-----------|------|
| fiction（小说/故事） | 「章」（前十章）→ 1 hit | **1** |
| script（剧本/脚本） | 无直接命中（"短剧"不在列表中） | 0 |
| article（文章/博客） | 无 | 0 |
| marketing（营销/文案） | 无 | 0 |

**结果**：fiction 胜出，score = 1，second_best = 0。

**严重偏差发现**：
- **D-L1 子类型错配**：任务本质是"小说改编短剧剧本"，最终产出是剧本/脚本（script），但由于"短剧"不在 script 子类型的关键词列表中，L1 错误地收敛到了 fiction（小说/故事）。这是一个方向性错误——后续所有 Phase 将按"小说创作"路径执行，而非"剧本/短剧创作"路径。

#### L2 平台/技术栈

**状态**：`platform-dict.json` **未创建**（IMPL-1 未解决）。

**走查发现**：
- **D-L2 字典缺失**：没有平台字典，L2 层无法执行确定性匹配。
- 输入中"短剧"隐含目标平台（抖音短剧、快手短剧等），但无法被识别。
- 当前只能标记为 inferred，走"通用默认规则"，但短剧有非常具体的平台规则（竖屏、每集时长1-3分钟、强钩子节奏等），通用规则完全不适用。

#### L3 结构参数提取

**状态**：`structural-patterns.json` **未创建**（IMPL-2 未解决）。

**手动提取结果**：

| 参数 | 值 | 提取方式 | 置信度 |
|------|-----|---------|--------|
| 集数 | 3（前三集） | 数字+范围词提取 | 高 |
| 来源章节数 | 10（前十章） | 数字+范围词提取 | 高 |
| 来源作品 | 第七声钟响 | 专有名词提取 | 高 |
| 每集时长 | 未指定 | — | 无 |
| 输出格式 | 未指定（剧本？分镜？视频？） | — | 无 |
| 视角 | 未指定 | — | 无 |
| 改编忠实度 | 未指定 | — | 无 |

**提取率估算**：3/7 ≈ 43%，**低于 50% 阈值**，应标记 high_inference_risk。

**偏差发现**：
- **D-L3 提取率不足**：7个核心结构参数仅提取3个，提取率43% < 50%阈值。
- 缺少 structural-patterns.json，无法自动化执行 L3 提取。
- "集"作为短剧核心量化单位，不在结构参数模式库中（如果模式库存在的话）。

#### L4 领域规则校验

**状态**：`domain-rules.json` **未创建**（IMPL-3 未解决）。

**手动推导应有规则**：

| 约束类型 | 规则 | 适用 |
|---------|------|------|
| 硬约束 | 每集必须有开场钩子（前3秒抓人） | 短剧通用 |
| 硬约束 | 每集结尾必须有悬念/反转（断章钩子） | 短剧通用 |
| 硬约束 | 单集时长 1-3 分钟（竖屏短剧标准） | 抖音短剧平台 |
| 软约束 | 对白占比 ≥ 60%（短剧节奏快） | 短剧通用 |
| 软约束 | 场景数 ≤ 5 个/集（成本控制） | 短剧通用 |
| 硬约束 | 改编不得改变原作核心人物设定 | 改编类任务 |
| 硬约束 | 10章内容需合理分配到3集，不得遗漏主线 | 改编类任务 |

**偏差发现**：
- **D-L4 规则库完全缺失**：domain-rules.json 未创建，L4 层无法执行任何硬约束校验。
- 短剧作为独立子类型，其专属规则（钩子节奏、时长、竖屏格式等）完全没有定义。
- "小说改编短剧"这类跨子类型任务的改编规则（忠实度、人物一致性等）也没有定义。

#### L5 唯一性锚定

**走查发现**：

由于 L2（平台字典）、L3（结构模式库）、L4（领域规则库）三层均有缺失或提取不足，L5 的质量评分公式（完整性、冲突率、精确度、规则合规度）无法有效计算：

- **完整性**：无法评估（缺少规则基准）
- **冲突率**：数据太少，无法检测冲突
- **精确度**：提取率仅43%，精确度低
- **规则合规度**：无规则可合

**实际状态**：L5 仅能基于 L0-L1 的极低保真结果做"伪收敛"——看起来收敛到了一条路径，但实际上是"信息不足导致的虚假唯一"，而非"穷举筛选后的真正最优"。

### 0.4 Phase 0 优化建议

1. **【P0】补充 script 子类型关键词**：在 subtype-dict.json 的 creative/script 中新增短剧相关关键词：短剧、竖屏剧、微短剧、集、每集、分集、剧集、抖音短剧、快手短剧、短剧剧本、分镜脚本等。这是最紧急的修复——否则 L1 方向性错误会导致后续全错。

2. **【P0】创建 platform-dict.json**：至少覆盖短剧领域的主要平台——抖音短剧（竖屏、1-3分钟/集、强钩子）、快手短剧、腾讯微短剧、通用短剧。每个平台定义格式、时长、节奏等规则。

3. **【P0】创建 structural-patterns.json**：建立结构参数正则模式库，覆盖：
   - 数量类：前N章、前N集、共N章、N集
   - 范围类：N-M字、N-M分钟
   - 枚举类：视角（第一/第三/多视角）、格式（docx/txt/pdf/md）
   - 专有名词：作品名提取模式

4. **【P0】创建 domain-rules.json**：按领域→子类型→平台的层级定义规则。至少包含：
   - creative/script/短剧：硬约束3条（钩子+时长+格式）、软约束3条（对白占比+场景数+节奏）
   - 改编类通用规则：人物一致性、主线完整性、忠实度分级

5. **【P1】扩充 creative 领域关键词**：在 domain-dict.json 的 creative 中增加"生成""改编""改""短剧""剧集"等创作动作和产出类型词，提高 L0 分类鲁棒性。

6. **【P1】增加跨子类型检测逻辑**：当输入同时包含来源子类型（fiction 的"章"）和目标子类型（script 的"剧/集"）时，应识别为"改编类任务"，以目标产出子类型（script）为主路径，以来源子类型（fiction）为输入约束。

### 1.4 Phase 1 优化建议

1. **【P0】修复 L1 子类型错误**：这是根源——必须先在 Phase 0 L1 中正确识别为 script（剧本），否则 Phase 1 的整个自审框架都是错的。

2. **【P0】领域变体强化**：V3.9.1 虽然引入了"领域权重"变体（创作类意图锚定1.5x），但这只是权重调整，不是**问题框架调整**。创作类任务的 44 问应使用创作专属的问题表述和维度定义：
   - "架构建模" → 改为"叙事结构建模"
   - "模块拆解" → 改为"分集/分场拆解"
   - "接口契约" → 改为"叙事衔接契约"（前后场景的人物、时间、地点连续性）
   - "实现路径" → 改为"创作路径"（先写大纲→分集→逐场→对白打磨）

3. **【P1】注入领域规则到自审**：Phase 0 L4 提取的硬约束必须注入到 Phase 1 的 44 问自审中，作为"已知条件"而非"推断项"。例如短剧的"每集必须有钩子"应直接作为 confirmed 项，不需要 Agent 再去"推断"。

4. **【P1】推断比例的分层计算**：当前 inferred 比例是44问全局平均，但核心维度（维度1-2）的推断比例才是关键。建议将硬约束修改为："维度1和维度2的 inferred 比例均不得超过 50%"，而不是简单的全局 < 30%。

5. **【P2】一致性校验的基准化**：CC-1~CC-5 的校验不应在两个推断值之间进行，而应以 Phase 0 提取的确定值为基准。例如 CC-4 数量一致性应该以 L3 提取的"3集"为基准，而不是两个互相推断的值互相校验。

### 2.1 预期结果

### 2.2 实际走查发现

#### 合理路径推导（测试工程师视角）

对于"用小说前十章生成短剧前三集"，一条合理的路径应该是：

```
起点：《第七声钟响》前十章原文
  ↓
节点1：原文解析与要素提取（情节线、人物、场景、关键对白）
  ↓
节点2：分集规划（10章→3集的结构映射，每集核心冲突+钩子）
  ↓
节点3：剧本改编（小说叙事→剧本格式，场景+对白+动作描写）
  ↓
节点4：短剧节奏优化（钩子强化、场景压缩、对白精炼）
  ↓
节点5：格式输出（最终剧本文件，含分场表+人物表）
  ↓
终点：短剧前三集完整剧本
```

**节点数**：5个（在3-10范围内）✅

#### 走查发现的问题

**问题1：起点定义模糊**
- 输入是"用第七声钟响前十章"，但"前十章"是以什么形式存在？
  - 如果是已有 .docx 文件（当前工作区有 `/workspace/第七声钟响_前十章.docx`）→ 起点是文件解析
  - 如果只有大纲 → 起点是大纲提取
  - 如果什么都没有 → 起点还需要包括"先写前十章"
- 输入本身未明确，但工作区中已存在前十章文档 → 应视为已知输入

**问题2：节点粒度不均匀**
- 节点1（原文解析）：相对简单的提取工作
- 节点3（剧本改编）：核心创作工作，工作量远大于其他节点
- 粒度不均匀导致后续 Phase 3/4 的接口和拆解难度不一

**问题3：路径的单向性 vs 迭代需求**
- 短剧改编通常需要迭代：剧本写出来后可能发现分集不合理，需要回去调整分集规划
- 但总纲路径是单向线性的（无循环约束），不支持迭代回退
- 这会导致"回程发现问题无法回退"——与双程边界违规判定规则冲突（回程不能改去程设计）

**问题4：缺少"改编忠实度"控制点**
- 从小说到短剧的改编，最核心的约束是忠实度（哪些保留、哪些可以改、哪些必须改）
- 但路径中没有一个独立的节点来定义和校验改编忠实度
- 这可能导致最终剧本与原作偏差过大而无人发现

**问题5：终点定义不够精确**
- "短剧前三集"到底交付什么？剧本？分镜脚本？拍摄清单？
- 不同的终点定义会导致完全不同的路径结构
- 输入未明确 → 推断 → 路径可能建错方向

### 2.3 Phase 2 偏差汇总（D-L0 到 D-L4 维度）

| 维度 | 满足度 | 偏差描述 | 严重度 |
|------|--------|---------|--------|
| D-L0 领域 | ⚠️ 路径框架偏差 | 创作类路径定义是按"从零创作"设计的，不适用于"改编类"任务。改编类有原作输入约束，路径结构不同。 | 中 |
| D-L1 子类型 | ❌ 子类型路径错配 | 继承 Phase 0 L1 错误，如果按 fiction 走，路径会是"大纲→章节→正文"的小说创作路径，而非"原文解析→分集→剧本"的改编路径。 | 致命 |
| D-L2 平台 | ❌ 平台节点缺失 | 短剧路径中缺少"平台适配"节点（竖屏格式、时长裁剪、钩子密度调整等）。如果最终面向抖音短剧，这些是必须的。 | 严重 |
| D-L3 结构参数 | ⚠️ 部分满足 | 节点数5个在3-10范围内。但缺少按"集数=3"动态调整路径结构的逻辑——3集和30集的路径应该不同。 | 中 |
| D-L4 规则约束 | ❌ 改编规则缺失 | 路径中缺少"改编忠实度校验"节点，也没有硬约束来保证原作核心元素不丢失。无循环规则与改编的迭代需求冲突。 | 严重 |

### 2.4 Phase 2 优化建议

1. **【P0】修复 L1 子类型传导错误**：先解决 Phase 0 的 L1 问题，路径才能按剧本/短剧类型生成。

2. **【P1】增加"改编类"路径模板**：创作类路径应区分三种模式：
   - 从零创作路径：灵感→大纲→正文→润色→交付
   - 改编创作路径：原文解析→改编方案→创作→适配→交付
   - 续写/扩展路径：已有内容→分析→扩展→整合→交付

3. **【P1】节点数与 L3 参数联动**：V3.9.1 已定义"节点数范围由 L3 结构参数动态调整"，但需要具体的联动公式。建议：
   - 集数 ≤ 5 → 节点数 = 4-6（简洁路径）
   - 集数 6-20 → 节点数 = 6-8（标准路径）
   - 集数 > 20 → 节点数 = 8-10（复杂路径，增加分集统筹等节点）

4. **【P2】引入"迭代节点"机制**：对于创作类任务，允许在路径中设置一个"迭代检查点"节点，该节点产出不满足要求时可以回退到前一个节点。这与"无循环"规则不矛盾——因为是受控的回退，而非无限循环。

5. **【P2】终点对齐的量化标准**：路径终点与 one_liner 的对齐不应只是语义判断，应有量化指标。例如：
   - 产出物类型匹配（剧本 vs 剧本）→ 是/否
   - 数量匹配（3集 vs 3集）→ 是/否
   - 来源匹配（前十章 vs 前十章）→ 是/否
   - 三项全部匹配才算终点对齐通过

## Phase 3：节点识别走查

### 3.1 预期结果

| 检查项 | 预期结果 | 标准 |
|--------|---------|------|
| 宏观节点数 | 2~4 个（创作类） | 领域变体：创作类 2-4 |
| 每宏观节点下模块数 | 2~6 个（创作类） | 领域变体：创作类 2-6 |
| 重叠检测 | 无 redundancy 标记 | 两节点映射同一 acceptance_criteria 维度且无明确分工差异 → 冗余 |
| 覆盖度 | 所有 acceptance_criteria 维度有对应节点 | 不遗漏验收维度 |
| 无多余节点 | 每个节点都映射到根目标的必要组成部分 | 不过度拆解 |

### 3.2 实际走查发现

#### 合理节点结构推导

基于 Phase 2 的5节点路径，按创作类规则进行层级划分：

**宏观节点1：素材解析与改编规划**（对应路径节点1-2）
- 模块1.1：前十章原文要素提取（情节线、人物弧光、核心场景）
- 模块1.2：改编策略制定（忠实度定位、增删改原则）
- 模块1.3：三集分集规划（每集核心冲突、钩子位置、人物出场）

宏观节点1下模块数：3个（在2-6范围内）✅

**宏观节点2：剧本创作与节奏优化**（对应路径节点3-4）
- 模块2.1：第一集剧本创作（含场景、对白、动作描写）
- 模块2.2：第二集剧本创作
- 模块2.3：第三集剧本创作
- 模块2.4：短剧节奏统一优化（钩子强化、场景压缩、时长校准）

宏观节点2下模块数：4个（在2-6范围内）✅

**宏观节点3：交付与质检**（对应路径节点5）
- 模块3.1：剧本格式化输出（标准短剧剧本格式）
- 模块3.2：改编忠实度校验（人物、主线、核心设定一致性检查）
- 模块3.3：质量自检（钩子密度、对白占比、节奏流畅度）

宏观节点3下模块数：3个（在2-6范围内）✅

**总计**：宏观节点3个（在2-4范围内）✅，模块总数10个

#### 走查发现的问题

**问题1：宏观节点划分的依据不明确**
- 当前划分是"前→中→后"的时间顺序，但总纲没有明确规定宏观节点的划分原则
- 不同 Agent 可能划分为不同数量的宏观节点（如2个：前期+后期；或4个：解析→规划→创作→交付）
- 这破坏了收敛唯一性——同一个输入可能产出不同的节点结构

**问题2：重叠检测的盲区——纵向重叠**
- overlap_detection 定义是"两个节点映射到同一 acceptance_criteria 维度"，这是横向重叠
- 但短剧改编中存在纵向重叠：改编忠实度校验在模块1.2（制定策略时）和模块3.2（交付前校验）都会涉及
- 这算不算冗余？按当前定义不算（因为不在同一层级），但实际上有重复工作

**问题3：模块粒度差异大**
- 模块2.1/2.2/2.3（每集剧本创作）：每个都是完整的一集创作，工作量大
- 模块3.1（格式化输出）：相对机械的工作，工作量小
- 粒度差异导致后续 Phase 5 递归拆解的深度不一致

**问题4：缺少 acceptance_criteria 到节点的反向映射校验**
- V3.9.1 规定了"acceptance_criteria 每个维度有对应节点"，但短剧任务的 acceptance_criteria 有哪些维度？
  - 数量维度：3集 → 映射到模块2.1/2.2/2.3 ✅
  - 质量维度：钩子密度 → 映射到模块2.4 ✅
  - 忠实度维度：原作一致性 → 映射到模块3.2 ✅
  - 格式维度：剧本格式 → 映射到模块3.1 ✅
- 但如果 Phase 1 的 acceptance_criteria 定义不完整（因为 inferred 比例高），这个校验就没有意义

**问题5："改编策略"节点的定位模糊**
- 改编策略（忠实度、增删改原则）是规划类工作，应该放在哪个层级？
- 放在宏观节点1（素材解析与改编规划）下合理，但它不是一个"可独立交付"的产出物
- 这与 Phase 2 的粒度标准（一个节点=一个独立可交付的产出物）冲突

### 3.3 Phase 3 偏差汇总（D-L0 到 D-L4 维度）

| 维度 | 满足度 | 偏差描述 | 严重度 |
|------|--------|---------|--------|
| D-L0 领域 | ⚠️ 数量合规，质量存疑 | 宏观节点数(3)和模块数(3-4)均在创作类领域变体范围内，但节点划分原则不明确，唯一性无法保证 | 中 |
| D-L1 子类型 | ❌ 短剧专属节点缺失 | 缺少短剧特有的节点类型：钩子设计模块、竖屏场景适配模块、时长控制模块。这些是短剧的核心质量控制点，不是通用剧本创作的节点。 | 严重 |
| D-L2 平台 | ❌ 平台约束未传导到节点 | 节点设计中没有体现平台约束（如抖音短剧的竖屏分镜、1-3分钟时长、强节奏）。平台规则应该在节点层面有对应的执行模块。 | 严重 |
| D-L3 结构参数 | ⚠️ 数量在范围内 | 宏观节点3个（2-4）、模块3-4个（2-6）均在范围内。但"集数=3"没有体现在模块数量的动态调整上（3集和30集的剧本创作模块数量应该不同）。 | 低 |
| D-L4 规则约束 | ❌ 重叠检测规则不完善 | 只有横向重叠检测，缺少纵向重叠检测。改编忠实度等跨阶段工作的重复出现无法被识别。节点划分原则缺失，导致收敛唯一性不足。 | 中 |

### 3.4 Phase 3 优化建议

1. **【P0】补充短剧专属节点模板**：在创作类/剧本子类型下，定义短剧专属的节点模板，包含钩子设计、竖屏适配、时长控制等短剧核心模块。

2. **【P1】明确宏观节点划分原则**：制定客观的划分标准，例如：
   - 按创作阶段划分：前期（解析+规划）、中期（核心创作）、后期（优化+交付）
   - 每个宏观节点必须有明确的"阶段交付物"
   - 宏观节点数量 = 阶段数（创作类通常为3阶段：前期+中期+后期）
   - 这样同一输入总能得到相同数量的宏观节点，保证唯一性

3. **【P1】增加纵向重叠检测**：overlap_detection 不仅检查同级节点，也应检查跨层级的功能重叠。如果多个层级都有"改编忠实度校验"，应明确区分各自的校验范围和深度，避免重复劳动。

4. **【P2】模块数量与集数联动**：剧本创作模块的数量应直接等于集数（3集=3个创作模块）。这是客观的、可量化的，不需要 Agent 主观判断。

5. **【P2】节点粒度均衡化约束**：定义"工作量均衡"规则——同一层级的节点工作量差异不应超过 3 倍。如果某个节点明显大于其他节点，应该拆分；如果明显小于其他节点，应该合并。

| 情节线 / 故事线 / 主线 | 解析→规划 | ⚠️ 语义相近但范围不同（情节线 > 故事线 ≈ 主线） |
| 场景描述 / 场景清单 / 分场表 | 全链路 | ❌ "场景描述"是文本，"场景清单"是列表，"分场表"是结构化表格——类型不同 |
| 钩子 / 悬念 / 反转 / 爆点 | 规划→优化 | ⚠️ 都是短剧节奏术语，但各有细微差别 |

**偏差发现**：创作类任务的字段命名比开发类更"软"，同义词更多，语义边界更模糊。现有的同义字段映射规则（title/name/heading、id/identifier/uuid 等）是开发类的，不适用于创作类。

#### 类型转换问题

| 转换对 | 类型 | 是否允许 | 说明 |
|--------|------|---------|------|
| 章节文本（string）→ 情节线（array of strings） | string→array | ⚠️ 未定义 | 当前规则只定义了 string↔number、object→string、array→object，没有 string→array 的规则 |
| 人物列表（array of strings）→ 人物设定（array of objects） | string[]→object[] | ❌ 信息增益 | 从纯名字到带属性的对象，是信息增加而非转换，但接口契约没有定义"信息增益节点"的概念 |
| 集数（number）→ 每集时长（number） | number→number | ✅ 允许 | 但这是"衍生计算"而非"类型转换"，规则中没有区分 |

**偏差发现**：创作类任务的接口类型转换模式与开发类完全不同：
- 开发类：数据在模块间传递，类型基本不变
- 创作类：信息在模块间增值（原始文本 → 结构化信息 → 创意加工 → 成品），每个阶段都在增加新信息
- 当前的类型转换规则是为开发类设计的，完全不适用于创作类

#### 数值约束一致性

| 数值参数 | 上游（spec） | 中游（nodes） | 下游（interfaces） | 一致性 |
|---------|-------------|-------------|-------------------|--------|
| 集数 | 3集 | 3个剧本模块 | 3集剧本输出 | ✅ 一致 |
| 来源章节 | 10章 | 解析模块输入10章 | 未传递到下游 | ⚠️ 半链路 |
| 每集时长 | 未指定 | 未指定 | 未指定 | ❌ 缺失 |
| 总时长 | 未指定 | 未指定 | 未指定 | ❌ 缺失 |

**偏差发现**：Phase 0 L3 提取的结构参数（10章、3集）只在路径起点附近出现，没有贯穿全链路传递到终点。下游模块不知道"基于前十章"这个约束，可能会凭空添加原作没有的内容。

### 4.3 Phase 4 偏差汇总（D-L0 到 D-L4 维度）

| 维度 | 满足度 | 偏差描述 | 严重度 |
|------|--------|---------|--------|
| D-L0 领域 | ❌ 契约类型错配 | V3.9.1 定义创作类接口为"叙事结构契约"（视角转换点、时间线节点等），但实际接口设计中仍然是字段级的 Schema 匹配。创作类的接口契约应该是"叙事连续性契约"而非"字段完整性契约"。 | 严重 |
| D-L1 子类型 | ❌ 短剧接口无定义 | 短剧作为 script 子类型的一个分支，其接口契约完全没有定义。短剧特有的数据结构（分场表、节拍表、钩子时间点）没有对应的 Schema。 | 严重 |
| D-L2 平台 | ❌ 平台格式未入契约 | 竖屏格式、时长限制、字幕样式等平台约束没有体现在接口契约中。下游模块不知道输出需要符合哪个平台的格式要求。 | 严重 |
| D-L3 结构参数 | ❌ 字段完整性严重不足 | 5个接口中4个低于80%阈值，平均仅54%。根本原因是创作类任务是信息增值过程，每个模块的产出多于消费——这与开发类的"数据流转"模式完全不同。80%的阈值是为开发类设计的。 | 严重 |
| D-L4 规则约束 | ❌ 规则不适用 | 同义字段映射是开发类的（title/name/id），类型转换规则也是开发类的（string↔number、object→string等）。创作类特有的同义词和信息增益模式没有规则。 | 严重 |

### 4.4 Phase 4 优化建议

1. **【P0】重新定义创作类接口契约范式**：创作类任务的接口契约不应该用"字段完整性百分比"来衡量，而应该用"叙事要素传递完整性"：
   - 人物一致性：上游出现的人物在下游是否保持设定一致
   - 时间线连续性：前后场景的时间逻辑是否自洽
   - 伏笔回收：上游埋设的伏笔在下游是否有呼应
   - 核心要素传递率：原作的核心情节/人物/设定在改编中保留了多少
   - 建议将创作类的 80% 阈值替换为"核心叙事要素传递率 ≥ 90%"

2. **【P0】区分"信息流转接口"和"信息增益接口"**：
   - 信息流转接口（开发类为主）：上游输出 = 下游输入，字段完整性应 ≥ 80%
   - 信息增益接口（创作类为主）：上游输出是下游输入的子集，下游通过创作增加新信息
   - 信息增益接口不适用"字段完整性 ≥ 80%"规则，而适用"增益有效性"规则（增益的信息是否服务于根目标）

3. **【P1】建立创作类同义字段词典**：
   - 人物/角色/人物设定/角色设定 → 同一类
   - 情节/故事线/主线/剧情线 → 同一类
   - 场景/分场/场次 → 同一类
   - 钩子/悬念/爆点/反转 → 同一类（但有差异，需标注）

4. **【P1】定义短剧专属接口 Schema**：
   - 分场表 Schema：场次号、场景名、时长、人物、地点、时间、内外景
   - 节拍表 Schema：节拍点、钩子位置、情绪曲线
   - 剧本格式 Schema：场景标题、动作描写、对白、旁白、镜头说明

5. **【P2】全链路数值参数穿透**：Phase 0 L3 提取的所有结构参数（集数、章节数、时长等）必须在全链路接口中传递，每个节点都能访问到这些约束。建议在每个接口的 Schema 中增加一个 `structural_constraints` 字段，透传 L3 参数。

### 根因分析

**根因1（最深层）：Phase 0 L1 子类型分类错误**
- "短剧"不在 script 子类型关键词中 → L1 错误收敛到 fiction
- 后续所有 Phase 都按 fiction（小说）路径执行，方向全错
- 这是多米诺骨牌的第一张

**根因2：字典和规则库的工程化缺失**
- platform-dict.json、structural-patterns.json、domain-rules.json 三个核心文件均未创建
- L2/L3/L4 三层无法有效执行，导致收敛引擎只有 L0/L1 两层在工作
- L5 的"唯一收敛"是信息不足导致的虚假唯一

**根因3：领域变体深度不足**
- V3.9.1 的领域变体主要是"权重调整"和"数量范围调整"
- 但真正需要的是**范式级差异**：
  - 开发类：数据流转、字段匹配、接口契约 → 用字段完整性衡量
  - 创作类：信息增值、叙事连续、创意质量 → 用叙事要素传递率衡量
  - 两类任务的接口契约范式完全不同，不能用同一套规则套权重

**根因4：改编类任务是盲区**
- 总纲的设计基准是"从零开始的创作/开发"
- 但"用A生成B"这类改编/转换/迁移任务有完全不同的路径结构和约束体系
- 输入侧的约束（原作的内容、格式、忠实度要求）和输出侧的要求（目标格式、平台规则）同等重要
- 当前总纲只关注输出侧，输入侧的约束没有贯穿全链路

### 最紧急的5项修复

1. **【P0】subtype-dict.json 补充短剧关键词**（修复 L1 方向性错误）
2. **【P0】创建 platform-dict.json**（至少覆盖短剧平台）
3. **【P0】创建 domain-rules.json**（短剧硬约束+改编规则）
4. **【P1】创作类接口契约范式替换**（字段完整性 → 叙事要素传递率）
5. **【P1】增加"改编类"任务路径模板**（输入约束贯穿全链路）
