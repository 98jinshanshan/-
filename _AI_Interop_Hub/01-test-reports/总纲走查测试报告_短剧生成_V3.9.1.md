## 一、Phase 5：递归拆解

### 5.1 原子性标准 ATOM-1/2/3

| 项目 | 内容 |
|------|------|
| **预期标准** | 叶节点必须同时满足 ATOM-1（单一交付物）、ATOM-2（无跨节点前置依赖）、ATOM-3（可量化验收）。三条全部满足 → 叶节点，去程终点。 |
| **实际走查发现** | **短剧场景下"叶节点"的定义存在歧义。** 总纲领域变体中创作类的原子性定义为"一章/一个场景"，但短剧生成涉及多层粒度：<br>1. **一集** — 满足 ATOM-1（一个视频文件），但不满足 ATOM-2（依赖前后集的剧情连贯性，有跨节点前置依赖）<br>2. **一个场景** — 满足 ATOM-1（一段场景视频）和 ATOM-2（可独立生成），但 ATOM-3 难以量化（"场景质量好"无法用数字表达）<br>3. **一个镜头** — 满足 ATOM-1（一帧/几秒视频）和 ATOM-2（可独立生成），ATOM-3 可量化（时长、分辨率），但粒度过细导致递归爆炸<br>4. **一个分镜脚本条目** — 纯文本层面的原子单元，但不是最终交付物形态 |
| **偏差项** | 创作类原子性定义"一章/一个场景"是文字创作语境，直接套用到视频/短剧生成存在粒度错位。短剧的最终交付物是视频，而视频生成的最小可独立验收单元是什么，总纲未定义。 |
| **严重程度** | **P1** — 不阻塞但影响拆解质量和递归深度计算 |
| **优化建议** | 1. 在创作类领域变体下新增"短剧/视频"子类型的原子性定义：<br>   - 叶节点 = 一个场景（scene）的视频片段<br>   - ATOM-1：单一交付物 = 一个 .mp4 场景视频文件<br>   - ATOM-2：无跨节点前置依赖 = 该场景的分镜和素材已就绪，可独立调用生成工具产出<br>   - ATOM-3：可量化验收 = 时长（秒）、分辨率、帧率、画面与分镜一致性评分<br>2. 区分"内容设计层叶节点"（分镜脚本条目）和"实现层叶节点"（视频片段），二者对应不同 Phase 的原子性判断 |

### 5.2 递归深度计算

| 项目 | 内容 |
|------|------|
| **预期标准** | `max_depth = ceil(log2(L3_extracted_chapter_count)) + 1`。L3 未提取 → 使用领域变体默认值（创作类=2）。 |
| **实际走查发现** | **深度计算公式的 L3 参数映射存在问题。**<br>测试输入经 Phase 0 L3 提取：`集数 = 3`（从"前三集"提取）。<br>按公式：`max_depth = ceil(log2(3)) + 1 = 2 + 1 = 3 层`。<br><br>但短剧生成的实际合理拆解树需要更多层级：<br>```<br>项目级（Level 0）：整部短剧（3集）<br>  ↓<br>宏观节点（Level 1）：每一集（共3集）<br>  ↓<br>模块（Level 2）：每集内的幕/段落（如开场、发展、高潮、结尾，约3-4个）<br>  ↓<br>子节点（Level 3）：每个场景（scene，每集约5-8个场景）<br>  ↓<br>叶节点（Level 4）：每个分镜/镜头组（每个场景约3-5个）<br>```<br>实际需要 4-5 层，远超公式算出的 3 层。<br><br>参考基线（小说任务）：10章 → 递归深度=2（项目级 → 章节即叶节点），符合创作类默认值。 |
| **偏差项** | 1. `L3_extracted_chapter_count` 的变量名是小说专属的，短剧场景应映射为 `episode_count` 或 `scene_count`。<br>2. `log2(N) + 1` 公式对于短剧场景严重低估深度，因为短剧的每一集内部还需要拆成场景→分镜→镜头，是乘法关系而非对数关系。<br>3. 创作类领域变体默认深度=2 只适用于纯文字创作，不适用于多模态（文字→分镜→视频）的短剧生成。 |
| **严重程度** | **P0** — 递归深度不足会导致拆解不彻底，叶节点不是真正的原子单元，回程实现时仍需二次拆解，违反"去程不写代码只做设计"原则 |
| **优化建议** | 1. 重命名 L3 提取变量为领域无关的 `structural_unit_count`，各领域映射到自己的单位（章/集/模块/指标）。<br>2. 深度公式按领域调整：<br>   - 文字创作：`max_depth = ceil(log2(chapter_count)) + 1`（保持不变）<br>   - 短剧/视频：`max_depth = ceil(log2(episode_count)) + 3`（+集内场景层 + 分镜层 + 镜头层）<br>   - 或更精确：根据 L3 提取的结构复杂度动态计算，而非简单对数公式<br>3. 引入"跨模态深度加成"概念：每跨越一种模态（文字→图像→视频），深度 +1 |

### 5.3 覆盖度校验

| 项目 | 内容 |
|------|------|
| **预期标准** | 子节点覆盖的维度并集 / 父节点 acceptance_criteria 维度 ≥ 100%。 |
| **实际走查发现** | **短剧场景下覆盖度校验有两个难点：**<br><br>1. **维度拆解的歧义性**：父节点 acceptance_criteria 可能包含"剧情连贯""画面质量""音效适配"等多维度。子节点（场景）覆盖的是剧情维度的子集，但画面质量和音效是全局属性，无法按场景切分覆盖。<br><br>2. **跨维度覆盖难以量化**：小说任务的覆盖度是"章节数/总章数"的线性关系，天然可计算。但短剧的 acceptance_criteria 包含叙事维度、视觉维度、听觉维度，三者正交，子节点在每个维度上的"覆盖比例"如何定义不明确。<br><br>参考基线（小说任务）：nodes.json 有 `necessity_audit.all_criteria_covered = true`，但实际只覆盖了 structure 一个维度，其他维度（word_count、perspective、hooks、logic、format）未做节点级映射。 |
| **偏差项** | 1. 覆盖度校验只验证了"结构维度"，未验证质量维度（画面、音效、节奏等）的覆盖。<br>2. 参考实现（小说任务）本身就存在覆盖度校验不彻底的问题 — `all_criteria_covered = true` 但只映射了 structure 维度。<br>3. 短剧场景的多模态属性使覆盖度问题更加突出。 |
| **严重程度** | **P1** — 覆盖度校验不完整会导致某些 acceptance_criteria 维度在拆解阶段就被遗漏，到 Phase 7 才发现缺失 |
| **优化建议** | 1. 覆盖度校验改为**多维度矩阵**：每个 acceptance_criteria 维度单独计算覆盖率，全部 ≥ 100% 才算通过。<br>2. 对于全局属性（如画面质量、音效），不做节点级拆分，而是标记为 `global_criteria`，由项目级统一验证。<br>3. cp_4 检查点增加"全局属性维度清单"校验项，确保未被遗漏。 |

### 5.4 合理的拆解树结构

| 项目 | 内容 |
|------|------|
| **预期标准** | 项目级 → 宏观节点 → 模块 → 子节点 → 叶节点（5层架构），每层节点数由领域变体定义。 |
| **实际走查发现** | **短剧场景的合理拆解树如下（以3集为例）：**<br>```<br>项目级（Level 0）：短剧《第七声钟响》前三集<br>  ├── 宏观节点1（Level 1）：第一集（邀请函篇·上）<br>  │   ├── 模块1.1（Level 2）：开场段（沈默收信+调查）<br>  │   │   ├── 子节点1.1.1（Level 3）：场景1-书房收信<br>  │   │   │   └── 叶节点1.1.1.1（Level 4）：分镜组A（3-5个镜头）<br>  │   │   │   └── 叶节点1.1.1.2（Level 4）：分镜组B（3-5个镜头）<br>  │   │   ├── 子节点1.1.2（Level 3）：场景2-记忆闪回<br>  │   │   └── 子节点1.1.3（Level 3）：场景3-决定赴约<br>  │   ├── 模块1.2（Level 2）：发展段（多人视角收信）<br>  │   ├── 模块1.3（Level 2）：结尾段（码头集结+钩子）<br>  ├── 宏观节点2（Level 1）：第二集（邀请函篇·中）<br>  └── 宏观节点3（Level 1）：第三集（邀请函篇·下/钟鸣馆）<br>```<br><br>按此结构：<br>- 宏观节点：3个（符合创作类2-4范围）<br>- 模块：每集3-4个（共9-12个，超出创作类2-6范围）<br>- 子节点：每个模块3-5个场景（共27-60个，超出创作类2-4范围）<br>- 叶节点：每个场景3-5个分镜组（数量巨大） |
| **偏差项** | 1. 创作类领域变体的节点数范围（宏观2-4、模块2-6、子节点2-4）是基于文字创作设定的，短剧场景节点数远超范围。<br>2. 5层架构（项目→宏观→模块→子→叶）对于短剧可能不够，可能需要6层（项目→季/篇→集→幕→场景→分镜）。<br>3. 叶节点如果是分镜/镜头，数量会非常庞大（3集 × 6场景 × 5分镜 = 90个叶节点），递归执行效率堪忧。 |
| **严重程度** | **P1** — 节点数范围不匹配会导致 cp_2 检查点（节点数在范围内）误判为失败 |
| **优化建议** | 1. 在创作类下新增"短剧/视频"子类型，独立设定节点数范围：<br>   - 宏观节点：2-10（集数）<br>   - 模块：3-8（每集内的幕/段落）<br>   - 子节点：3-10（场景数）<br>2. 允许"叶节点为场景级"而非必须到镜头级 — 视频生成工具通常接受场景级prompt直接产出视频片段，不需要拆到镜头。<br>3. 引入"批量叶节点"概念：相似属性的叶节点（如同一人物的多个镜头）可以批量处理，减少递归开销。 |

## 二、Phase 6：实现集成

### 6.1 质量指标

| 项目 | 内容 |
|------|------|
| **预期标准** | 创作类质量指标：内容字数偏差 ≤ 30%，章数 ≥ 承诺 80%。 |
| **实际走查发现** | **短剧生成的质量指标与纯文字创作完全不同，现有指标无法复用。**<br><br>短剧生成需要的可量化质量指标：<br><br>**叙事质量（与小说有重叠）：**<br>- 集数完成率：实际产出集数 / 承诺集数（对标章数指标）<br>- 时长偏差：每集实际时长 / 目标时长（对标字数偏差）<br>- 剧情完整度：关键情节点覆盖率（对标内容完整性）<br><br>**视觉质量（新增维度）：**<br>- 画面一致性：人物形象在不同场景/镜头中的一致性评分<br>- 场景还原度：视频画面与分镜脚本的匹配度评分<br>- 画质指标：分辨率达标率、帧率稳定性、无闪烁/崩坏帧比例<br><br>**听觉质量（新增维度）：**<br>- 配音质量：语音清晰度、情感匹配度<br>- BGM适配度：音乐与场景情绪的匹配度<br>- 音效完整性：关键音效是否齐全<br><br>**技术质量（新增维度）：**<br>- 格式合规性：视频编码、封装格式、音轨数量<br>- 文件完整性：文件可播放、无损坏<br><br>参考基线（小说任务）：blueprint_comparison.json 对比的是章节数和字数，共2个核心指标。短剧场景至少需要 8-10 个指标。 |
| **偏差项** | 1. 创作类质量指标只有2个（字数偏差、章数偏差），短剧场景缺少视觉、听觉、技术三个维度的质量指标定义。<br>2. 视觉/听觉质量多为主观评分，难以像测试覆盖率那样精确量化，总纲未提供"主观指标如何量化验收"的方法论。<br>3. cp_5 检查点的"所有 applicable quality_metrics 达标"规则，在短剧场景下 applicable 的范围不明确。 |
| **严重程度** | **P0** — 质量指标缺失导致 Phase 6 无法判断实现是否合格，cp_5 检查点形同虚设 |
| **优化建议** | 1. 为"短剧/视频"子类型定义三级质量指标体系：<br>   - **硬指标（可自动检测）**：分辨率、帧率、时长、文件格式、文件大小、可播放性 → 必须 100% 达标<br>   - **软指标（AI评分）**：画面一致性、场景还原度、配音质量、BGM适配度 → 设定阈值（如一致性评分≥0.7）<br>   - **叙事指标（人工/AI校验）**：剧情完整度、人物连贯性、节奏合理性 → 关键情节点覆盖率≥90%<br>2. 建立"质量指标字典"，每个领域子类型有独立的 quality_metrics 清单，Phase 0 L4 规则校验时加载。<br>3. 引入"生成一致性校验"作为短剧专属质量指标 — 同一人物在不同镜头中的外观相似度（可用人脸识别相似度算法量化）。 |

### 6.2 偏差分类

| 项目 | 内容 |
|------|------|
| **预期标准** | 正向纠偏（Agent 发现蓝图遗漏，主动补充）→ 标记 [ADDED]，不 FAIL。偏离（未标记且与蓝图不一致）→ cp_5 FAIL。 |
| **实际走查发现** | **短剧场景下偏差分类的边界更加模糊。**<br><br>**明确的正向纠偏：**<br>- 分镜脚本中未指定具体镜头运动，生成时自动添加合理的推拉摇移 → [ADDED]<br>- 场景之间添加合理的转场效果 → [ADDED]<br>- 补充蓝图中未明确的环境音效（如风声、脚步声）→ [ADDED]<br><br>**明确的偏离：**<br>- 私自增减集数（3集变2集）且未标记 → FAIL<br>- 剧情走向与分镜脚本不一致 → FAIL<br>- 人物形象与角色设定表不符且未标记 → FAIL<br><br>**灰色地带（难以判定）：**<br>- 分镜脚本写"沈默走进房间"，生成的视频中沈默走的路线与想象不同 — 算合理演绎还是偏离？<br>- 分镜脚本写"紧张的BGM"，实际用了悬疑风格音乐 — 算风格偏差还是正向补充？<br>- 某个场景生成效果不好，Agent 自行调整了 prompt 重生成 — 算自行修改设计还是实现细节优化？ |
| **偏差项** | 1. "蓝图遗漏的实现细节"与"修改设计"的边界在视频生成场景下更加模糊。文字创作中"补充描写细节"是明确的正向纠偏，但视频生成中很多视觉选择既是实现细节也是设计决策。<br>2. 双程边界违规判定中，回程修改 spec.json 的 acceptance_criteria 是违规，但修改分镜脚本的视觉描述算什么层级的修改，总纲未定义。<br>3. 短剧生成的迭代性强（同一镜头可能多次重生成以优化效果），每次迭代都要走"上报→评估→回退"流程会严重降低效率。 |
| **严重程度** | **P2** — 不阻塞执行，但会导致 cp_5 检查点的判定主观性强，不同 Agent 可能有不同判定结果 |
| **优化建议** | 1. 为短剧场景细化偏差分级：<br>   - **L1 细节偏差**（镜头运动、光影角度、具体构图）：允许回程自主调整，标记 [ADDED]<br>   - **L2 风格偏差**（服装颜色、场景布局、BGM选择）：需标记 [VARIANT]，不 FAIL 但需记录<br>   - **L3 结构偏差**（增减场景、修改剧情、调换顺序）：必须上报 supervisor，可能触发回退<br>2. 在 interfaces.json 中定义"分镜脚本的可变维度"和"不可变维度"，明确哪些是回程可自主调整的范围。<br>3. 引入"实现迭代次数限制"：同一叶节点最多允许 N 次重生成，超过则标记 quality_issue 上报。 |

## 三、Phase 7：通路验证

### 7.1 验证阈值与五层验证

| 项目 | 内容 |
|------|------|
| **预期标准** | 五层验证：叶节点单元测试（覆盖率≥80%，≥3用例）→ 子节点集成（≥3用例，通过率100%）→ 模块集成（≥3用例，通过率100%）→ 宏观节点（通过率100%）→ 项目E2E（通过率100%）。 |
| **实际走查发现** | **短剧场景下"通路"的定义需要重新诠释，五层验证模型需要适配：**<br><br>**短剧场景的五层验证映射：**<br><br>**第1层：叶节点通路（场景级视频）**<br>- 验证内容：单个场景视频能正常播放、时长达标、画面无崩坏、音画同步<br>- 类比"单元测试"：每个场景是一个独立单元<br>- 验证用例：正常播放测试、画质测试、音画同步测试、时长校验<br>- 通过率标准：≥3个测试项全部通过<br><br>**第2层：子节点通路（场景组/段落）**<br>- 验证内容：多个场景串联后叙事连贯、转场自然、人物形象一致<br>- 类比"集成测试"：场景之间的衔接<br>- 验证用例：叙事连贯性测试、人物一致性测试、转场流畅度测试<br><br>**第3层：模块通路（单集内的幕）**<br>- 验证内容：一幕内的节奏合理、情节完整、情绪弧线完整<br>- 类比"模块测试"：功能模块的完整性<br><br>**第4层：宏观节点通路（单集）**<br>- 验证内容：一整集视频完整可播放、剧情起承转合完整、结尾有钩子<br>- 类比"组件测试"：单集作为独立可交付单元<br><br>**第5层：项目E2E通路（全三集）**<br>- 验证内容：三集连播叙事连贯、人物弧光完整、整体符合"第七声钟响前十章"改编要求<br>- 类比"端到端测试"：全链路验证<br><br>参考基线（小说任务）：verification.json 只有项目级验证（8项检查），没有分层验证。叶节点级、子节点级、模块级、宏观节点级的验证全部缺失。 |
| **偏差项** | 1. 总纲定义了五层验证模型，但参考实现（小说任务）只做了最顶层（项目E2E）的验证，中间四层完全跳过 — 这是参考实现的执行偏差，也是总纲落地性的问题。<br>2. 短剧场景下每层验证的"测试用例"如何定义不明确 — 开发类有明确的输入/输出断言，创作类（尤其是视频）的"测试用例"形态是什么？<br>3. "覆盖率≥80%"在视频场景下无法直接映射 — 什么是视频的"覆盖率"？关键帧覆盖率？情节点覆盖率？<br>4. "通过率100%"对于主观质量指标（如叙事连贯性、画面美感）难以实现绝对判定。 |
| **严重程度** | **P0** — 通路验证是总纲的核心机制，如果验证标准不明确，"全链路通路 = 开发完成"的结论就不可靠 |
| **优化建议** | 1. 为短剧场景定义每层验证的具体内容和判定方法：<br>   - 叶节点层：**技术验证**（硬指标检测）— 可播放性、分辨率、帧率、时长 → 自动检测，100%通过<br>   - 子节点层：**一致性验证** — 人物形象一致性、场景风格一致性 → AI评分≥阈值<br>   - 模块层：**叙事验证** — 关键情节点覆盖率、叙事连贯性 → 结构化检查清单<br>   - 宏观节点层：**完整性验证** — 单集结构完整性、钩子有效性 → 结构化检查清单<br>   - 项目E2E层：**整体验证** — 三集连贯性、改编忠实度、整体质量评分 → 综合评分<br>2. 将"覆盖率"概念重命名为领域通用的"关键指标达成率"，各领域映射自己的指标。<br>3. 明确"通过率100%"仅适用于硬指标（技术类），软指标（质量类）使用"达标率≥阈值"模式。<br>4. 参考实现的小说任务也应补齐分层验证，作为正确示范。 |

### 7.2 降标判定

| 项目 | 内容 |
|------|------|
| **预期标准** | 偏差 < 10% 且不影响核心 acceptance_criteria → 合理调整。偏差 ≥ 10% 或影响核心功能 → 降标，cp_6 FAIL。 |
| **实际走查发现** | **短剧场景下降标判定存在多个具体问题：**<br><br>1. **"承诺3集产出2集半"算不算降标？**<br>   - 按 10% 偏差规则：2.5/3 = 83.3%，偏差 16.7% ≥ 10% → 降标<br>   - 但"半集"是什么定义？是时长减半？还是剧情推进到一半？还是只有画面没有配音？<br>   - 小说任务中"章"是原子单元，没有"半章"概念。但短剧可以有"半成品"（有画面无配音、有配音无画面、只有前半段等），降级程度有别。<br><br>2. **10%偏差规则的计算基数不明确：**<br>   - 按集数算：3集 → 10% = 0.3集 → 少0.3集算偏差？不合理<br>   - 按总时长算：3集×10分钟=30分钟 → 10% = 3分钟 → 合理<br>   - 按关键情节点数算：30个关键节点 → 10% = 3个 → 合理<br><br>3. **多维度降标的复合判定：**<br>   - 如果集数达标（3/3），但画质降标（4K变1080P），算降标吗？<br>   - 如果画面齐全但没有配音和BGM，算降标吗？降多少？<br>   - 总纲只有单一的"偏差≥10%→降标"，无法处理多维度复合降标的情况。<br><br>参考基线（小说任务）：cp_6 检查中"实际产出≥承诺量70%"是 cp_7 的标准，cp_6 用的是"偏差<10%"，但小说任务的验证报告中两个阈值混用了。 |
| **偏差项** | 1. 10%偏差规则的计算基数在短剧场景下不明确（集数？时长？情节点数？）。<br>2. 缺少"部分交付"的分级定义 — 短剧有多种降级交付形态（无声、无特效、低分辨率、半成品等），总纲只有"达标/降标"二元判定。<br>3. cp_6（偏差<10%）和 cp_7（实际产出≥70%）的阈值关系不清晰 — 70%和90%两个阈值的适用边界是什么？<br>4. 多维度复合降标的判定规则缺失。 |
| **严重程度** | **P1** — 降标判定不清晰会导致 cp_6 检查点执行不一致，也影响用户对交付质量的预期管理 |
| **优化建议** | 1. 明确定义短剧场景的偏差计算基数：**按关键叙事节点数**计算（而非集数或时长），因为集数是离散的、时长可能注水。<br>2. 建立"交付质量分级"体系：<br>   - **S级（完整交付）**：所有维度达标，偏差<5%<br>   - **A级（轻微偏差）**：核心指标达标，次要指标偏差5%-10%，标记 [ADJUSTED]<br>   - **B级（降标交付）**：核心指标偏差10%-30%，需明确声明降标内容和原因<br>   - **C级（不合格）**：核心指标偏差>30%或关键功能缺失 → cp_6 FAIL<br>3. 区分 cp_6 和 cp_7 的阈值定位：<br>   - cp_6（通路验证）：核心功能偏差<10% → 通过<br>   - cp_7（交付收尾）：实际产出≥承诺量70% → 可交付（允许更大偏差但必须标记）<br>4. 多维度复合降标采用"短板原则"：取偏差最大的核心维度作为整体偏差率。 |

## 四、Phase 8：交付收尾

### 8.1 可运行性检查

| 项目 | 内容 |
|------|------|
| **预期标准** | 创作类：内容能正常解析（HTML 能打开、Markdown 能渲染）。 |
| **实际走查发现** | **短剧的"可运行"定义需要大幅扩展：**<br><br>**短剧可运行性检查清单：**<br><br>1. **文件层面（硬检查）**<br>   - 视频文件存在且非空<br>   - 文件格式正确（.mp4 / .mov 等）<br>   - 视频编码合规（H.264 / H.265）<br>   - 音频轨道存在且编码合规<br>   - 文件可正常播放（无损坏）<br><br>2. **技术层面（硬检查）**<br>   - 分辨率达标（如 1920×1080）<br>   - 帧率达标（如 24fps / 30fps）<br>   - 时长在目标范围内<br>   - 音画同步（偏差<100ms）<br>   - 无黑屏/花屏/静音段<br><br>3. **内容层面（软检查）**<br>   - 每集有片头/片尾标识<br>   - 集数标注正确<br>   - 字幕（如有）与语音对应<br>   - 三集文件命名规范、顺序正确<br><br>参考基线（小说任务）：health_check.json 应该包含可运行性检查结果，但小说任务的可运行性很简单 — .docx文件能打开、格式正确。 |
| **偏差项** | 1. 创作类的"内容可正常解析"标准完全是文字文档语境，不适用于视频文件。<br>2. 缺少视频专属的可运行性检查项：编码、分辨率、帧率、音画同步、播放完整性等。<br>3. 可运行性检查的"hard/soft"分级在短剧场景下需要重新定义 — 视频编码错误是 hard fail，片头设计简陋是 soft issue。 |
| **严重程度** | **P1** — 可运行性检查标准不匹配会导致 cp_7 检查点无法有效把关，可能交付无法播放的视频文件 |
| **优化建议** | 1. 为"短剧/视频"子类型定义独立的可运行性检查清单，分为三级：<br>   - **Hard Fail级**：文件不存在/损坏、编码错误、无法播放、音轨缺失 → cp_7 FAIL<br>   - **Soft Warn级**：分辨率不达标、帧率波动、轻微音画不同步 → 标记 warning，不 FAIL<br>   - **Quality级**：片头片尾不规范、命名不规范 → 记录，不阻塞交付<br>2. 提供自动化检测脚本（如用 ffprobe 检测视频元数据），实现可运行性检查的自动化。<br>3. cp_7 检查点增加"视频元数据校验"作为必检项。 |

### 5.1 监督者检查点（cp_0 到 cp_7）

**总检查项**：8个检查点在短剧场景下的适用性评估

#### cp_0：Phase 1 自审质量

#### cp_1：Phase 2 路径锚定

| 项目 | 内容 |
|------|------|
| **预期标准** | 节点数在 structural_parameters 范围内；prev/next 字段形成完整链，无断点；无循环引用；路径终点描述与 spec.one_liner 一致。 |
| **实际走查发现** | 基本适用。短剧的路径图可以是"原著文本 → 剧本改编 → 分镜设计 → 视频生成 → 后期合成 → 最终交付"，节点数和链路都能清晰定义。<br>但有一个问题：短剧生成的路径是**线性流水线**（文本→剧本→分镜→视频→后期），而不是开发类的**并行模块组装**。总纲的路径图模型是基于"多个宏观节点并行"假设的，短剧场景下可能是单链串行。 |
| **偏差项** | 路径模型假设多节点并行，短剧场景可能是单链串行 — prev/next 链是对的，但"宏观节点并行"的心智模型不匹配。 |
| **严重程度** | P2 |
| **优化建议** | 路径图支持两种模式：并行分支型（开发类）和串流水线型（创作/短剧类），两种模式的检查点标准略有差异。 |

#### cp_2：Phase 3 节点识别

| 项目 | 内容 |
|------|------|
| **预期标准** | 每层级节点数在 structural_parameters 范围内；overlap_detection 无 redundancy 标记；acceptance_criteria 每个维度有对应节点。 |
| **实际走查发现** | 部分不适用。问题在于：<br>1. 短剧的节点（集、场景、分镜）天然按顺序排列，不存在"两个节点映射到同一 acceptance_criteria 维度"的重叠问题 — overlap_detection 意义不大。<br>2. 但短剧有另一种节点间依赖问题：**时序依赖** — 场景A必须在场景B之前，因为剧情有因果关系。总纲的节点识别不检查时序合理性。<br>3. acceptance_criteria 多维度覆盖问题已在 Phase 5 覆盖度校验中讨论，此处不再重复。 |
| **偏差项** | overlap_detection 不适用于时序型任务；缺少时序合理性检查。 |
| **严重程度** | P2 |
| **优化建议** | 节点识别增加"时序依赖校验"：检查节点顺序是否符合叙事逻辑/因果逻辑，避免出现剧情顺序颠倒。 |

#### cp_3：Phase 4 接口契约

| 项目 | 内容 |
|------|------|
| **预期标准** | consumer.consumes / producer.outputs ≥ 0.80；semantic_match 无 forbidden type_conversion；数值约束与 spec.json scope 一致。 |
| **实际走查发现** | **接口契约模型在短剧场景下需要重新定义。**<br><br>开发类的接口是函数调用/API 调用，有明确的输入输出 Schema。短剧场景的"接口"是什么？<br>- 集与集之间：前集结尾 → 后集开头 的剧情衔接（叙事接口）<br>- 场景与场景之间：转场方式、叙事连贯性（场景接口）<br>- 分镜与分镜之间：镜头衔接、视觉连贯性（镜头接口）<br><br>这些"接口"的 consumer.consumes / producer.outputs ≥ 80% 如何计算？剧情衔接度如何量化？<br><br>参考基线（小说任务）：interfaces.json 定义的是文件之间的 schema 匹配（spec→nodes→leaf→content），不是节点间的接口契约。这实际上是"流水线接口"而非"组件接口"。 |

### 5.2 Sprint Contract

| 项目 | 内容 |
|------|------|
| **预期标准** | acceptance_criteria 从 Phase 0 收敛引擎自动派生：L4 hard 约束 → must_have；L4 soft 约束 → should_have；L3 结构参数 → numeric_targets。Contract 检查：acceptance_criteria 必须包含 Phase 0 L4 的所有 hard 约束。 |
| **实际走查发现** | **从测试输入派生出的 Sprint Contract 分析：**<br><br>测试输入："用第七声钟响前十章生成一部短剧的前三集"<br><br>**Phase 0 L3 结构参数提取：**<br>- 集数：3集（从"前三集"提取）<br>- 原著范围：前十章（从"第七声钟响前十章"提取）<br>- 作品类型：短剧（从"短剧"提取）<br>- 提取率：3/5 个关键参数 ≈ 60% → 高于 50% 阈值，继续<br>- 未提取参数：每集时长、视频格式、分辨率、风格等 → inferred<br><br>**Phase 0 L4 领域规则（假设）：**<br>Hard 约束（短剧类）：<br>- 每集必须有完整的起承转合结构<br>- 改编必须忠实于原著核心情节和人物设定<br>- 视频必须可正常播放（技术完整性）<br>- 人物形象在全剧中保持一致<br><br>Soft 约束（短剧类）：<br>- 每集结尾有钩子（悬念）<br>- 画面风格统一<br>- 配音质量达标<br><br>**派生的 acceptance_criteria：**<br>- must_have：3集完整产出、改编自前十章、视频可播放、人物一致<br>- should_have：每集有钩子、画面风格统一、配音达标<br>- numeric_targets：集数=3、原著范围=第1-10章 |
| **偏差项** | 1. **短剧类的 L4 领域规则库为空。** 总纲定义了 L4 规则校验机制，但短剧类的 hard/soft 约束清单没有定义。收敛引擎在 L4 层无法加载短剧专属规则。<br>2. **"改编忠实度"难以量化为 acceptance_criteria。** 开发类的 must_have 是明确的功能点，创作类（尤其是改编）的"忠实度"是连续光谱，难以用 must_have/should_have 二元分类。<br>3. **numeric_targets 不够。** 短剧的数值目标不仅是集数，还包括时长、分辨率、帧率等，但这些在 Phase 0 L3 可能提取不到（用户没说），需要 inferred 后写入 contract。 |
| **严重程度** | **P1** — Sprint Contract 是监督者的核心依据，如果派生规则不完整，验收标准就不明确 |
| **优化建议** | 1. 建立短剧类的 L4 领域规则库，包含至少 5 条 hard 约束和 5 条 soft 约束。<br>2. 对于"改编忠实度"这类连续性指标，采用"关键情节点保留率 ≥ X%"的量化方式，而非 must_have 二元判定。<br>3. numeric_targets 支持"推断值"字段，明确标记哪些数值是用户明确指定的、哪些是 Agent 推断的。推断值在 Phase 1 自审中作为 inferred 项统计。 |

### 5.3 领域变体

| 项目 | 内容 |
|------|------|
| **预期标准** | 4 大领域（创作/开发/分析/研究），每个领域有子类型，每个子类型有 Phase 领域变体定义。 |
| **实际走查发现** | **短剧生成的领域归属分析：**<br><br>1. **是否属于"创作"领域？**<br>   - 表面上是 — 生成内容（视频）<br>   - 但创作类现有子类型（fiction, article, script, marketing）都不匹配<br>   - 本体定义中 creative 的 subtypes 有 `script`（剧本），但短剧生成是生成视频，不是写剧本<br>   - 短剧生成 = 剧本创作 + 分镜设计 + 视频生成 + 后期合成，是多模态创作<br><br>2. **是否跨领域？**<br>   - 剧本改编 → 创作类（文字创作）<br>   - 分镜设计 → 创作类（视觉设计）<br>   - 视频生成 → 创作类？还是一个新领域？<br>   - 技术合成（剪辑、配音、配乐）→ 开发类？还是创作类？<br><br>3. **4领域是否够用？**<br>   - 短剧生成可以塞进"创作"领域，但需要新增子类型和大量变体定义<br>   - 另一种思路：短剧生成是**跨领域任务**（创作+开发），需要跨领域协作机制<br>   - 总纲已有"跨领域处理"规则：子节点 maps_to_root_goal 指向不同领域 → 重新执行 Phase 0 收敛引擎 |
| **偏差项** | 1. 创作类缺少"短剧/视频生成"子类型 — 现有子类型（fiction, article, script, marketing）都是文字导向的。<br>2. 短剧生成天然是多模态的（文字→图像→视频→音频），但总纲的领域变体是单模态假设。<br>3. "视频生成"究竟是创作的子类型，还是一个独立的第5领域，需要明确 — 它既有创作属性（内容创作），也有开发属性（技术流程、工具链调用）。 |
| **严重程度** | **P0** — 领域归属不明确会导致 Phase 0 收敛引擎分类错误，后续所有 Phase 的变体定义都用错 |
| **优化建议** | 1. **短期方案**：在创作领域下新增 `short_video`（短剧/短视频）子类型，定义完整的 8 Phase 变体。这是最快落地的方式。<br>2. **中期方案**：引入"多模态任务"概念 — 一个任务可以包含多个模态阶段，每个阶段使用对应领域的变体。例如短剧生成：<br>   - Phase 1-3：创作·剧本（文字模态）<br>   - Phase 4-5：创作·分镜（视觉模态）<br>   - Phase 6：视频生成（视频模态）<br>   - Phase 7-8：集成验证（多模态混合）<br>3. **长期方案**：评估是否需要第 5 个领域（如"生成"或"多模态"），或者将 4 领域扩展为"创作/开发/分析/研究/生成"5领域。 |

### 5.4 双程规则

| 项目 | 内容 |
|------|------|
| **预期标准** | 去程（Phase 1-5）不写代码，只做设计；回程（Phase 6-8）不设计，只做实现。回程发现设计错误 → 不得自行修改，必须标记 suspect_design_issue → 上报 supervisor → 评估是否回退去程。 |
| **实际走查发现** | **短剧场景下去程/回程的边界分析：**<br><br>**去程（设计）应该包含：**<br>- Phase 1：确定短剧整体规格（集数、时长、风格、改编范围）<br>- Phase 2：规划改编路径（原著章节 → 剧集分配）<br>- Phase 3：识别每集的场景和关键节点<br>- Phase 4：定义场景间的叙事衔接契约<br>- Phase 5：拆解到分镜/镜头级（叶节点）<br><br>**回程（实现）应该包含：**<br>- Phase 6：逐场景生成视频 + 配音 + BGM<br>- Phase 7：逐集验证 + 整体验证<br>- Phase 8：交付<br><br>**边界模糊点：**<br><br>1. **分镜脚本是设计还是实现？**<br>   - 按总纲：去程只做设计不写代码/内容<br>   - 但分镜脚本是"设计文档"还是"内容初稿"？<br>   - 如果分镜算设计，那么去程产出分镜脚本（文字+画面描述），回程生成视频<br>   - 如果分镜算实现的一部分，那么去程到场景级就停，回程再细分镜<br>   - 类比开发类：详细设计文档是去程产物，写代码是回程 → 分镜脚本 = 详细设计，属于去程<br><br>2. **回程发现分镜有问题能改吗？**<br>   - 场景：生成第3集视频时，发现第2集结尾的分镜设计导致第3集开头接不上<br>   - 按双程规则：不得自行修改 → 上报 → 评估 → 可能回退到去程修改分镜<br>   - 但实际操作中，视频生成的试错成本高，很多问题只有生成后才发现（如人物形象不一致、场景衔接不自然）<br>   - 如果每次都要完整回退去程，效率极低 |
| **偏差项** | 1. 短剧生成的"设计"和"实现"边界比纯文字创作或纯开发更模糊 — 分镜设计既是设计也是创作内容。<br>2. 视频生成的试错性强，很多问题只能在实现阶段发现，严格的"回程不改设计"规则会导致大量回退，效率低下。<br>3. 双程边界违规判定中，"修改 spec.json 的 acceptance_criteria"是违规，但"修改分镜脚本的画面描述"算什么层级的变更？总纲未明确定义变更的严重度分级。 |
| **严重程度** | **P1** — 边界不清晰会导致双程规则执行不一致，效率和质量难以平衡 |
| **优化建议** | 1. 明确短剧场景的去程终点：**分镜脚本（storyboard）是去程终点的产出物**，即叶节点的定义是"一个分镜脚本条目"，而非"一个视频片段"。回程从分镜脚本开始生成视频。<br>2. 引入"设计变更分级"：<br>   - **微调整**（L1）：修改分镜的构图、镜头角度等视觉细节 → 回程可自主决定，标记 [ADJUSTED]<br>   - **小变更**（L2）：修改场景内的动作、对白 → 需标记 [DESIGN_CHANGE_L2]，记录原因，不触发回退<br>   - **大变更**（L3）：增减场景、修改剧情走向、调换顺序 → 必须上报 supervisor，评估是否回退去程<br>   - **核心变更**（L4）：修改 acceptance_criteria → 必须回退 Phase 1 重做<br>3. 对短剧场景设置"回程微调预算"：每集允许最多 N 个 L2 级变更，超过则触发回退评估。 |

### 5.5 最小可行链路

| 项目 | 内容 |
|------|------|
| **预期标准** | 8个维度评估完成度，任一维度低于止损阈值则标记 blocked。综合完成度 = 各维度加权平均。 |
| **实际走查发现** | **短剧生成任务的最小可行链路各维度完成度估算：**<br><br>```<br>| 维度 | 完成度 | 止损阈值 | 状态 | 说明 |<br>|------|--------|---------|------|------|<br>| 意图锚定（Phase 1） | 60% | 50% | 未阻塞 | 44问可适配短剧但需大量推断，inferred比例可能>30% |<br>| 门禁强制执行 | 80% | 50% | 未阻塞 | 门禁引擎机制通用，不依赖领域 |<br>| 去程拆解（Phase 2-5） | 30% | 30% | 临界 | 结构定义存在但领域变体不匹配，实际拆解质量存疑 |<br>| 目标对齐（防漂移） | 50% | 40% | 未阻塞 | 监督者机制通用，但检查点量化标准不匹配 |<br>| Sprint Contract | 30% | 无单独阈值 | - | 短剧L4规则库为空，派生规则不完整 |<br>| 回程实现（Phase 6-8） | 20% | 无单独阈值 | - | 质量指标缺失，验证标准不明确，交付物不完整 |<br>| IDE 通信 | 60% | 无单独阈值 | - | 与领域无关，保持不变 |<br>| 自动化执行 | 15% | 20% | 低于阈值 | 视频生成需要调用外部工具（AI视频生成API），工具链未集成 |<br>```<br><br>**综合完成度估算：约 35-40%**（低于小说任务的 50-55%）<br><br>**阻塞维度分析：**<br>1. **自动化执行（15% < 20%阈值）** — 视频生成需要外部工具/API集成，当前总纲的自动化执行是基于本地代码生成的，不包含AI视频生成工具链。但止损阈值表中"自动化执行"的后果是"全链路无法自动跑通。可手动执行，不阻塞" — 所以标记为 blocked 但不阻塞其他维度。<br>2. **去程拆解（30% = 30%阈值）** — 刚好卡在阈值上，实际能否进入 Phase 6 存疑。如果领域变体不匹配导致拆解质量差，即使数量上通过了，质量上也是不合格的。<br>3. **回程实现（20%）** — 虽然没有单独的止损阈值，但质量指标和验证标准的缺失是实质性阻塞。 |
| **偏差项** | 1. 最小可行链路的 8 个维度是通用的，但缺少"工具链集成度"维度 — 短剧生成严重依赖外部AI视频生成工具，工具链是否就绪直接影响可行性。<br>2. 止损阈值表只有 5 个维度有阈值，其他 3 个维度（Sprint Contract、回程实现、IDE通信）没有阈值定义。<br>3. "自动化执行"维度的阈值是 20%，但短剧场景下自动化执行的瓶颈不在流程自动化，而在工具集成 — 即使流程 100% 自动化，没有视频生成工具也无法产出。 |
| **严重程度** | **P1** — 最小可行链路评估是项目可行性判断的核心依据，如果维度和阈值不匹配，会导致错误的可行性判断 |
| **优化建议** | 1. 新增"外部工具链就绪度"维度，纳入最小可行链路评估，专门衡量外部 API/工具的可用性和集成度。<br>2. 为所有 8 个维度设定止损阈值，不留空白。<br>3. 止损阈值按领域差异化 — 创作类的回程实现阈值可以低于开发类（因为创作类的质量验证主观性强），但工具链维度的阈值应更高。<br>4. 引入"领域成熟度系数"：新领域（如短剧）的最小可行链路完成度天然低于成熟领域（如开发），评估时应考虑领域成熟度。 |

### 5.6 递归自相似

| 项目 | 内容 |
|------|------|
| **预期标准** | 每一层级都执行完整的 Phase 1-8 循环。跨领域情况：子节点 maps_to_root_goal 指向不同领域 → 重新执行 Phase 0 收敛引擎。 |
| **实际走查发现** | **短剧场景下递归自相似性的分析：**<br><br>**1. 每一集是否都是完整的 Phase 1-8 循环？**<br>   - 理论上是 — 每一集都有自己的 spec（该集的目标）、path（该集的叙事路径）、nodes（该集的场景）、interfaces（场景间衔接）、leaf_nodes（分镜）、实现、验证、交付<br>   - 但实际操作中，每一集的 Phase 1 不需要完整 44 问自审 — 很多信息继承自项目级 spec<br>   - 总纲定义递归层级的 Phase 1 是"Agent 读取父节点的 spec，自问自答确认本节点的目标"，问题数减少，这是合理的<br><br>**2. 跨领域情况如何处理？**<br>   - 短剧生成内部可能存在跨领域：剧本改编（创作·文字）→ 分镜设计（创作·视觉）→ 视频生成（？）→ 后期合成（开发·工具链）<br>   - 但这些是同一任务内部的不同阶段/模态，不是不同子节点分属不同领域<br>   - 总纲的跨领域处理是"子节点 maps_to_root_goal 指向不同领域" — 这是指不同子节点属于不同领域（如一个模块是开发，另一个模块是研究）<br>   - 短剧生成的"跨模态"与总纲的"跨领域"是不同概念 |
| **偏差项** | 1. 递归自相似的前提是"每一层都是同一类任务"，但短剧生成的不同层级可能属于不同模态（文字→视觉→视频），虽然都在创作领域内，但工具、方法、验收标准完全不同。<br>2. 总纲定义了"跨领域"处理机制，但没有定义"跨模态"处理机制 — 同一领域内不同模态的转换如何处理。<br>3. 递归深度公式假设每层都是同质的（都是文字或都是代码），但短剧的每层模态不同，深度计算应该考虑模态转换。 |
| **严重程度** | **P2** — 不阻塞基本执行，但影响递归架构的优雅性和可扩展性 |
| **优化建议** | 1. 在领域变体内引入"模态"维度，每个子类型可以包含多个模态阶段，每个阶段有自己的 Phase 变体。<br>2. 递归自相似的"自相似"定义从"Phase 结构相同"细化为"Phase 框架相同 + 领域/模态变体不同" — 框架一致，细节按模态变化。<br>3. 跨模态转换点（如文字分镜→视频生成）作为特殊的 Phase 4 接口契约处理，明确转换规则和验收标准。 |

## 七、核心结论

总纲 V3.9.1 的双程递归架构和监督者机制在**框架层面**适用于短剧生成场景 — 去程拆解、回程合成、全程监督的核心逻辑成立。但在**领域变体层面**存在大量缺口，核心问题可以归纳为三类：

1. **领域定义缺失**：创作领域缺少"短剧/视频"子类型，导致所有 Phase 的变体定义、L4 规则库、质量指标、验收标准全部缺失。

2. **模态转换空白**：短剧生成是多模态任务（文字→视觉→视频→音频），总纲的递归自相似模型基于同模态假设，缺少跨模态转换的处理机制。

3. **量化标准不足**：视频生成的很多质量指标（画面一致性、叙事连贯性、表演质量）本质上是主观的，总纲现有的硬量化标准（覆盖率≥80%、字数偏差≤30%）无法直接映射，需要建立"AI评分+结构化检查清单"的混合验证体系。

**建议下一步**：优先修复 5 个 P0 级偏差（领域归属 → 递归深度 → 接口模型 → 质量指标 → 验证体系），形成短剧生成场景的完整领域变体定义，然后才能端到端跑通短剧生成任务。
