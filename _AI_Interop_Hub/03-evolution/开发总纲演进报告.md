## 二、执行验证：V3.6 门禁引擎落地

### 2.1 问题诊断

第三方审查（Superpowers 技能）指出 V3.6 总纲的**根因：门禁引擎是"纸面协议"**——JSON 中定义了 gate_file/produce_file 链，但 orchestrator.js 并未真正解析这些定义来执行检查。Agent 可以"理解"门禁的存在，但代码层面没有强制约束。

### 2.2 修复方案

完全重写了 `orchestrator.js`，新增以下 6 大核心能力：

| 能力 | 函数 | 说明 |
|------|------|------|
| 动态工作区检测 | `detectWorkspaceDir()` | 支持 `--workspace` 参数、`CONSTITUTION_WORKSPACE` 环境变量、`process.cwd()` 三级回退 |
| JSON 门禁解析 | `loadGateEngine()` | 真正读取并解析 `开发总纲_可执行版.json` 中的 `phase_gate_engine` 章节 |
| 逐项门禁检查 | `checkSingleGate()` | 使用 `fs.existsSync` 强制检查每个 Phase 的 gate_file 和 produce_file |
| 反跳过机制 | `detectSystemicSkip()` | 检测 ≥2 个 produce_file 缺失，认定为系统性故障并阻断 |
| 智能路径解析 | `extractPath()` | 解析含中文注释的 JSON 路径，剥离中文说明，处理 `或` 分支 |
| 独立门禁模式 | `--gate-check` | 不创建新任务，仅验证已有任务的门禁链完整性 |

### 2.3 修复过程中的 Bug

| Bug | 根因 | 修复 |
|-----|------|------|
| Phase 1 gate_file 解析错误 | `"gate_file": "无（Phase 1 是起点）"` 被当作路径 | 添加 `startsWith('无')` 跳过逻辑 |
| Phase 5 produce_file 目录检查 | `"produce_file": "{task_dir}/nodes/"` 是目录而非文件 | 添加 `endsWith('/')` 目录检查 + leaf_nodes.json 替代 |
| Phase 8 produce_file 数组格式 | 使用 `produce_file_1/2/3` 而非单一字段 | 遍历所有 `produce_file` 前缀的 key |
| Phase 5/6 gate_file_2 "或" 分支 | `"nodes/ 或 leaf_nodes.json"` 导致路径解析失败 | extractPath 中分割 "或" 并取第一部分 |

> **验证结果：** `node orchestrator.js --gate-check` 执行成功，**全链路通过 (8/8)**。门禁引擎从"纸面协议"升级为"代码强制执行"。

## 四、架构升级：Skill + MCP + Plugin 混合架构

### 4.1 从"能不能转成 Skill"到"混合架构"

用户提出了关键问题：**"总纲转化成一个开发 Skill 能不能行得通？"**

**Skill 能解决的：**
- ✅ 路径配置（动态工作区检测）
- ✅ 可脚本化的门禁规则
- ✅ 标准化分发（跨项目复用）

**Skill 无法解决的：**
- ❌ 代码级门禁强制执行（需要 MCP）
- ❌ 跨 IDE 实例通信（需要文件系统）
- ❌ 运行时状态管理（需要持久化）

### 4.2 三层混合架构

```
┌─────────────────────────────────────────────────┐
│  Skill 层 — 提示注入                              │
│  Skill 定义文件 → Agent 启动时注入 Prompt          │
│  → 门禁规则 + 工作流规范                           │
├─────────────────────────────────────────────────┤
│  MCP 层 — 代码执行                                │
│  MCP Server → gate-check / workspace-init /       │
│  health-check 工具 → fs.existsSync 强制执行        │
├─────────────────────────────────────────────────┤
│  Plugin 层 — 分发部署                             │
│  Plugin Package → 跨 IDE 安装                     │
│  → VS Code / Cursor / Trae                       │
└─────────────────────────────────────────────────┘
```

| 层级 | 职责 | 技术手段 | 解决的问题 |
|------|------|----------|------------|
| **Skill** | 提示注入层 | Skill 定义文件 → Agent 上下文 | Agent 知道"应该做什么" |
| **MCP** | 代码执行层 | MCP Server → 工具调用 | Agent 被强制"必须做什么" |
| **Plugin** | 分发部署层 | Plugin 包 → 跨 IDE 安装 | 一次部署，所有 IDE 可用 |

### 4.3 三步执行计划

1. **Step 1 ✅ 已完成** — 修复 orchestrator.js（动态工作区 + JSON 门禁解析 + 代码级强制执行）
2. **Step 2 · 待执行** — 封装为 Skill（将 AGENT_INSTRUCTION.md + 开发总纲打包为 Skill 定义文件）
3. **Step 3 · 待执行** — 添加 MCP 层（将 orchestrator.js 核心能力封装为 MCP Server 工具）

## 五、知识补充：本体论（Ontology）系统性研究

用户分享了一篇抖音文章《为什么 Agent 都开始做本体（Ontology）？》，触发了对本体论的系统性调研。这是**对开发总纲的"决定性补充"**——我们不再基于猜测开发，而是基于行业标准。

### 5.1 Skan AI AOW：14 个规范实体

Skan AI 的 Agentic Operations Workbench (AOW) 定义了 14 个规范实体，形成 4 层认知-操作栈 [1]：

| 层级 | 实体 | 关系 |
|------|------|------|
| **认知层** | Objective（目标）、Intent（意图）、Context（上下文）、Policy（策略） | 输入层，定义"为什么"和"什么约束" |
| **操作层** | Orchestrator（编排器）、Agent（代理）、Skill（技能）、Result（结果） | 执行层，定义"谁做什么" |
| **结果层** | Outcome（产出）、Confidence（置信度）、Assurance Level（保障级别） | 评估层，定义"做得怎么样" |
| **反馈层** | Feedback（反馈）、Memory（记忆）、Guardian（守护者） | 闭环层，定义"如何改进" |

### 5.2 Palantir 三层本体

Palantir 的 Ontology 架构将本体分为三层 [2]：

| 层级 | 定义 | 核心问题 | 对应总纲概念 |
|------|------|----------|--------------|
| **Semantic Layer** | 定义"存在什么"——实体、关系、属性 | What exists? | spec.json 的 goal_trace / acceptance_criteria |
| **Kinetic Layer** | 定义"能做什么"——操作、动作、变换 | What can be done? | Phase 4 interfaces.json 的接口契约 |
| **Dynamic Layer** | 定义"AI 如何参与"——决策、推理、学习 | How does AI participate? | Phase 5 递归拆解 + Phase 6 实现 |

### 5.3 OAG vs RAG：本体增强生成

**OAG（Ontology-Augmented Generation）** 是比 RAG 更进一步的模式：不是将文本注入 LLM 上下文，而是将**本体对象（结构化实体 + 关系 + 约束）**注入上下文 [3]。这直接减少了幻觉，因为 LLM 在结构化约束下推理，而非在自由文本中猜测。

> **关键发现：** 开发总纲 V3.6 的 `spec.json → path.json → nodes.json → interfaces.json` 链实际上已经是一个**隐式的 OAG 管道**——每个 Phase 产出结构化 JSON（而非自由文本），下游 Phase 消费上游的结构化产出。但我们没有显式定义 Ontology 层，也没有将本体对象作为"一等公民"来管理。

### 5.4 神经符号融合（Neurosymbolic Fusion）

Skan 的神经符号融合架构将 AI 流程分为五个阶段 [1]：

```
Observe（观察）→ Structure（结构化）→ Reason（推理）→ Act（执行）→ Measure（度量）→ 回到 Observe
```

对应到总纲：
- **Phase 1-2** → Observe（收集和理解需求）
- **Phase 3-4** → Structure（节点识别和接口契约）
- **Phase 5** → Reason（递归拆解）
- **Phase 6** → Act（实现）
- **Phase 7-8** → Measure（验证和交付）

### 5.5 Gartner 四级比例治理

Gartner 提出了 Guardian Agents（守护代理）的四级比例治理模型 [4]：

| 级别 | 治理力度 | 触发条件 | 总纲对应 |
|------|----------|----------|----------|
| Level 1 | 日志记录 | 所有操作 | manifest.json + health_check.json |
| Level 2 | 告警通知 | 偏离阈值 | verification.json 的 actual_vs_target |
| Level 3 | 熔断阻断 | 严重偏离 | detectSystemicSkip() + blockage.json |
| Level 4 | 人工升级 | 3 次失败 | three_strikes 机制 |

## 六、差距分析：6 层标准框架对照

### 6.1 框架来源

综合 Anthropic Building Effective Agents [5]、2026 年五大框架对比 [6]、Governance Gaps 论文 [7]、MCP/A2A/AG-UI 协议分析 [8] [9]，我们提炼出 6 层标准 Agent 开发框架：

```
Layer 6: 治理 Governance
├── Guardian 独立性
├── 人工升级
└── 审计回放

Layer 5: 生命周期 Lifecycle
├── Tracing 追踪
├── Feedback Loop
└── AgentOps

Layer 4: 协议 Protocol
├── MCP — Agent↔Tool
├── A2A — Agent↔Agent
└── AG-UI — Agent↔User

Layer 3: 本体 Ontology
├── 语义层 — 实体/关系
├── 动力层 — 操作/动作
└── 动态层 — AI 参与

Layer 2: 编排 Orchestration
├── 5 种编排模式
├── Handoff + Timeout
└── Idempotency

Layer 1: 增强 LLM
├── Prompt Engineering
├── Memory 记忆
└── Tool Use
```

### 6.2 完成度评估

| 层级 | 完成度 | 状态 |
|------|--------|------|
| Layer 1 · 增强 LLM | **30%** | 有基础（AGENT_INSTRUCTION.md、44 问自审清单） |
| Layer 2 · 编排 | **40%** | 有基础（8 Phase 顺序管道、反跳过机制） |
| Layer 3 · 本体 | **25%** | 有概念但缺实现（隐式结构化管道，无显式 Ontology） |
| Layer 4 · 协议 | **0%** | 完全缺失（仅文件轮询，无 MCP/A2A/AG-UI） |
| Layer 5 · 生命周期 | **10%** | 几乎缺失（仅 manifest.json + health_check.json） |
| Layer 6 · 治理 | **5%** | 几乎缺失（仅 three_strikes + blockage.json） |

**综合完成度：约 18%**

> ⚠️ **纠偏标注（2026.07.16）**：上述 18% 是基于"6 层行业标准框架"的评估——这是一个**错误的坐标系**。6 层框架中的 L4（协议标准化）、L5（AgentOps 监控）、L6（四级治理）在当前阶段（端到端链路尚未跑通）根本不需要。如果以"一句话→最终目标的最小可行链路"重新评估，当前完成度应在 **50-60%**。详见下文"纠偏与反思"章节。

### 6.3 4 个冲突 / 偏离

| # | 冲突 | 我们的做法 | 行业标准 | 严重程度 |
|---|------|-----------|----------|----------|
| 1 | **Self-check 模式** | Agent 自检替代独立监督者 | Guardian 必须独立于执行 Agent（Gartner L3） | 🔴 高 |
| 2 | **仅顺序管道** | 8 Phase 顺序执行，无分支/并行 | 5 种编排模式：Sequential / Fan-out / Supervisor / Hierarchical / Debate | 🟡 中 |
| 3 | **文件轮询** | 通过文件系统 .pending/.result 异步通信 | MCP（Agent↔Tool）、A2A（Agent↔Agent）标准协议 | 🟡 中 |
| 4 | **缺失 OAG** | 隐式结构化管道，无显式本体层 | OAG（本体增强生成）将本体对象注入 LLM 上下文 | 🔴 高 |

### 6.4 各层详细缺口

| 层级 | 已有 | 缺失 |
|------|------|------|
| **L1 · 增强 LLM** | AGENT_INSTRUCTION.md（Prompt）、44 问自审清单 | 跨任务 Memory 持久化、Tool Use 标准化 |
| **L2 · 编排** | 8 Phase 顺序管道、反跳过机制 | Fan-out / Supervisor / Hierarchical / Debate 模式、Handoff Schema、Timeout、Idempotency |
| **L3 · 本体** | 隐式结构化管道（spec→path→nodes→interfaces） | 显式本体定义、OAG 模式、Palantir 三层本体映射 |
| **L4 · 协议** | 文件系统 .pending/.result 轮询 | MCP Server 封装、A2A 通信、AG-UI 交互 |
| **L5 · 生命周期** | manifest.json、health_check.json | Tracing（全链路追踪）、Feedback Loop、AgentOps 监控 |

## 八、优先级执行路线图

基于"独立性优先"原则（被依赖的先做，不依赖其他层的先做）：

| 优先级 | 层级 | 任务 | 依赖 | 预估工作量 |
|--------|------|------|------|------------|
| **P0** | L3 本体 | 在 `开发总纲_可执行版.json` 中添加 `ontology` 章节 | 无 | 中 |
| **P1** | L2 编排 | 补充 Handoff Schema、Timeout 配置、Idempotency Key | 无 | 小 |

| **P3** | L5 生命周期 | 添加 Tracing 追踪 + Feedback Loop | L3 本体 | 大 |

> **关键原则：** P0-P2 是"低垂的果实"——不需要 MCP 封装，直接在 JSON 总纲中定义即可。P3-P5 需要 MCP 层支持，应在 P4（MCP 封装）完成后统一推进。当前总纲 V3.6 的 ~18% 完成度，在执行 P0-P2 后可提升至 **~45%**。

## 纠偏与反思：研究漂移带的识别与修正

> **2026.07.16 审计结论**：基于"一句话生成最终目标"这一唯一标尺，对 10 轮演进进行系统性审计，识别出 Sessions 4-6 形成的"研究漂移带"，并制定三条纠偏规则。

### 审计方法论

以总纲 Phase 1 自审清单中的核心逻辑评估每一轮：**该轮产出是否直接缩短了"用户输入一句话 → 系统产出不偏离预期的完整结果"之间的距离？**

"偏离预期"不是指研究没有价值，而是指研究是否脱离了**工程落地**这条主线，滑向了学术完备性或行业对标。判断标准：**"这个研究结果会被哪个代码文件/函数使用？"**——如果回答不上来，说明研究时机未到。

### 逐 Session 评估

| Session | 评估 | 判断依据 |
|---------|------|----------|
| Session 1 | ✅ 符合预期 | 门禁引擎直接解决"Agent 跳过 Phase 导致产出偏离"的核心问题，无多余动作 |
| Session 2 | ✅ 符合预期 | 外部审查精准定位"纸面协议"根因，推动 orchestrator.js 彻底重写 |
| Session 3 | ✅ 符合预期 | Skill+MCP+Plugin 三层架构回答了"总纲怎么落地执行"，规划充分 |
| Session 4 | ⚠️ 部分偏离 | 本体论研究有价值，但 14 实体完整定义、Gartner 四级治理在当前阶段是**过度设计** |
| Session 5 | ❌ 偏离预期 | 5 大框架对比 + 3 大协议分析 + AgentOps 调研形成"学术化行业全景图"，与工程主线脱节 |

| Session 10 | ✅ 符合预期 | 4 个融合点全部有工程锚点，T+0min→T+6h 时序图将理念变为可计时的工程流程 |

### 偏离模式分析

Sessions 4-6 形成了一个清晰的"研究漂移带"，其发生机制如下：

```
Session 4（本体论）→ 触发条件："看到一篇好文章，想深入了解一下"
    ↓ 连锁反应
Session 5（行业标准）→ 心理驱动："既然研究了本体论，不如把行业标准也研究全"
    ↓ 连锁反应
Session 6（差距分析）→ 心理驱动："既然研究了行业标准，不如看看我们差多少"
    ↓
Session 7（用户纠偏）→ "我们是不是忘了 Skill/MCP？"
```

**漂移的根因**：研究从"服务于总纲"变成了"知识全景展示"。Session 4 的 14 个规范实体中，实际能用到的不超过 5 个；Session 5 的 5 大框架对比对总纲的架构决策没有实质性影响；Session 6 的 18% 数字传递了错误的信号。

**对照 Session 8-9 的正确模式**：同样是研究，但 Session 8 是"找到验证我们方向的证据"，Session 9 是"找到我们还没做的事情"。前者增强信心，后者制造焦虑。关键区别在于——**Session 8-9 的每个研究结论都有明确的工程锚点**（Sprint Contract → `开发总纲_可执行版.json`，IDE 自动化 → `automations/*.md`）。

### 重新评估：最小可行链路完成度

以"一句话→最终目标的最小可行链路"为坐标系重新评估：

| 维度 | 已有 | 完成度 | 说明 |
|------|------|--------|------|
| 意图锚定（Phase 1） | 44 问自审 + goal_trace + spec.json | **70%** | 核心链路完整，自审自动化待验证 |
| 门禁强制执行 | orchestrator.js + phase_gate_engine | **80%** | 代码级强制，8 Phase 全链路通过 |

**剩余关键缺口**（按优先级）：
1. **P0**：端到端跑通一次完整链路（当前最大缺口——所有模块都定义了但未串联验证）
2. **P1**：在 3 个 IDE 中实际配置自动化 prompt 并运行
3. **P2**：Sprint Contract 的端到端验证（从签订到审核到打回重做）
4. **P3**：第三个 IDE 角色确定（Windsurf）并部署 Bridge 插件

### 三条纠偏规则

基于以上审计，制定以下规则指导后续演进：

**规则一：研究必须有工程锚点。** 任何研究启动前，必须先回答"这个研究结果会被哪个文件/哪个函数使用？"如果回答不上来，说明研究时机未到。这条规则可以防止 Session 4-6 的重演。

**规则二：标准对齐从需求出发，不从行业出发。** 不是"行业有 6 层标准，我们差多少"，而是"我们要实现一句话→最终目标，最少需要哪几层？"当前阶段，L1（增强 LLM）+ L2（编排）+ L3（本体中的目标追踪）就够了，L4-L6 可以等到端到端跑通后再补。

**规则三：完成度评估用"最小可行链路"，不用"行业标准框架"。** 18% 是一个基于错误坐标系得出的数字。后续每一次完成度评估，都必须以"一句话→最终目标的最小可行链路"为唯一标尺。

## 十、Session 8：Anthropic Harness 工程研究

> 用户分享了一篇 Anthropic 工程博客的截图——《Harness design for long-running application development》，触发了对"长时间自主运行 AI Agent"的系统性研究。这是**对总纲"一句话实现最终目标"理念的决定性验证**。

### 10.1 文章来源

- **原文**：Anthropic Engineering — "Harness design for long-running application development"（2026 年 3 月）
- **作者**：Prithvi Rajasekaran（Anthropic Labs）
- **深度分析**：James Phoenix, "Generator-Evaluator Harness Design"（UnderstandingData, 2026 年 3 月）
- **韩文再分析**：HaruLogs 论坛深度解读

### 10.2 核心概念：什么是 Harness？

**Harness（驯服装置）** 不是 prompt，不是代码，而是**AI Agent 的工作环境设计**。如果说 prompt engineering 是"告诉 AI 做什么"，Harness engineering 就是"设计一个让 AI 无法犯错的环境"。

Anthropic 借用了 GAN（生成对抗网络）的架构思想：让两个 AI 互相对抗——一个造东西，一个挑毛病。通过结构性的分离来消除"自我评价偏差"。

### 10.3 三 Agent 架构

| 角色 | 职责 | 对应总纲角色 |
|------|------|-------------|
| **Planner（规划者）** | 把用户 1-4 句话扩展为完整产品规格 | OpenClaw（协调者）Phase 1 |
| **Generator（生成者）** | 按 sprint 逐个实现功能，自评后提交 | Codex（研究员）Phase 2-5 |
| **Evaluator（评估者）** | 用 Playwright 实际运行应用，点击测试，给出具体 bug 报告 | OpenClaw（监督者）所有检查点 |

**关键发现：AI 永远不会批评自己的作品。** Anthropic 实验证明，让同一个 AI 评价自己写的代码，永远是"看起来不错"。**"把生成和评估彻底分离，比训练 AI 自我批评容易得多。"**

### 10.4 Sprint Contract（冲刺合同）

这是 Anthropic 文章中最精妙的设计。在 Generator 开始干活之前，Evaluator 先和它"签合同"——明确什么算"完成"：

```
合同条款：矩形填充工具应该能拖拽填充区域
评估结果：FAIL — 拖拽只放了起点和终点的 tile，中间区域没填充。
         fillRectangle 函数存在但 mouseUp 事件没触发它。
         LevelEditor.tsx:892 需要修复。
```

这种具体到文件名和行号的反馈，Generator 能直接修复。对比"这个功能不太好"这种模糊反馈，效率天差地别。

**对总纲的意义**：Phase 门禁引擎的 `gate_file → produce_file` 链可以升级为 `gate_file → contract_file → produce_file` 链。不只检查"文件是否存在"，而是检查"文件内容是否满足验收合同"。

### 10.5 上下文焦虑（Context Anxiety）

Sonnet 4.5 在对话上下文接近满载时，会**仓促收尾**，假装做完了。Anthropic 发现**直接清空上下文重来**比压缩上下文效果更好——给 Agent 一个干净的 slate。

但 Opus 4.5+ 之后这个问题基本消除，可以一个 session 跑到底。

**对总纲的意义**：递归树的每个子节点天然开新 session，上下文是干净的。但总纲没有显式规定"单节点执行超过 N 轮后强制重置上下文"。这是一个可以加进去的防御机制。

### 10.6 真实数据对比

| 方式 | 时间 | 成本 | 结果 |
|------|------|------|------|
| 单个 Agent 直接干 | 20 分钟 | \$9 | UI 出来了，但核心功能全是坏的 |
| 三 Agent Harness | 6 小时 | \$200 | 16 个功能完整可玩，可迭代 |

22 倍的成本换来一个**能用的产品** vs 一个**看起来像但完全不能用的废品**。

### 10.7 Harness 随模型进化而退化

最反直觉的结论：**模型越好，Harness 越简单。** 最初为 Sonnet 4.5 设计的复杂架构，到了 Opus 4.6 很多组件可以砍掉。Harness 的每个组件都编码了一个假设："这个模型做不到什么"。假设会过期。

**对总纲的意义**：Phase 门禁引擎、44 问自审、8 个检查点——这些都可能随着模型升级而变得冗余。需要在总纲里加一个**"Harness 退化机制"**：当模型能力达到某个阈值时，自动跳过某些检查点。

### 10.8 Evaluator 的 Few-shot 校准

Evaluator 一开始太"仁慈"——发现 bug 后自己说服自己"也没那么严重"。Anthropic 通过 few-shot 样本校准了 Evaluator 的严格程度。

**对总纲的意义**：OpenClaw 监督者需要**"校准样本集"**——几个典型的 pass/fail 案例，防止过于宽容或苛刻。

### 10.9 语言编码价值观

"the best designs are museum quality" 这句话会引导模型产出特定风格的设计。Prompt 的语言不只是指令，它在塑造输出的分布。

**对总纲的意义**：总纲里的"一句话"不是随便一句话。它对最终产出的方向有巨大的塑造力。应该规定"一句话"的最小质量标准——它应该包含什么信息，不能只是"帮我写个小说"。

## 十一、Session 9：IDE 自动化能力全量研究

> 用户提出关键问题："三个 IDE 的自动化功能是否可以用来驱动通信插件？" 触发了对 5 个主流 AI IDE 自动化能力的系统性研究。核心发现：**不是所有 IDE 都跑在云端，Codex 和 OpenClaw 的自动化是本地运行的。**

### 11.1 五个 IDE 自动化能力对比

| 维度 | Codex | OpenClaw | Cursor | TRAE | Claude Code |
|------|-------|----------|--------|------|-------------|
| **自动化功能名** | Thread Automations | Cron Jobs | Automations | 无内置调度 | Routines |
| **运行位置** | 🟢 本地桌面应用 | 🟢 本地 Gateway 进程 | 🔴 云端远程 | 无 | 🔴 云端 |
| **能访问本地文件** | ✅ | ✅ | ❌ | N/A | ❌ |
| **能访问 localhost** | ✅ | ✅ | ❌ | N/A | ❌ |
| **调度粒度** | cron + 分钟级 | cron + 一次性 + 固定间隔 | cron + GitHub/Slack 事件 | N/A | 每小时/每天/每周 |
| **上下文保留** | ✅ 跨运行保留 | ✅ Isolated / Main / Custom | ❌ 每次独立 | N/A | ❌ 每次独立 |
| **执行模式** | Standalone + Thread | Main Session / Isolated / Custom | Cloud Agent | N/A | Web 基础设施 |

### 11.2 关键发现：云沙箱边界

**Cursor 的致命缺陷**：Cursor Automations 在云端沙箱中运行，**无法访问本地文件系统，无法访问 localhost**。这意味着 Cursor 的自动化：
- 读不到 `.bridge/` 状态文件
- 连不上 `localhost:9527` 的 Bridge
- 无法直接触发本地的 `ide-bridge.js`

**Codex 和 OpenClaw 的优势**：两者的自动化都是本地运行的，可以直接读写 `.bridge/` 文件，可以连接 `localhost:9527`。这让它们天然适合做通信插件的自动化驱动。

### 11.3 OpenClaw 的 Cron 执行模式（关键）

OpenClaw Cron 支持四种执行模式，这对我们的架构至关重要：

| 模式 | 行为 | 适用场景 |
|------|------|----------|
| **Main Session** | 将系统事件注入到当前正在聊天的会话中 | 你正在跟 OpenClaw 聊天时，自动收到"Codex 提交审核了" |
| **Isolated** | 开一个独立 session，不打扰用户 | 后台自动审核，结果写入文件 |
| **Current Session** | 绑定到创建时的会话 | 持续性任务跟踪 |
| **Custom Session** | 持久化命名会话，跨运行积累上下文 | 长期跟踪某个项目 |

**推荐方案**：审核类任务用 **Isolated 模式**（后台自动审核，不打扰用户），设计类任务用 **Main Session 模式**（把事件注入当前对话，让用户看到进展）。

### 11.4 Cursor 的替代方案研究

| 方案 | 配置 | IDE 数量 | 优势 | 劣势 |
|------|------|----------|------|------|
| **A：Windsurf 替代 Cursor** | OpenClaw + Codex + Windsurf | 3 | VS Code 100% 兼容，Cascade Agent 本地运行 | Windsurf 无内置 cron，需 OpenClaw 驱动 |
| **B：两个 Codex 实例** | OpenClaw + Codex（研究员）+ Codex（开发者） | 2 | 统一 IDE，Thread Automation 本地运行 | 同一个 IDE 两个线程，可能上下文冲突 |
| **C：OpenClaw 统一调度** | OpenClaw Cron → spawn Codex 子 Agent | 1 | 最简洁，不需要同时开多个 IDE | 失去物理隔离，调试困难 |

**推荐方案**：**A（Windsurf 替代 Cursor）**。Windsurf 是 VS Code 的直接 fork，与我们的 IDE Bridge 插件 100% 兼容。Cascade Agent 可以做本地实现工作。虽然没有内置 cron，但可以让 OpenClaw 的 Cron 来驱动它。

### 11.5 云沙箱与本地桥接的通用模式

研究发现了 5 种桥接模式，但大多数对我们不适用。只有一种模式值得关注：

**MCP 续传模式**：远程 Agent 生成 tool_calls 但不执行，返回给客户端，客户端在本地执行工具，再将结果送回云端。如果需要 Cursor 云端自动化参与，可以通过这个模式让 Cursor 的云端 Agent 发出 HTTP 请求到本地 Bridge 的 REST 接口。

## 十二、Session 10：架构融合 — 总纲 × Harness × IDE Bridge

> 这是三天探讨的收束点。将 Anthropic 的 Harness 工程、我们的总纲体系、IDE Bridge 通信插件，三个维度融合为一个统一的执行架构。

### 12.1 "一句话"与"深度工作 6 小时"的同构性

将总纲与 Anthropic Harness 逐层对照：

| 总纲 | Anthropic Harness | 共通本质 |
|------|-------------------|----------|
| "你说一句话" | "User prompt (1-4 sentences)" | 极简输入 → 全自动展开 |
| Phase 1: 44 问自审 → spec.json | Planner 把简短 prompt 扩展为完整产品规格 | 理解意图 + 制定计划 |
| Phase 2-5: 去程拆解 | Generator 按 sprint 逐个实现 | 分解任务 + 执行 |
| Phase 门禁引擎（gate_file → produce_file） | Sprint Contract（开始前签合同） | 强制执行链，不可跳过 |
| 监督者 8 个检查点 | Evaluator 用 Playwright 实际测试 | 独立审核，不自己做自己审 |
| 递归树：叶节点 → 全链路 | Generator-Evaluator 循环：5-15 轮迭代 | 逐层验证，不合格打回重做 |
| 全链路通路 = 开发完成 | 6 小时 /\$200 = 可玩的产品 | 完整交付 |

**结论：两者本质上是同一件事，只是从两个视角描述的。** 总纲用递归树管"整体结构"，Harness 用对抗循环管"每层质量"。两者不是替代关系，是**层级关系**——树管宏观，循环管微观。

### 12.2 融合后的完整架构

```
用户一句话
    ↓
┌──────────────────────────────────────────────────┐
│ Planner（Phase 1：44 问自审 → spec.json）         │
│ 对应 Anthropic 的 Planner + 总纲的 Phase 1       │
│ 产出：spec.json + sprint_contract.json           │
└──────────────────────┬───────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│ 去程（Phase 2-5）：递归拆解                         │
│   每个节点内部跑 Generator-Evaluator 循环：         │
│                                                   │
│   ┌─────────────┐     ┌──────────────────┐       │
│   │ Generator    │ ←→  │ Evaluator         │       │
│   │ (Codex/Cursor)│    │ (OpenClaw 监督者)  │       │
│   │ 产出设计/代码  │    │ Sprint Contract   │       │
│   │              │     │ 自动化检查 + 语义审查 │       │
│   └─────────────┘     └──────────────────┘       │
│         5-15 轮迭代直到 pass                       │
│                                                   │
│   每个节点有 gate_file（前置条件）                   │
│   和 produce_file（强制产出）                       │
│   + Sprint Contract（验收合同）                    │
│                                                   │
│   通信层：IDE Bridge WebSocket 实时推送             │
│   Codex 完成 → 发 checkpoint.request → Bridge     │
│   OpenClaw Cron（Isolated）→ 自动审核 → 回复结果    │
└──────────────────────┬───────────────────────────┘
                       ↓
┌──────────────────────────────────────────────────┐
│ 回程（Phase 6-8）：逐层合成 + 验证                   │
│   Phase 6: 实现（有蓝图对照）                       │
│   Phase 7: 通路验证（自动化内容校验）                 │
│   Phase 8: 交付 + 产物体检                          │
│                                                   │
│   全链路通路 + 全节点对齐                           │
│   + health_check 全部通过                          │
│   + Sprint Contract 全部 pass                      │
│   = 开发完成                                       │
└──────────────────────────────────────────────────┘
```

### 12.3 四个具体融合点

#### 融合点 1：Sprint Contract 层（加到 Phase 门禁引擎）

在现有的 `gate_file → produce_file` 链上，增加一个 `contract_file`：

```json
{
  "sprint_contract": {
    "phase": 2,
    "task_id": "task_xxx",
    "acceptance_criteria": [
      {
        "id": "ac_1",
        "description": "路径图必须包含从起点到终点的完整节点序列",
        "verification": "自动化：检查 path.json 的 nodes 数组长度 > 0 且 path.is_connected == true",
        "weight": "critical"
      }
    ],
    "negotiated_by": "openclaw_evaluator",
    "negotiated_at": "2026-07-16T..."
  }
}
```

每个 Phase 开始前，Evaluator（OpenClaw）先和 Generator（Codex/Cursor）"签合同"。执行完成后，对照合同逐条检查。不通过 → 标记具体哪条不通过 + 具体原因 → 打回重做。

#### 融合点 2：上下文焦虑防御机制

在总纲的递归执行算法里加一条规则：

```
每个节点执行超过 15 轮对话后，强制上下文重置。
重置时保留：
  - spec.json（当前节点目标）
  - sprint_contract.json（当前验收合同）
  - 已产出的文件
丢弃：
  - 对话历史
  - 中间推理过程
然后重新开始当前 Phase。
```

#### 融合点 3：Harness 退化机制

在总纲里加一个 `model_capability_profile` 字段：

```json
{
  "model_capability_profile": {
    "model": "sonnet_4.5",
    "capabilities": {
      "self_evaluation_reliable": false,
      "long_context_stable": false,
      "complex_decomposition": true
    },
    "active_checks": ["cp_0", "cp_1", "cp_2", "cp_3", "cp_4", "cp_5", "cp_6", "cp_7"],
    "notes": "Sonnet 4.5 有上下文焦虑，需要完整检查点链"
  }
}
```

当模型升级到 Opus 4.6 级别时，可以自动减少检查点、简化 Sprint Contract 的严格程度。

#### 融合点 4：IDE Bridge 驱动的自动化时序

完整的一次"一句话 → 6 小时深度工作"时序：

```
T+0min   用户输入一句话
T+0.1min OpenClaw 插件自动创建任务 + 分配 Sprint Contract
T+0.2min Bridge 推送 task.assign → Codex
T+0.3min Codex 插件收到消息 → 写入 .bridge/current_task.json
T+0.5min Codex Thread Automation 检测到新任务 → 开始 Phase 1
T+5min   Codex 完成 Phase 1 → 发 checkpoint.request → Bridge
T+5.1min OpenClaw Cron（Isolated）收到审核请求 → 自动审核
T+5.5min OpenClaw 回复 checkpoint.result（pass）→ Bridge
T+5.6min Codex 收到 pass → 继续 Phase 2
         ... 循环 5-15 次，覆盖 Phase 2-8 ...
T+6h     全链路通路 = 开发完成
T+6h+1s  OpenClaw 弹出通知："任务完成！"
```

**用户参与次数：0 次。** 从输入一句话到收到完成通知，中间 6 小时不需要任何人工干预。

### 12.4 自动化覆盖度评估

| Phase | 自动化方式 | 人工介入 |
|-------|-----------|----------|
| Phase 1（意图锚定） | Codex Thread Automation 检测 task.assign → 自动执行 | 0 |
| Phase 2-5（去程拆解） | Codex Thread Automation 持续执行 + 每个 Phase 提交审核 | 0 |
| 审核（每个检查点） | OpenClaw Cron（Isolated）自动审核 + Sprint Contract 对照 | 0 |
| Phase 6（实现） | Windsurf Cascade Agent 或 Codex Thread Automation 2 号实例 | 0 |
| Phase 7-8（验证+交付） | OpenClaw Cron 自动化验证 + 产物体检 | 0 |
| **总计** | **全自动化** | **0 次** |

### 12.5 一句话总结融合后的体系

> **你说一句话 → Planner 自动展开 → 递归树逐层拆解（每层有 Sprint Contract + 门禁引擎）→ Evaluator 独立审核（自动化 + 语义审查）→ 不通过打回重做 → 通过放行 → 全链路通路 = 6 小时深度工作 = 开发完成。你参与 0 次。**
